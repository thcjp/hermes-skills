# 代码审查哨兵

在代码合并前执行结构化质量审查。以"高级工程师是否会批准这次合并"为标准,从五个维度全面把关。

## 使用场景

| 场景 | 触发条件 | 说明 |
|:-----|:---------|:-----|
| PR 审查 | 提交 Pull Request | 合并前五维度审查 |
| 变更评估 | 代码变更较大 | 评估是否需要拆分 |
| 质量门禁 | CI/CD 流水线 | 自动化质量检查 |
| 代码健康度 | 定期代码审计 | 评估代码库健康趋势 |
| 新人指导 | 新成员提交代码 | 提供建设性反馈 |

## 工作流

### 1. 变更概览

1. **获取变更差异**:`git diff` 或 PR diff
2. **统计变更规模**:新增行数/删除行数/涉及文件数
3. **变更尺寸评估**:
   - 理想:~100 行(易审查、易回滚)
   - 可接受:100-300 行(需分段审查)
   - 需拆分:>300 行(建议拆分为多个 PR)
4. **变更分类**:新功能/Bug修复/重构/配置/文档

### 2. 五维度审查

#### 维度一:正确性
- 逻辑是否正确实现需求
- 边界条件是否处理(空值/零值/极值/并发)
- 错误处理是否完整(异常捕获/错误传播/降级)
- 是否引入回归

#### 维度二:可维护性
- 命名是否清晰表达意图
- 函数/类职责是否单一
- 是否存在重复代码(应抽象)
- 复杂度是否可控(圈复杂度 < 10)
- 注释是否解释"为什么"而非"是什么"

#### 维度三:安全性
- 输入是否校验(防注入/XSS)
- 权限检查是否到位
- 密钥/凭证是否泄露
- 依赖是否有已知漏洞
- 是否遵循最小权限原则

#### 维度四:性能
- 是否有不必要的计算/查询
- N+1 查询问题
- 内存泄漏风险
- 大数据集处理是否分页/流式
- 是否阻塞主线程/事件循环

#### 维度五:可测试性
- 是否有对应的测试
- 测试是否覆盖关键路径
- 是否易于编写测试(依赖注入/可 mock)
- 测试是否快速且稳定

### 3. 严重度标签

| 标签 | 含义 | 处理方式 |
|:-----|:-----|:---------|
| **Blocking** | 必须修复才能合并 | 阻塞合并 |
| **Should Fix** | 强烈建议修复 | 合并前修复 |
| **Nit** | 吹毛求疵的小问题 | 可选修复 |
| **Optional** | 可选的改进建议 | 后续迭代 |
| **FYI** | 仅供参考的信息 | 无需行动 |
| **Praise** | 值得表扬的写法 | 鼓励保持 |

### 4. 拆分策略(当变更过大时)

1. **按功能拆分**:一个 PR 一个功能点
2. **按层拆分**:数据层/业务层/接口层分开
3. **按文件拆分**:独立文件变更独立 PR
4. **先重构后功能**:重构 PR 先合并,再提功能 PR
5. **拆分顺序**:确保每个子 PR 独立可编译可测试

### 5. 审查报告

1. **总体评估**:Approve / Request Changes / Needs Discussion
2. **五维度评分**:每维度 0-10 分
3. **问题清单**:按严重度排序
4. **改进建议**:具体可执行的修改建议
5. **亮点表扬**:值得学习的优秀写法

## 审查原则

1. **尊重作者**:对事不对人,建设性反馈
2. **解释原因**:不仅指出问题,还解释为什么是问题
3. **提供方案**:给出具体的修改建议或代码示例
4. **区分优先级**:不把 Nit 和 Blocking 混为一谈
5. **及时反馈**:24 小时内完成审查
6. **代码健康度优先**:每次合并都应提升而非降低代码健康度

## 输出规范

- 审查报告:`output/{pr-id}/review-report.md`
- 问题清单:`output/{pr-id}/issues.md`
- 拆分建议:`output/{pr-id}/split-plan.md`(如需)

## 依赖说明

### 运行环境
- **Agent平台**: Claude Code / Cursor / Codex / Gemini CLI / Windsurf 等支持SKILL.md的任意Agent
- **操作系统**: Windows / macOS / Linux
- **运行时**: 需要Agent支持exec(命令行执行)能力

### 第三方依赖
| 依赖项 | 类型 | 是否必需 | 获取方式 |
|:-------|:-----|:---------|:---------|
| Git | 工具 | 可选 | git diff获取变更差异 |
| LLM API | API | 可选 | 由Agent内置LLM提供代码分析 |

### API Key 配置
- 本Skill无需额外API Key配置

### 纯Markdown使用说明
本Skill为代码审查方法论指导。需要exec工具执行git diff获取变更,审查本身由Agent LLM完成。

只需将SKILL.md文件放入Agent的skills目录即可直接使用。
如果Skill中包含exec工具调用,需要Agent支持命令行执行能力。

### 可用性分类
- **分类**: MD+EXEC
- **说明**: 纯Markdown,但需要exec能力(命令行执行),用于文件读写和命令调用
