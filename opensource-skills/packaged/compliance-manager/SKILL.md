# 合规管理器

构建企业级合规操作系统,将合规从一次性审计转变为持续运营。从控制项管理到证据收集,从审计就绪到持续监控,全流程合规管理。

## 适用场景

| 场景 | 输入 | 输出 |
|:-----|:-----|:-----|
| ISO 27001 | 企业系统范围+资产清单 | 风险评估报告+控制项目录+适用性声明 |
| SOC 2 | 业务类型+信任准则选择 | 控制项清单(30+)+证据收集计划+差距分析 |
| GDPR/个人信息保护法 | 个人数据处理流程 | 数据映射图+DPIA模板+权利实现流程 |
| 等保2.0 | 信息系统等级+网络架构 | 定级报告+差距分析+整改方案 |
| 审计准备 | 审计范围+时间表 | 证据清单+访谈准备+差距补救计划 |
| 持续合规 | 控制项状态+变更记录 | 监控报告+事件台账+培训记录 |

### 不适用于
- 实施渗透测试与漏洞扫描(请使用安全测试工具)
- 法律意见提供(本Skill为合规管理框架,非法律咨询)
- 审计师角色执行(本Skill仅辅助被审计方准备,不替代审计师)
- 实时安全监控告警(请使用SIEM/SOC系统)
- 自动化合规凭证颁发(需认证机构参与)

## 核心能力

1. **中外合规框架全覆盖**:ISO 27001/SOC 2/GDPR/HIPAA国际框架 + 网络安全法/数据安全法/个人信息保护法/等保2.0国内合规
2. **控制项生命周期管理**:目录建立、实施指导、监控评估、偏差补救全流程
3. **证据收集与维护**:自动证据(日志/配置)+手动证据(截图/签核)、版本控制、过期识别
4. **审计就绪与支持**:差距分析、证据完整性检查、访谈准备、发现项响应
5. **持续合规运营**:变更管理、事件响应(72小时监管报告)、培训与意识

## 使用流程

### Step 1: 合规框架选择与范围界定
1. 用户提供业务类型、目标合规框架、系统范围
2. 确定合规框架:海外业务用ISO 27001/SOC 2/GDPR,国内业务用等保2.0/网络安全法/数据安全法/个人信息保护法
3. 界定系统范围(哪些服务/数据/流程)
4. 识别数据流(数据映射)
5. 确定利益相关者

### Step 2: 风险评估
1. 资产识别与分类(信息资产/物理资产/人员资产)
2. 威胁与漏洞评估
3. 风险评分(可能性 x 影响,1-5级)
4. 风险处置计划(接受/缓解/转移/规避)

### Step 3: 控制项设计与实施
1. 选择适用控制项(ISO Annex A / SOC 2 Trust Criteria / 等保要求)
2. 定义控制项描述与实施方式
3. 分配控制项所有者
4. 设定控制项频率(持续/月度/季度/年度)
5. 实施技术控制(WAF/SIEM/加密)+ 管理控制(政策流程)+ 物理控制

### Step 4: 证据收集与管理
1. 自动收集:API/日志/配置导出
2. 手动收集:截图/文档/签核
3. 证据分类:设计证据/运行证据
4. 集中化证据库存储,时间戳与版本控制

### Step 5: 审计就绪检查
1. 审计范围确认
2. 控制项差距分析
3. 证据完整性检查
4. 审计员沟通与访谈准备
5. 持续监控:变更影响评估、事件响应、定期培训

## 工作流(框架对照)

### 1. 合规框架搭建

1. **范围界定**
   - 确定合规框架(ISO 27001/SOC 2/GDPR/HIPAA/等保2.0)
   - 界定系统范围(哪些服务/数据/流程)
   - 识别数据流(数据映射)
   - 确定利益相关者
2. **风险评估**
   - 资产识别与分类
   - 威胁与漏洞评估
   - 风险评分(可能性 x 影响)
   - 风险处置计划(接受/缓解/转移/规避)
3. **控制项设计**
   - 选择适用控制项(Annex A / Trust Criteria / 等保要求)
   - 定义控制项描述与实施方式
   - 分配控制项所有者
   - 设定控制项频率(持续/月度/季度/年度)

### 2. 控制项管理

1. **控制项目录**
   - 访问控制(认证/授权/密码/会话)
   - 加密(传输/存储/密钥管理)
   - 网络安全(防火墙/IDS/分段)
   - 运营安全(变更管理/补丁/备份)
   - 人员安全(背景调查/培训/终止)
   - 物理安全(门禁/监控/环境)
   - 供应商管理(尽职调查/合同/监控)
   - 事件管理(检测/响应/报告)
2. **控制项实施**
   - 技术控制:自动化实施(WAF/SIEM/加密)
   - 管理控制:政策与流程文档
   - 物理控制:设施安全措施
   - 纠正控制:漏洞修复/事件响应
3. **控制项监控**
   - 自动化监控(配置扫描/日志分析)
   - 定期测试(渗透测试/控制测试)
   - 偏差识别与补救
   - 控制有效性评估

### 3. 证据管理

1. **证据收集**
   - 自动收集:API/日志/配置导出
   - 手动收集:截图/文档/签核
   - 证据分类:设计证据/运行证据
   - 证据存储:集中化证据库
2. **证据维护**
   - 时间戳与版本控制
   - 证据链完整性
   - 过期证据识别
   - 定期证据更新

### 4. 审计就绪

1. **审计前准备**
   - 审计范围确认
   - 控制项差距分析
   - 证据完整性检查
   - 审计员沟通
2. **审计支持**
   - 证据快速检索
   - 控制项演示
   - 访谈准备
   - 发现项响应
3. **审计后跟进**
   - 发现项分类(观察项/发现项/严重发现)
   - 补救计划制定
   - 补救跟踪与验证
   - 报告归档

### 5. 持续合规运营

1. **变更管理**
   - 系统变更合规影响评估
   - 新服务合规审查
   - 控制项更新
2. **事件管理**
   - 安全事件分类与响应
   - 合规影响评估
   - 监管报告(GDPR 72小时/网络安全法立即)
   - 事件后改进
3. **培训与意识**
   - 年度安全培训
   - 钓鱼模拟
   - 政策签核
   - 合规文化推广

## 错误处理

| 错误场景 | 原因 | 处理方式 |
|:---------|:-----|:---------|
| 控制项失败 | 技术实施缺陷/人员疏忽 | 立即评估风险,启动补救计划,记录例外,通知审计员 |
| 审计发现重大问题 | 证据缺失/控制设计缺陷 | 分类严重程度,制定补救时间表,跟踪验证,必要时升级处理 |
| 法规变更 | 新法规生效/现有法规修订 | 影响评估,控制项更新,培训通知,审计员沟通 |
| 数据泄露事件 | 攻击/内部威胁/配置错误 | 启动事件响应,72小时内监管报告(网络安全法),受影响方通知,证据保留 |
| 供应商违规 | 第三方未履行合规义务 | 合同审查,控制项重新评估,替换评估,过渡计划 |
| 证据过期 | 证据未及时更新 | 补充收集,控制项重新测试,记录偏差,更新证据库 |
| 跨境数据传输受限 | 数据本地化要求 | 评估数据流向,使用本地化部署,签署SCC标准合同条款 |

import boto3
import json
from datetime import datetime, timedelta
from pathlib import Path

EVIDENCE_DIR = Path("output/soc2/evidence")

def collect_cloudtrail_logs():
    """收集CloudTrail登录日志（CC6.1 逻辑访问证据）"""
    client = boto3.client('cloudtrail')
    end_time = datetime.utcnow()
    start_time = end_time - timedelta(days=1)
    
    response = client.lookup_events(
        LookupAttributes=[{'AttributeKey': 'EventName', 'AttributeValue': 'ConsoleLogin'}],
        StartTime=start_time,
        EndTime=end_time,
        MaxResults=50
    )
    
    output = EVIDENCE_DIR / "CC6.1" / "login-logs" / f"{end_time.strftime('%Y-%m-%d')}.json"
    output.parent.mkdir(parents=True, exist_ok=True)
    output.write_text(json.dumps(response['Events'], indent=2, ensure_ascii=False))
    print(f"收集登录日志: {len(response['Events'])} 条 → {output}")

def collect_mfa_status():
    """收集MFA启用状态（CC6.1 MFA证据）"""
    client = boto3.client('iam')
    users = client.list_users()['Users']
    
    mfa_status = []
    for user in users:
        mfa_devices = client.list_mfa_devices(UserName=user['UserName'])
        mfa_status.append({
            'user': user['UserName'],
            'mfa_enabled': len(mfa_devices['MFADevices']) > 0,
            'mfa_devices': [d['SerialNumber'] for d in mfa_devices['MFADevices']]
        })
    
    output = EVIDENCE_DIR / "CC6.1" / "mfa-status" / f"{datetime.utcnow().strftime('%Y-%m-%d')}.json"
    output.parent.mkdir(parents=True, exist_ok=True)
    output.write_text(json.dumps(mfa_status, indent=2, ensure_ascii=False))
    
    # 检查是否有未启用MFA的用户
    no_mfa = [u for u in mfa_status if not u['mfa_enabled']]
    if no_mfa:
        print(f"⚠ 发现 {len(no_mfa)} 个用户未启用MFA: {[u['user'] for u in no_mfa]}")
    else:
        print(f"✓ 所有 {len(mfa_status)} 个用户均已启用MFA")

def collect_password_policy():
    """收集密码策略配置（CC6.3 证据）"""
    client = boto3.client('iam')
    policy = client.get_account_password_policy()['PasswordPolicy']
    
    output = EVIDENCE_DIR / "CC6.3" / "password-policy" / f"{datetime.utcnow().strftime('%Y-%m-%d')}.json"
    output.parent.mkdir(parents=True, exist_ok=True)
    output.write_text(json.dumps(policy, indent=2, ensure_ascii=False, default=str))
    print(f"密码策略: 最低{policy.get('MinimumPasswordLength', 'N/A')}位")

# 执行收集
if __name__ == "__main__":
    collect_cloudtrail_logs()
    collect_mfa_status()
    collect_password_policy()
    print("证据收集完成")

## 常见问题

### Q1: SOC 2和ISO 27001应该先做哪个?
A: 取决于客户群体。如果客户主要是美国SaaS/科技公司,SOC 2 Type II优先(审计周期6-12个月)。如果客户为全球企业或欧洲市场,ISO 27001更通用(认证周期3-6个月)。国内业务优先等保2.0(法律强制要求)。两者控制项有60%+重叠,可同步推进。建议从SOC 2 Type I开始(时点审计),再过渡到Type II(期间审计)。

### Q2: 个人信息保护法和GDPR有什么主要区别?
A: (1)适用范围:GDPR看数据处理是否在欧盟,个保法看是否在中国境内提供产品服务。(2)数据主体权利:个保法增加"解释说明权"。(3)跨境传输:GDPR需SCC/BCR,个保法需安全评估或认证。(4)罚款:GDPR最高4%全球营收,个保法最高5000万元或5%营收。(5)DPO要求:GDPR强制指定DPO,个保法要求指定个人信息保护负责人。建议两者同时合规,控制项可复用80%。

### Q3: 审计前发现控制项差距,补救时间不够怎么办?
A: (1)与审计员沟通,说明补救计划与时间表,部分差距可纳入"观察项"而非"发现项"。(2)对高风险差距优先补救,低风险差距列入下一审计周期。(3)考虑先做Type I(时点)审计,获得部分合规凭证,后续再做Type II(期间)。(4)对无法补救的差距,签署"例外声明"并制定补救承诺。

### Q4: 等保2.0的级别如何确定?
A: 根据系统受破坏后对公民/法人/国家安全、社会秩序、公共利益的损害程度。二级:损害公民法人合法权益(通用SaaS);三级:严重损害社会秩序公共利益(金融/政务/医疗);四级:严重损害国家安全(关键基础设施)。SaaS创业公司一般定二级,测评周期2年;涉及金融数据定三级,测评周期1年。

- 控制项实施需要工程团队配合落地,本Skill仅输出实施指南
- 法规更新频繁,需定期联网查询最新版本(建议季度检查)
- 跨境业务合规复杂,建议同时咨询当地律师
- 不涵盖物理安全现场评估(需实地核查)
- 不替代内部审计职能,仅辅助被审计方准备

## 安全与合规

- 本Skill不存储任何API Key或敏感凭证
- GRC工具集成由用户自行配置,凭证不进入Skill上下文
- 证据文件存储在本地 `output/` 目录,建议加密存储敏感证据
- 审计证据含个人数据时,遵循最小化原则处理
- 不上传内部数据至外部服务,所有分析在Agent本地完成
- 数据泄露事件需在72小时内(网络安全法)向网信办报告
