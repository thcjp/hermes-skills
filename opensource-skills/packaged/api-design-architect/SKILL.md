# API设计架构师

契约优先的 API 设计方法。先定义接口契约,再实现。确保 API 可演化、可维护、不破坏现有用户。

## 使用场景

| 场景 | 触发条件 | 说明 |
|:-----|:---------|:-----|
| 新 API 设计 | 从零设计接口 | 契约优先,先定义后实现 |
| 模块边界 | 划分系统模块 | 明确接口与依赖方向 |
| 公共接口 | 暴露给外部的能力 | 谨慎设计,难以修改 |
| API 演化 | 现有 API 迭代 | 向后兼容的演进策略 |
| 接口审查 | 评估接口质量 | 检查设计原则遵循度 |

## 工作流

### 1. 契约优先设计

1. **明确能力**:API 提供什么能力?解决什么问题?
2. **定义资源/实体**:
   - REST:资源 + HTTP 方法
   - GraphQL:类型 + Query/Mutation
   - RPC:方法 + 参数
3. **定义契约**:
   - OpenAPI/Swagger 规范(REST)
   - GraphQL Schema
   - Protocol Buffers(gRPC)
4. **先契约后实现**:契约确认后再编码,避免实现驱动设计

### 2. 设计原则

#### Hyrum 定律
> "当你有足够多的 API 用户,你在合同中承诺什么并不重要:你系统的所有可观察行为都会被某人依赖。"

- 假设所有可观察行为都会被依赖
- 谨慎暴露实现细节(顺序、时序、内部状态)
- 改变"未承诺"的行为也可能破坏用户

#### One-Version 规则
- 同一能力只暴露一个版本
- 避免多版本并存导致的维护地狱
- 演化通过加字段(向前兼容)而非版本号
- 必须破坏性变更时,提供迁移期

#### 最小惊讶原则
- API 行为符合用户直觉
- 命名一致、行为可预测
- 错误信息可操作

### 3. 错误语义设计

1. **错误分类**:
   - 客户端错误(4xx):请求有问题,客户端可修正
   - 服务端错误(5xx):服务有问题,客户端可重试
   - 业务错误:请求合法但业务规则不允许
2. **错误格式统一**:
   ```json
   {
     "error": {
       "code": "VALIDATION_FAILED",
       "message": "Email format is invalid",
       "details": [{"field": "email", "issue": "missing @"}],
       "requestId": "req_xxx"
     }
   }
   ```
3. **错误码设计**:
   - 机器可读的大写枚举(`VALIDATION_FAILED`)
   - 不用 HTTP 状态码作为唯一错误信息
   - 区分错误类型(可重试 vs 不可重试)
4. **幂等性**:写操作支持幂等键,防止重复执行

### 4. 边界校验

1. **入口校验**(信任边界):
   - 校验所有输入参数(类型/长度/范围/格式)
   - 拒绝不合规请求,返回明确错误
   - 白名单优于黑名单
2. **出口校验**(响应边界):
   - 校验响应数据结构符合契约
   - 不泄露内部实现细节
   - 敏感字段过滤
3. **校验位置**:
   - 网关层:基本格式校验
   - 业务层:业务规则校验
   - 数据层:数据完整性校验

### 5. API 演化策略

#### 向后兼容变更(安全)
- 新增可选字段
- 新增端点
- 新增枚举值(客户端需处理未知值)

#### 破坏性变更(需迁移)
- 删除/重命名字段
- 改变字段类型
- 改变行为语义
- 迁移策略:
  1. 新旧并存(双写)
  2. 通知用户迁移
  3. 监控旧版使用率
  4. 设定废弃时间表
  5. 最终下线旧版

### 6. 版本与文档

1. **版本策略**:
   - URL 版本(`/v1/`)或 Header 版本
   - 语义化版本(SemVer)用于 SDK
2. **文档要求**:
   - 每个端点:用途、参数、响应、错误
   - 示例请求与响应
   - 认证方式
   - 速率限制
3. **变更日志**:记录每个版本的变更

## 设计检查清单

- [ ] 契约先于实现定义
- [ ] 资源/方法命名一致
- [ ] 错误格式统一且可操作
- [ ] 输入校验完整
- [ ] 响应不泄露内部细节
- [ ] 写操作支持幂等
- [ ] 破坏性变更有迁移策略
- [ ] 文档完整且同步
- [ ] 考虑了 Hyrum 定律影响
- [ ] 遵循 One-Version 规则
