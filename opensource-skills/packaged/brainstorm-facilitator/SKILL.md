# 头脑风暴引导师

在编码前通过苏格拉底式提问，深入理解真实需求。不急于写代码，先通过对话探索需求边界、对比方案、验证设计，最终输出结构化设计文档。

## 核心能力

- **苏格拉底式提问**：通过目标/约束/边界三类问题，层层递进挖掘真实需求，避免表面需求误导设计
- **方案对比与推荐**：基于需求生成 2-3 个可行方案，权衡矩阵评估（成本/速度/可维护性/扩展性），提供推荐与渐进式路径
- **设计验证与风险评估**：边界压力测试（用户量 10 倍/数据量 100 倍）、技术/业务/运营三维风险评估
- **结构化设计文档输出**：项目概述 + 用户故事 + 技术方案 + 实现计划 + 风险缓解，配套架构图/数据流图/时序图
- **分段评审迭代**：设计分段呈现，每段请求用户确认，根据反馈迭代优化

## 适用场景

| 场景 | 输入 | 输出 |
|---|---|---|
| 新项目启动 | 项目想法 + 业务背景 | 苏格拉底提问清单 + 设计文档 + 技术栈推荐 |
| 功能设计 | 功能描述 + 用户角色 | 用户故事 + 边界定义 + 验收标准 + 方案对比 |
| 架构决策 | 技术选型需求 + 约束 | 方案权衡矩阵 + 推荐方案 + ADR 决策记录 |
| 需求澄清 | 模糊需求描述 | 边界探索问题清单 + 假设验证 + 明确需求文档 |
| 重构规划 | 现状描述 + 痛点 | 现状评估 + 目标定义 + 渐进式迁移策略 |
| PoC 设计 | 概念验证需求 | 假设定义 + 验证方法 + 成功标准 + PoC 范围 |

**不适用于**：
- 已有完整 PRD 的执行阶段（需求已明确，直接进入实现）
- 纯 UI/UX 视觉设计（属设计师范畴）
- 项目进度管理与跟踪（属 PM 工具职责）
- 代码实现与调试（属开发阶段）

## 使用流程

### Step 1: 目标探索（苏格拉底式提问）
- 核心目标："你想要实现什么？"
- 业务动机："为什么需要这个？"
- 用户角色："谁会使用？"
- 时间约束："什么时候需要？"
- 验收标准："成功是什么样的？"

### Step 2: 约束探索
- 技术限制："有什么技术约束？"
- 成本约束："预算范围？"
- 人力约束："团队技能？"
- 集成约束："已有系统需要对接？"
- 合规约束："是否有行业合规要求？"

### Step 3: 边界探索
- MVP 范围："什么是必须有的？"
- 延伸范围："什么可以后续做？"
- 明确排除："什么不做？"
- 参考标杆："有没有类似产品？"

### Step 4: 方案对比
- 基于需求生成 2-3 个可行方案
- 每个方案：技术栈 / 架构 / 工作量 / 风险
- 权衡矩阵：成本 / 速度 / 可维护性 / 扩展性
- 推荐方案 + 替代方案 + 渐进式路径

### Step 5: 设计验证
- 设计分段呈现（每段不超过屏幕一页）
- 每段请求用户确认，根据反馈迭代
- 边界压力测试：用户量 10 倍 / 数据量 100 倍 / 离线运行 / 安全要求提高
- 风险评估：技术风险 / 业务风险 / 运营风险

### Step 6: 输出设计文档
- 项目概述：目标 / 背景 / 范围
- 用户故事：角色 / 场景 / 流程
- 技术方案：架构 / 技术栈 / 数据模型
- 实现计划：阶段 / 任务 / 里程碑
- 风险与缓解：风险清单 / 应对措施
- 可视化：架构图 / 数据流图 / 时序图 / 部署图
- 保存为 `design.md`，版本化（日期/版本号），供后续 plan-architect 使用

## 错误处理

| 错误场景 | 原因 | 处理方式 |
|:------|------:|:------|
| 用户需求模糊 | 用户自己也没想清楚 | 继续提问，从不同角度探索，提供参考标杆帮助明确 |
| 用户急于编码 | 用户认为设计浪费时间 | 解释设计阶段价值，提供"快速设计模式"（15 分钟最小设计） |
| 方案过于复杂 | 用户要求过多功能 | 简化范围，推荐 MVP 先行，明确"后续迭代"清单 |
| 技术可行性存疑 | 选型未经验证 | 建议先做 PoC 验证，定义成功标准与时间盒（1-3 天） |
| 需求冲突 | 不同干系人需求矛盾 | 指出冲突点，请求优先级排序，提供权衡矩阵辅助决策 |
| 设计文档过长 | 用户无耐心阅读 | 分段呈现，每段不超过屏幕一页，请求逐段确认 |
| 联网调研失败 | 网络不可达或搜索无果 | 基于已有知识给出方案，标注"需人工验证" |

## 常见问题

### Q1: 苏格拉底式提问会不会让用户觉得啰嗦？
A: 提问聚焦关键决策点，避免无意义寒暄。每个问题都服务于"明确需求"目标。对于已有清晰需求的项目，可跳过部分提问直接进入方案对比。提供"快速模式"（3 个核心问题）和"深度模式"（全流程）。

### Q2: 设计文档输出后如何衔接实现阶段？
A: 设计文档保存为 `design.md`，可对接 `plan-architect` 技能将设计转为实现计划（任务拆分 + 里程碑 + 文件路径）。文档结构已对齐后续实现工具的输入格式。

### Q3: 方案对比时如何保证客观性？
A: 权衡矩阵使用 4 个维度（成本/速度/可维护性/扩展性）量化评分，每个方案列出优势与劣势。推荐方案必须说明理由，并提供替代方案（若推荐不可行）。避免单一方案的"确认偏误"。

### Q4: 没有 LLM 的环境能否使用？
A: 不能。本 Skill 的核心是 LLM 驱动的对话式提问与方案生成。纯静态文档无法实现动态交互。建议使用任意支持 Agent Skills 的 LLM 平台。

## 已知限制

- 依赖 LLM 能力，复杂领域知识（如金融/医疗合规）可能需要人工专家补充
- 设计文档质量取决于用户回答的完整度，模糊回答会导致方案偏差
- 不生成实际代码实现，仅输出设计文档与方案建议
- 联网调研能力受 Agent 平台限制，部分平台不支持联网
- 对话式流程在非交互式环境（如 CI/CD）中无法发挥全部价值

## 安全

- **API Key 零暴露**：本 Skill 不涉及任何外部 API 调用，无需配置密钥
- **敏感信息提醒**：提醒用户不在 AI 对话中输入商业机密、客户数据、财务细节
- **数据本地化**：所有设计文档保存在本地 `output/` 目录，不上传外部服务
- **假设标注**：基于不完整信息生成方案时，明确标注假设项，避免误导决策
