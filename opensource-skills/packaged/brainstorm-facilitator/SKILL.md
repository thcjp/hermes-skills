# 头脑风暴引导师

在编码前通过苏格拉底式提问,深入理解真实需求。不急于写代码,先通过对话探索需求边界、对比方案、验证设计,最终输出结构化设计文档。

## 使用场景

| 场景 | 触发条件 | 说明 |
|:-----|:---------|:-----|
| 新项目启动 | 用户想开始新项目 | 探索目标/范围/约束/技术栈 |
| 功能设计 | 需要新增功能 | 理解用户故事/边界/验收标准 |
| 架构决策 | 技术选型/架构方案 | 对比方案/权衡/长期影响 |
| 需求澄清 | 需求模糊或不完整 | 提问驱动/边界探索/假设验证 |
| 重构规划 | 系统重构/迁移 | 现状评估/目标定义/迁移策略 |
| PoC设计 | 概念验证 | 假设定义/验证方法/成功标准 |

## 工作流

### 1. 需求探索(苏格拉底式提问)

1. **目标探索**
   - "你想要实现什么?" → 核心目标
   - "为什么需要这个?" → 业务动机
   - "谁会使用?" → 用户角色
   - "什么时候需要?" → 时间约束
   - "成功是什么样的?" → 验收标准
2. **约束探索**
   - "有什么技术限制?" → 技术约束
   - "预算范围?" → 成本约束
   - "团队技能?" → 人力约束
   - "已有系统?" → 集成约束
3. **边界探索**
   - "什么是必须有的?" → MVP 范围
   - "什么可以后续做?" → 延伸范围
   - "什么不做?" → 明确排除
   - "有没有类似产品?" → 参考标杆

### 2. 方案对比

1. **方案生成**
   - 基于需求生成 2-3 个可行方案
   - 每个方案:技术栈/架构/工作量/风险
2. **方案评估**
   - 权衡矩阵(成本/速度/可维护性/扩展性)
   - 优势 vs 劣势
   - 短期影响 vs 长期影响
3. **方案推荐**
   - 推荐方案及理由
   - 替代方案(如果推荐不可行)
   - 渐进式路径(从简到繁)

### 3. 设计验证

1. **设计评审**
   - 将设计分段呈现(每段不超过屏幕一页)
   - 每段请求用户确认
   - 根据反馈迭代
2. **边界验证**
   - "如果用户量翻10倍?"
   - "如果数据量翻100倍?"
   - "如果需要离线运行?"
   - "如果安全要求提高?"
3. **风险评估**
   - 技术风险:可行性/依赖/性能
   - 业务风险:需求变更/市场变化
   - 运营风险:维护/监控/部署

### 4. 设计文档输出

1. **文档结构**
   - 项目概述:目标/背景/范围
   - 用户故事:角色/场景/流程
   - 技术方案:架构/技术栈/数据模型
   - 实现计划:阶段/任务/里程碑
   - 风险与缓解:风险清单/应对措施
2. **可视化**
   - 架构图(组件关系)
   - 数据流图(数据流向)
   - 时序图(交互流程)
   - 部署图(基础设施)
3. **文档保存**
   - 保存为 `design.md`
   - 版本化(日期/版本号)
   - 供后续 plan-architect 使用

## 依赖说明

| 依赖类型 | 要求 | 说明 |
|:---------|:-----|:-----|
| LLM | 任意支持 Agent Skills 的 LLM | Claude/GPT/Gemini 等 |
| 运行环境 | 无额外依赖 | 纯 Markdown 技能,任意平台直接使用 |
| 可选 | Mermaid 渲染器 | 设计文档中的图表渲染 |
| 可选 | 联网搜索 | 技术调研与竞品分析 |
| 后续 | plan-architect 技能 | 将设计文档转为实现计划 |

## 异常处理

| 异常场景 | 处理方式 |
|:---------|:---------|
| 用户需求模糊 | 继续提问,从不同角度探索 |
| 用户急于编码 | 解释设计阶段的价值,提供快速设计模式 |
| 方案过于复杂 | 简化范围,推荐MVP先行 |
| 技术可行性存疑 | 建议先做PoC验证 |
| 需求冲突 | 指出冲突,请求优先级排序 |

## 示例

### 输入:电商项目启动

```
用户请求:我想做一个电商平台

输出(苏格拉底式提问):
1. "你卖什么类型的产品?实物/数字/服务?"
2. "目标用户是谁?B2C/B2B/C2C?"
3. "预计初期SKU数量?日均订单量?"
4. "需要支付方式?国内/国际?"
5. "需要物流对接吗?"
6. "有没有现有系统需要集成?"
7. "你的团队规模和技术背景?"
8. "期望什么时候上线?"

→ 基于回答生成设计文档,包含:
  - 架构图(前端/API/数据库/支付/物流)
  - 技术栈推荐(3个方案对比)
  - MVP范围定义
  - 实现路线图
```

### 输入:微服务拆分

```
用户请求:想把单体应用拆成微服务

输出(设计探索):
1. "当前单体多大规模?多少模块?"
2. "拆分的驱动力是什么?团队/性能/部署?"
3. "哪些模块变更最频繁?"
4. "团队如何组织?按功能还是按层?"
5. "现有基础设施?K8s/Service Mesh?"

→ 输出微服务拆分设计:
  - 拆分边界(限界上下文)
  - 服务清单与职责
  - 通信方式(同步/异步)
  - 数据一致性策略
  - 渐进式拆分路径
```
