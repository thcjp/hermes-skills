# 调试医生

基于系统化调试方法论,进行根因分析与修复。不猜测,不试错,用科学方法找到真正的 Bug 根因,并确保修复有效。

## 使用场景

| 场景 | 触发条件 | 说明 |
|:-----|:---------|:-----|
| 复杂Bug | 难以定位的Bug | 4阶段科学调试法 |
| 生产事故 | 线上故障 | 快速止血 → 根因分析 → 防复发 |
| 间歇性故障 | 偶发性问题 | 条件等待 + 日志增强 |
| 性能问题 | 响应慢/内存高 | 性能剖析 → 瓶颈定位 → 优化 |
| 并发问题 | 竞态/死锁 | 线程分析 + 复现条件 |
| 回归Bug | 之前正常现在出错 | Git bisect 二分定位 |

## 工作流

### 阶段1: 复现(Reproduce)

1. **精确复现**
   - 记录完整复现步骤
   - 确定环境(操作系统/浏览器/版本/依赖)
   - 确定输入数据
   - 确定前置条件
2. **复现可靠性**
   - 每次都能复现?间歇性?
   - 间歇性:记录触发频率与条件
   - 不可复现:增加日志/监控,等待下次触发
3. **复现环境**
   - 本地复现(最佳)
   - Staging 复现
   - 生产日志分析(不可直接调试)

### 阶段2: 定位(Isolate)

1. **二分查找法**
   - 注释掉一半代码,问题是否消失?
   - 禁用一半模块,问题是否消失?
   - Git bisect:二分提交历史定位引入点
   - 数据二分:减少输入数据量
2. **日志增强**
   - 在关键路径添加日志
   - 记录变量值/执行路径/耗时
   - 使用条件断点
3. **工具辅助**
   - Debugger:逐步执行/变量监视
   - Profiler:性能剖析/内存分析
   - 网络抓包:Charles/Wireshark
   - 日志聚合:ELK/Grafana

### 阶段3: 缩减(Minimize)

1. **最小复现用例**
   - 移除不相关代码
   - 简化输入数据
   - 移除外部依赖
   - 创建独立测试用例
2. **根因确认**
   - 理解为什么这个 Bug 会发生
   - 确认根因而非症状
   - 检查是否有多个根因
   - 验证假设(修改后是否修复)

### 阶段4: 修复(Fix)

1. **修复实施**
   - 最小化修改范围
   - 不引入新问题
   - 保持代码风格一致
   - 添加注释说明修复原因
2. **验证闭环**
   - 编写回归测试(先验证测试失败)
   - 运行修复后的测试(验证通过)
   - 运行完整测试套件(无回归)
   - 代码审查
3. **防御深度**
   - 输入验证
   - 边界检查
   - 断言保护
   - 监控告警

## 高级调试技术

### 条件等待(Condition-Based Waiting)

1. **间歇性故障处理**
   - 在可疑代码处添加条件日志
   - 设置超时与重试
   - 使用 Watchpoint 监控变量变化
   - 记录线程状态快照

### 并发调试

1. **竞态条件**
   - Thread Sanitizer 检测
   - 日志记录线程ID与时间戳
   - 人工强制时序(Thread.sleep)
2. **死锁**
   - 线程转储(jstack/pstack)
   - 锁图分析(等待环)
   - 超时机制

### 性能调试

1. **CPU 瓶颈**
   - Flame Graph 火焰图
   - 采样剖析(perf/py-spy)
   - 热点函数定位
2. **内存泄漏**
   - 内存快照对比
   - 对象引用链分析
   - GC 日志分析
3. **I/O 瓶颈**
   - 磁盘 I/O 监控
   - 网络延迟分析
   - 数据库慢查询日志

## 异常处理

| 异常场景 | 处理方式 |
|:---------|:---------|
| 无法复现 | 增加日志覆盖率,部署监控,等待下次触发 |
| 修复后回归 | 立即回滚,重新分析根因,检查修复副作用 |
| 多个根因 | 逐个修复,每修一个验证一次 |
| 第三方库Bug | 确认版本,查Issue Tracker,寻找workaround或升级 |
| 环境差异 | 容器化确保一致性,使用Docker/Vagrant |
| 性能修复引入新Bug | 性能优化必须配套回归测试 |

## 示例

### 输入:间歇性API超时

```
用户请求:API间歇性超时(约5%请求),日志无明显错误

输出(4阶段调试):
阶段1-复现:在Staging用高并发模拟,复现率约3%
阶段2-定位:
  - 添加连接池日志:发现连接获取偶尔>5s
  - 数据库慢查询日志:特定查询偶发慢
  - 锁等待:表锁竞争
阶段3-缩减:
  - 最小用例:单表并发更新触发
  - 根因:缺失索引导致表扫描,锁升级为表锁
阶段4-修复:
  - 添加缺失索引
  - 验证:并发测试通过,超时消失
  - 防御:添加查询超时 + 连接池监控告警
  - 回归测试:并发更新测试用例
```

### 输入:内存泄漏

```
用户请求:Java服务每48小时OOM重启

输出:
阶段1-复现:本地压测72小时,内存持续增长
阶段2-定位:
  - jmap dump:堆内存快照
  - MAT分析:HashMap占80%内存
  - 引用链:缓存未设置过期时间
阶段3-缩减:
  - 最小用例:循环写入缓存,不读取
  - 根因:缓存无TTL,无LRU淘汰
阶段4-修复:
  - 改用Caffeine缓存(TTL=1h, maxSize=10000)
  - 验证:72小时压测内存稳定
  - 防御:添加内存使用率告警(80%)
  - 回归测试:缓存淘汰测试
```
