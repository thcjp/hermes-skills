# 法律助手Pro

为法律团队提供专业辅助。从合同审查到引文核验,从版本对比到状态报告,提升法务效率。

> **重要声明**:本技能为辅助工具,不构成法律意见。所有法律判断须由执业律师做出。

## 核心能力

1. **NDA分诊审查**:识别单向/双向NDA类型,逐条审查保密信息定义、保密期限、除外条款、返还销毁义务、违约救济、管辖法律等关键条款,按红/黄/绿三级风险标记并给出修改文本。
2. **多方合同版本对比**:收集原始版/对方修改版/我方修改版,逐条进行字词级diff,区分实质性变更与编辑性变更,评估每个变更的影响并给出接受/反驳/反建议。
3. **法律引文核验**:从文书中提取法条、判例、法规、司法解释等引文,核验引文是否存在、是否准确、是否现行有效、是否支持论述,输出核验报告与原始来源链接。
4. **会议简报生成**:结构化整理法律会议的讨论要点、决议事项、待办事项、分歧点,标注可能涉及律师-客户特权的内容,按责任人分组输出行动清单。
5. **状态报告合成**:从时间记录、邮件、文档、会议中汇总本周工作、下周计划、阻塞风险,按团队标准模板输出适合邮件/Slack分发的格式。

## 适用场景

| 场景 | 输入 | 输出 |
|:-----|:-----|:-----|
| NDA分诊审查 | NDA原文文档 | 风险标记报告+修改建议文本,输出到`output/{nda-id}/nda-review.md` |
| 多方版本对比 | 原始版+对方修改版+我方修改版 | 字词级diff+变更分类+影响评估+回复建议,输出到`output/{contract-id}/version-diff.md` |
| 法律引文核验 | 含引文的法律文书 | 每个引文的核验结果+原始来源链接,输出到`output/{doc-id}/citation-check.md` |
| 会议简报生成 | 会议记录/录音转写 | 结构化简报+特权标注+行动清单,输出到`output/{meeting-id}/briefing.md` |
| 周五状态合成 | 本周时间记录/邮件/文档 | 标准模板状态报告,输出到`output/{week}/status-report.md` |

**不适用于**:
- 替代执业律师出具正式法律意见
- 涉及刑事辩护、诉讼出庭等需要执业资质的事务
- 跨境法律冲突的最终判定(需当地律师确认)
- 实时法规更新追踪(法规变化频繁,需人工核验现行有效性)

## 使用流程

### Step 1: 确认场景与输入
1. 识别用户需求属于NDA审查/版本对比/引文核验/会议简报/状态报告中的哪一类
2. 确认输入文档格式(支持 .docx/.pdf/.txt/.md)
3. 确认输出路径规范(默认 `output/{业务id}/{文件名}.md`)

### Step 2: 执行对应工作流
- **NDA审查**:按"NDA分类识别→关键条款审查→风险标记→修改建议"四步执行
- **版本对比**:按"版本收集→逐条对比→变更分类→影响评估→建议回复"五步执行
- **引文核验**:按"引文识别→引文类型分类→四维核验→核验报告"四步执行
- **会议简报**:按"会议信息提取→内容结构化→特权标注→行动清单"四步执行
- **状态报告**:按"数据源收集→工作汇总→格式统一→分发准备"四步执行

### Step 3: 特权与保密处理
1. 识别并标注可能涉及律师-客户特权的内容
2. 律师工作产品标记
3. 敏感信息脱敏后再输出(当事人姓名、金额等可用占位符)

### Step 4: 输出与免责声明
1. 按规范路径输出文件
2. 所有输出末尾附"本文件不构成法律意见"免责声明
3. 记录审查操作日志,便于追溯

## 示例

### 示例1: NDA分诊审查

**输入**:
```
收到一份双向NDA,对方是技术供应商,需要审查风险。NDA原文(节选):
"保密信息包括接收方在合作期间接触到的所有披露方信息,保密期限为合作结束后10年,接收方不得以任何方式使用保密信息,违约赔偿上限为合同金额的100%。"
```

**输出** (`output/nda-2024-001/nda-review.md`):
```markdown
# NDA审查报告 - 双向NDA(技术供应商)

## 风险标记汇总
| 条款 | 风险等级 | 问题 |
|:-----|:---------|:-----|
| 保密信息定义 | 黄色 | "所有信息"范围过宽,应限定为标识为保密的信息 |
| 保密期限 | 红色 | 10年过长,技术类NDA建议3-5年 |
| 使用限制 | 红色 | "不得以任何方式使用"过严,应限定为"不得用于合作外目的" |
| 违约赔偿 | 黄色 | 100%合同金额上限可能不足,建议增加禁令救济条款 |

## 修改建议
### 保密期限(红色)
原文: 保密期限为合作结束后10年
建议: 保密期限为合作结束后5年,或根据信息类型分别设定(技术信息5年,商业信息3年)

### 使用限制(红色)
原文: 接收方不得以任何方式使用保密信息
建议: 接收方不得将保密信息用于本合作以外的任何目的,但可为履行合作义务而合理使用

## 免责声明
本文件不构成法律意见,所有法律判断须由执业律师做出。
```

**输入**:
```
原始版条款: "付款方式为签订后7日内支付50%,验收后7日内支付50%。"
对方修改版: "付款方式为签订后30日内支付30%,验收后30日内支付70%。"
```

**输出** (`output/contract-2024-001/version-diff.md`):
```markdown
# 版本对比报告

## 实质性变更
| 条款 | 原始版 | 对方修改版 | 变更类型 | 影响 |
|:-----|:-------|:-----------|:---------|:-----|
| 付款方式 | 7日内付50%+7日内付50% | 30日内付30%+30日内付70% | 实质性 | 现金流延迟23天,首付款比例降低20% |

## 建议回复
- 接受: 30日付款周期(行业惯例)
- 反驳: 首付款比例应保持50%,以保障我方现金流
- 反建议: "签订后30日内支付50%,验收后30日内支付50%"
```

## 错误处理

| 错误场景 | 原因 | 处理方式 |
|:---------|:-----|:---------|
| NDA类型无法识别 | 文档缺少标准NDA结构或为混合协议 | 标注"非标准NDA",按最接近类型审查并人工复核 |
| 版本对比diff混乱 | 多方修改同一条款且格式不一致 | 先统一格式再diff,或逐句人工对齐 |
| 引文核验无法联网 | 网络不可达或法律数据库访问受限 | 标注"待人工核验",列出需查询的引文清单 |
| 会议记录不完整 | 录音转写缺失或会议纪要简略 | 标注信息缺口,要求参会人补充 |
| 法规已修订或废止 | 引用旧版法规未及时更新 | 标注"可能已修订",提示查询最新版本 |
| 文档加密无法读取 | PDF/DOCX设有密码保护 | 要求提供密码或解密版本,不支持暴力破解 |
| 特权标注误判 | AI无法准确判断律师-客户特权边界 | 所有特权标注均提示"需律师最终确认" |

### 案例1: 复杂双向NDA分诊审查(技术合作场景)

**输入**:
```
场景:与AI芯片供应商签订双向NDA,涉及源代码共享
NDA原文(节选):
"1. 保密信息包括双方在合作期间披露的所有技术资料、商业信息及口头沟通内容。
2. 保密期限为合作终止后永久有效。
3. 接收方应将保密信息仅限于'必要人员'知悉,但必要人员定义由接收方自行决定。
4. 违约赔偿上限为直接损失的50万元,不含间接损失和利润损失。
5. 管辖法律为供应商所在地(美国加州)法律。
6. 接收方在合作终止后应于30日内返还或销毁所有保密信息。"
```

**LLM生成输出** (`output/nda-2024-015/nda-review.md`):
```markdown
# NDA审查报告 - 双向NDA(AI芯片技术合作)

## NDA分类
- 类型:双向NDA(互惠保密协议)
- 场景:技术合作,涉及源代码与AI芯片设计资料共享
- 风险等级:中高风险(涉及核心技术+跨境管辖)

## 风险标记汇总
| 条款 | 风险等级 | 核心问题 |
|:-----|:---------|:---------|
| 保密信息定义 | 黄色 | "所有信息"范围过宽,缺少保密标识要求 |
| 保密期限 | 红色 | "永久有效"对技术信息合理,但商业信息应设期限 |
| 必要人员 | 红色 | "由接收方自行决定"导致控制权缺失 |
| 违约赔偿 | 红色 | 50万上限过低,芯片技术泄露损失远超此数 |
| 管辖法律 | 黄色 | 加州法律对我方不利,建议改为中立地或中国法 |
| 返还销毁 | 绿色 | 30日返还/销毁义务合理 |

## 逐条修改建议

### 第1条 保密信息定义(黄色)
**原文**: 保密信息包括双方在合作期间披露的所有技术资料、商业信息及口头沟通内容。
**问题**: "所有信息"范围过宽,口头沟通难以举证。
**建议修改**: "保密信息指披露方以书面、电子或标注'保密'形式披露的信息,以及口头披露后7日内以书面确认的信息。以下信息不属于保密信息:(a)已公开知悉的;(b)接收方独立开发的;(c)第三方合法提供的。"

### 第2条 保密期限(红色)
**原文**: 保密期限为合作终止后永久有效。
**问题**: 商业信息永久保密不合理,且增加管理成本。
**建议修改**: "技术信息(含源代码、芯片设计)保密期限为合作终止后7年;商业信息(含定价、客户名单)保密期限为合作终止后3年。"

### 第3条 必要人员(红色)
**原文**: 必要人员定义由接收方自行决定。
**问题**: 接收方自行决定等于无限制,核心芯片技术可能扩散。
**建议修改**: "必要人员指为履行本合作协议而必须知悉保密信息的员工,且须签署不低于本协议保密义务的个别保密协议。接收方应每季度向披露方提供必要人员清单。"

### 第4条 违约赔偿(红色)
**原文**: 违约赔偿上限为直接损失的50万元,不含间接损失和利润损失。
**问题**: 50万远低于芯片技术泄露的潜在损失,排除间接损失对披露方极不利。
**建议修改**: "违约赔偿包括直接损失和可合理预见的间接损失。赔偿上限为[合作金额的200%]或[500万美元],以较高者为准。恶意违约不受上限约束。"

### 第5条 管辖法律(黄色)
**原文**: 管辖法律为美国加州法律。
**问题**: 加州法律对我方不熟悉,诉讼成本高。
**建议修改**: "管辖法律为中华人民共和国法律。争议提交[中国国际经济贸易仲裁委员会]仲裁,仲裁地为北京。"

## 整体评估
- 可否签署: 否,需完成上述红色条款修改后方可签署
- 优先修改: 第3条(必要人员)>第4条(违约赔偿)>第2条(保密期限)
- 特别提示: 涉及AI芯片核心技术,建议同步签署知识产权归属协议,明确衍生技术成果权属

## 免责声明
本文件不构成法律意见,所有法律判断须由执业律师做出。
```

**效果验证**: ✓NDA类型正确识别(双向+技术合作) ✓红/黄/绿三级风险标记准确 ✓每条给出原文+问题+修改建议 ✓优先修改顺序合理 ✓附加IP保护提示(超越基础审查)

### 案例2: 三方合同版本对比(采购合同谈判)

**输入**:
```
合同类型:IT设备采购合同
我方角色:采购方(甲方)
原始版条款: "甲方应在设备验收合格后15个工作日内支付合同总价的100%。质保期为验收后12个月。"
供应商修改版: "甲方应在设备交付后60个工作日内支付合同总价的100%。质保期为交付后12个月。"
我方修改版: "甲方应在设备验收合格后30个工作日内支付合同总价的100%。质保期为验收后18个月。"
```

**LLM生成输出** (`output/contract-2024-088/version-diff.md`):
```markdown
# 三方版本对比报告 - IT设备采购合同

## 变更概览
| 维度 | 原始版 | 供应商版 | 我方版 | 争议点 |
|:-----|:-------|:---------|:-------|:-------|
| 付款触发 | 验收合格 | 设备交付 | 验收合格 | 验收 vs 交付 |
| 付款周期 | 15工作日 | 60工作日 | 30工作日 | 周期长短 |
| 质保起算 | 验收后 | 交付后 | 验收后 | 起算时点 |
| 质保期限 | 12个月 | 12个月 | 18个月 | 期限长短 |

## 逐项变更分析

### 1. 付款触发条件
| 版本 | 内容 | 变更类型 |
|:-----|:-----|:---------|
| 原始版 | 验收合格后付款 | 基准 |
| 供应商版 | 设备交付后付款 | 实质性变更(对甲方不利) |
| 我方版 | 验收合格后付款 | 维持原始版 |

**影响评估**: 供应商将"验收合格"改为"设备交付",意味着甲方在未完成验收、可能存在质量问题时就需付款,丧失质量异议的付款筹码。交付到验收通常间隔2-4周,影响资金安全。
**建议回复**: 坚持验收合格作为付款触发条件。理由:(1)验收是确认设备符合合同规格的必要环节;(2)交付仅表示物理到货,不代表功能达标;(3)行业惯例以验收为付款前提。

### 2. 付款周期
| 版本 | 内容 | 变更类型 |
|:-----|:-----|:---------|
| 原始版 | 15工作日 | 基准 |
| 供应商版 | 60工作日 | 实质性变更(对甲方有利) |
| 我方版 | 30工作日 | 实质性变更(折中) |

**影响评估**: 供应商提出60工作日比原始版15工作日更长,表面对甲方有利(更晚付款),但结合供应商将触发条件改为"交付",实际净效果是:从交付到付款共60工作日,而从验收到付款可能是30工作日(我方版)。两者实际差异不大,但供应商版的风险在于付款不以验收为前提。
**建议回复**: 接受30工作日(我方版)作为折中方案。理由:(1)30工作日为IT采购行业惯例;(2)配合验收触发条件,30日足够完成财务审批流程。

### 3. 质保起算时点
| 版本 | 内容 | 变更类型 |
|:-----|:-----|:---------|
```
