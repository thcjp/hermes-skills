# 计划架构师

将设计文档转化为可执行的详细实现计划。每个任务都是 2-5 分钟可完成的粒度,包含精确的文件路径、完整的代码片段、明确的验证步骤。让一个"热情但缺乏判断力的初级工程师"也能按计划执行。

## 使用场景

| 场景 | 触发条件 | 说明 |
|:-----|:---------|:-----|
| 项目实施 | 设计文档已批准 | 将设计转为可执行计划 |
| 功能开发 | 新功能开发 | 拆分任务/TDD驱动/验证步骤 |
| 重构计划 | 系统重构 | 渐进式重构/安全步骤/回滚策略 |
| 迁移计划 | 技术迁移 | 数据迁移/服务切换/回滚方案 |
| Bug修复计划 | 复杂Bug修复 | 根因修复/测试覆盖/防御措施 |
| 技术债务 | 债务清理 | 优先级排序/渐进清理/验证 |

## 工作流

### 1. 计划输入分析

1. **设计文档解析**
   - 读取 `design.md` 或设计输入
   - 提取:目标/范围/技术栈/数据模型/架构图
   - 识别依赖关系与执行顺序
2. **约束确认**
   - 技术栈版本
   - 团队规模与技能
   - 时间约束
   - 测试要求

### 2. 任务拆分(2-5分钟粒度)

1. **拆分原则**
   - 每个任务 2-5 分钟可完成
   - 每个任务有明确的"完成"定义
   - 每个任务可独立验证
   - 每个任务有精确的文件路径
2. **拆分维度**
   - 按功能模块拆分
   - 按层级拆分(数据层→服务层→API层→UI层)
   - 按测试优先拆分(TDD:先测试后实现)
3. **依赖排序**
   - 无依赖任务可并行
   - 有依赖任务按序执行
   - 标注阻塞关系

### 3. 任务描述格式

每个任务包含:

```
### Task N: [任务名称]
- **文件**: `src/path/to/file.ts` (新建/修改)
- **描述**: [具体做什么]
- **代码**: [完整代码片段或关键改动]
- **验证**: [如何验证这个任务完成]
  - 命令: `npm test -- --grep "test name"`
  - 预期: 测试通过
- **依赖**: Task M (如果有)
- **时间预估**: ~3分钟
```

### 4. TDD 驱动计划

1. **红-绿-重构循环**
   - 先写失败测试(Red)
   - 再写最小实现(Green)
   - 最后重构(Refactor)
2. **测试策略**
   - 单元测试:每个函数/方法
   - 集成测试:模块间交互
   - 端到端测试:关键用户流程
   - 测试覆盖率目标:80%+
3. **测试金字塔**
   - 80% 单元测试(快速/隔离)
   - 15% 集成测试(模块交互)
   - 5% E2E 测试(用户流程)

### 5. YAGNI 与 DRY 原则

1. **YAGNI(You Aren't Gonna Need It)**
   - 只实现当前需要的功能
   - 不提前实现"可能需要"的功能
   - 不过度设计接口
   - 不添加未使用的配置项
2. **DRY(Don't Repeat Yourself)**
   - 重复代码提取为函数
   - 重复配置提取为常量
   - 重复模式提取为组件
   - 但不过度抽象(Rule of Three)

### 6. 计划输出

1. **计划文档**
   - 保存为 `plan.md`
   - 任务列表(带编号)
   - 依赖关系图
   - 时间预估总计
2. **执行检查清单**
   - 每个任务完成后勾选
   - 验证命令一键运行
   - 回滚步骤(如需要)

## 依赖说明

| 依赖类型 | 要求 | 说明 |
|:---------|:-----|:-----|
| LLM | 任意支持 Agent Skills 的 LLM | Claude/GPT/Gemini 等 |
| 运行环境 | 无额外依赖 | 纯 Markdown 技能 |
| 前置 | brainstorm-facilitator 输出的设计文档 | 计划基于设计文档制定 |
| 后续 | plan-executor 或 subagent-orchestrator | 执行实现计划 |
| 测试框架 | 项目对应的测试框架 | Jest/pytest/Go test 等 |
| 可选 | Git | 版本控制,每个任务一个提交 |

## 异常处理

| 异常场景 | 处理方式 |
|:---------|:---------|
| 设计文档不完整 | 标注缺失部分,建议补充后再计划 |
| 任务太大无法拆分 | 进一步分解,或拆为多阶段 |
| 依赖不确定 | 标注假设,提供替代路径 |
| 测试无法编写 | 先写最小验证(日志/手动测试),标注技术债 |
| 计划执行偏差 | 允许调整,但记录偏差原因 |
