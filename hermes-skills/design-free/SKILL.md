# Design LITE

设计偏好基础版,通过观察用户在 UI 设计中的选择与反馈,自动学习视觉偏好。支持 Aesthetic 与 Never 两大分类,偏好档案随使用逐步积累。

**范围外**（本技能不做）: 多媒介适配（图形/视频/印刷）、品牌专属偏好、dimensions.md 维度参考、criteria.md 格式规范（需升级付费版）。

## 核心能力

- **偏好自动检测**: 从用户的选择和反馈中识别 UI 视觉偏好模式
- **两大偏好分类**: Aesthetic（美学风格）、Never（禁忌清单）
- **UI 媒介适配**: 专注 UI 界面设计偏好记忆
- **稳定确认**: 检测到 3 次以上一致偏好后自动确认写入
- **精简条目**: 保持偏好条目紧凑,每条不超过 1 行
### 偏好自动检测

执行偏好自动检测操作,处理用户输入并返回结果。

**输入**: 用户提供偏好自动检测所需的参数和指令。

**输出**: 返回偏好自动检测的处理结果。- 验证执行结果，确认输出符合预期格式
- 参考`偏好自动检测`相关配置参数进行设置
### 两大偏好分类

执行两大偏好分类操作,处理用户输入并返回结果。

**输入**: 用户提供两大偏好分类所需的参数和指令。

**输出**: 返回两大偏好分类的处理结果。- 验证执行结果，确认输出符合预期格式
- 参考`两大偏好分类`相关配置参数进行设置
### UI 媒介适配

执行UI 媒介适配操作,处理用户输入并返回结果。

**输入**: 用户提供UI 媒介适配所需的参数和指令。

**输出**: 返回UI 媒介适配的处理结果。- 验证执行结果，确认输出符合预期格式

## 偏好档案结构

偏好档案分为两大分区,初始为空,随使用自动填充:

### Aesthetic（美学风格）

记录用户对 UI 美学元素的偏好。

示例条目格式:
```
- 配色: 偏好低饱和度暖色调,避免高对比霓虹色
- 字体: 无衬线优先,标题用 Inter
- 圆角: 8-12px,阴影柔和
```

### Never（禁忌清单）

记录用户明确不喜欢的视觉元素:

```
- 不用: 渐变背景、毛玻璃效果
- 避免: 全大写标题、斜体正文
```

> **升级提示**: 按媒介细分（UI/图形/视频/印刷）、品牌专属偏好等高级分类仅在 design 付费版中提供。

## 偏好学习机制

### 检测信号

从以下用户行为中提取偏好信号:

1. **选择行为**: 用户在多个方案中选择某一个时,记录被选方案的视觉特征
2. **反馈表达**: 用户对设计输出给出评价时,提取评价指向的视觉元素
3. **修改行为**: 用户要求修改某个视觉元素时,记录修改前后的差异

### 确认规则

- 同一偏好方向出现 **3 次以上** 一致信号后,自动确认写入偏好档案
- 仅 1-2 次出现的信号标记为"待观察",不写入正式档案
- 出现矛盾信号时,保留两条记录并标注冲突
- 已确认的偏好可被用户显式覆盖或删除

> 付费版确认阈值为 2 次以上,学习速度更快。

### 条目维护规则

- **精简紧凑**: 每条偏好不超过 1 行
- **具体可执行**: 记录具体数值或明确指令（如"圆角 8px"而非"圆角适中"）
- **分类归档**: 按 Aesthetic / Never 两区归档
- **定期整理**: 当条目超过 20 条时,合并相似条目

## 适用场景

### 场景一: UI 配色偏好记忆

设计师在多个项目中反复选择低饱和度配色,引擎检测到 3 次以上一致偏好后写入 Aesthetic 分区,后续 UI 设计自动遵循该风格。

### 场景二: 禁忌清单积累

用户多次要求移除毛玻璃效果,引擎在 3 次确认后写入 Never 分区,后续设计不再使用该元素。

**不适用于**: 多媒介偏好（图形/视频/印刷）、品牌专属偏好管理、维度系统化参考（需升级付费版）

## 使用流程

1. 检查偏好档案是否为空,为空则进入观察模式
2. 在 UI 设计交互中观察用户的选择、反馈与修改行为
3. 提取视觉特征信号,与已有偏好比对
4. 同一偏好方向出现 3 次以上一致信号后,写入对应分区
5. 后续设计输出前,先读取偏好档案,确保输出符合已确认偏好
6. 用户显式否定某偏好时,从档案中移除或标注冲突

## 案例展示

### 案例一： UI 配色偏好学习

**场景**: 设计师在连续 3 个 UI 项目中都选择了低饱和度暖色调配色方案

**观察记录**:
```
信号1: 项目A - 用户选择了 #f5f0eb 背景,反馈"这个配色舒服"
信号2: 项目B - 用户从方案中选了暖灰系,弃选高对比方案
信号3: 项目C - 用户要求将冷蓝配色改为暖棕色调
```

**确认写入** Aesthetic 分区:
```
- 配色: 偏好低饱和度暖色调,避免高对比霓虹色
```

**说明**: 3 次一致信号达到确认阈值,自动写入。后续 UI 设计默认使用低饱和暖色调。

### 案例二： 禁忌清单积累

**场景**: 用户在多次设计中明确表示不喜欢毛玻璃效果

**观察记录**:
```
信号1: 设计稿A - 用户要求移除毛玻璃卡片,反馈"太花哨"
信号2: 设计稿B - 用户否决含毛玻璃的方案
信号3: 设计稿C - 用户再次要求移除毛玻璃
```

**确认写入** Never 分区:
```
- 不用: 毛玻璃效果（backdrop-blur）
```

**说明**: 3 次否定信号达到阈值,确认写入禁忌清单。后续设计不再使用毛玻璃效果。

## 异常处理

| 错误场景 | 错误信息 | 原因分析 | 处理方式 |
|---------|---------|---------|---------|
| preference_conflict | `Conflict: 圆角 vs 直角` | 同一维度出现矛盾偏好信号 | 保留两条记录并标注冲突,提示用户显式选择 |
| entry_format_error | `Invalid entry format` | 偏好条目格式不符合规范 | 按 `- 属性: 具体描述` 格式重写后写入 |
| archive_overflow | `Archive exceeds 20 entries` | 偏好条目超过 20 条上限 | 触发整理流程,合并相似条目 |
| medium_not_supported | `Unsupported medium: xxx` | 媒介类型不在 UI 范围内 | 引导用户仅记录 UI 相关偏好,其他媒介需升级付费版 |
| signal_too_weak | `Signal strength below threshold` | 仅 1-2 次出现且反馈强度不足 | 标记为"待观察",不写入正式档案 |

## 常见问题

### Q1: 偏好档案什么时候开始有内容?
A: 偏好档案初始为空。引擎在用户进行 UI 设计交互时观察选择与反馈,同一偏好方向出现 3 次以上一致信号后自动写入。通常经过 3-4 次交互后开始有初始条目。

### Q2: 免费版和付费版有什么区别?
A: 免费版（LITE）支持 UI 媒介偏好学习、Aesthetic 与 Never 两大分类、3 次确认阈值。付费版（design）额外提供:
- 多媒介适配（UI/图形/视频/印刷）
- 四大偏好分类（含 By Medium 按媒介细分、Brands 品牌专属）
- 2 次确认阈值（学习速度更快）
- dimensions.md 维度参考与 criteria.md 格式规范
- 3 个完整案例（vs 免费版 2 个基础案例）
- 8 种异常处理（vs 免费版 5 种）

### Q3: 为什么需要 3 次确认?
A: 免费版采用 3 次确认阈值,确保偏好记录稳定可靠,避免偶发选择被误判为长期偏好。付费版将阈值降低到 2 次,学习速度更快。用户也可以随时手动添加偏好条目。

### Q4: 可以手动编辑偏好档案吗?
A: 可以。用户可以显式添加、修改或删除偏好条目。手动编辑的条目优先级与自动检测的条目相同。建议每条偏好保持 1 行精简格式。

## 错误处理


| 错误场景 | 原因 | 处理方式 |
|---------|------|---------|
| LLM响应超时或无响应 | 网络延迟或模型负载过高 | 执行ping命令测试网络连通性,检查防火墙和代理设置连接，执行ping命令测试网络连通性,检查防火墙和代理设置连接后重新执行命令请求；确认Agent平台LLM服务正常 |
| 输入内容格式不正确 | 用户输入不符合skill预期格式 | 检查输入是否符合skill使用说明中的格式要求，参考示例章节 |
| 执行结果与预期不符 | 指令描述不够明确或上下文不足 | 提供更详细的指令描述，补充必要的上下文信息 |
| 命令执行失败 | 运行环境不满足要求或权限不足 | 确认运行环境符合依赖说明中的要求；检查命令权限设置 |

## 已知限制

1. **仅 UI 媒介**: 不支持图形/视频/印刷等媒介偏好（需升级付费版）
2. **仅两大分类**: 不含 By Medium 与 Brands 分区（需升级付费版）
3. **确认阈值较高**: 需 3 次以上一致信号,学习速度慢于付费版的 2 次
4. **无维度参考**: 不含 dimensions.md 与 criteria.md 参考文件
5. **条目上限 20 条**: 超过后触发整理,少于付费版的 50 条上限
6. **需 LLM 支持**: 偏好检测依赖 LLM 理解用户反馈语义
