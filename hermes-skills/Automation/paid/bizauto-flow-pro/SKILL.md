# 业务自动化师（专业版）

> **企业级业务自动化架构系统。五阶段全生命周期，事件驱动+人机协同+五级容错，自动化永不中断。**

每个业务都运行在可重复的流程上。专业版不仅帮你找到瓶颈、设计自动化、实现监控，更提供事件驱动架构、人机协同审批、五级容错降级与多系统同步策略，确保自动化在企业环境中稳定运行、永不中断.
## 核心理念：5x规则

**只自动化满足以下条件之一的流程**：
- 每周至少发生5次
- 单次耗时超过30分钟

否则自动化的成本将超过手动工作的成本。这是所有自动化决策的起点.

直接告诉Agent你的痛点：

```text
帮我审计业务中的自动化机会，我的团队每天处理大量发票和客户邮件
```

### 标准搭建（<120秒）

1. **发起审计**：使用快速开始命令
2. **填写流程映射模板**：Agent生成YAML结构化记录
3. **查看打分结果**：四维矩阵打分，输出优先级排序
4. **设计工作流**：对最高分流程，生成含并行分支与决策点的架构

## 核心能力
### 功能一：业务流程审计

通过5个发现性问题，映射你的流程全景：

1. 团队最重复的5项任务是什么？
2. 哪些环节经常卡在等人处理？（瓶颈）
3. 哪些任务需要在系统间复制数据？（集成点）
4. 有人请假时什么会中断？（单点故障）
5. 哪些报表是手动生成的？（报表自动化）

### 功能二：四维打分矩阵

| 维度 | 0分 | 1分 | 2分 | 3分 |
|:-----|:-----|:-----|:-----|:-----|
| 频率 | 每月 | 每周 | 每天 | 每天多次 |
| 时间成本 | <5分钟 | 5-15分钟 | 15-60分钟 | >1小时 |
| 错误影响 | 无关紧要 | 需返工 | 客户可见 | 收入损失 |
| 复杂度 | 5+决策 | 3-4决策 | 1-2决策 | 纯规则 |

**打分结果解读**：
- 10-12分：立即自动化，最高ROI
- 7-9分：强候选，下个迭代计划
- 4-6分：考虑部分自动化
- 0-3分：跳过，手动即可

**输入**: 用户提供功能二：四维打分矩阵所需的指令和必要参数.

  - 实时处理（无轮询延迟）

### 功能五：人机协同审批门 — 专业版启用

并非所有流程都应全自动。设计审批门：

```yaml
approval_gate:
  name: "经理审批"
  trigger: "amount > 5000 OR new_vendor = true"
  action:
    - Send approval request via Slack/email
    - Include: summary, amount, context, approve/reject buttons
    - Set deadline: 24 hours
  on_approve: "continue_workflow"
  on_reject: "notify_requestor_with_reason"
  on_timeout:
    - Escalate to next level
    - Or: auto-approve if amount < 10000
```

**审批门设计原则**：
- 高风险决策保留人工审批（金额阈值、新供应商、异常情况）
- 审批请求包含足够上下文（摘要、金额、原因、影响范围）
- 设置审批截止时间，超时自动升级或按规则处理
- 审批结果记录归档，支持追溯

### 功能六：五级容错降级 — 专业版启用

每个自动化都应优雅处理失败：

```text
Level 1: 重试（瞬时错误 — API超时、速率限制）
Level 2: 回退（使用缓存数据、替代API、简化逻辑）
Level 3: 队列（保存待服务恢复后处理）
Level 4: 告警（通知人工，提供上下文与建议修复）
Level 5: 安全停止（暂停工作流，保留状态，不丢数据）
```

fault_tolerance:
  level_1_retry:
    max_attempts: 3

    use_cached_data: true
    alternative_api: "backup_endpoint"
    simplified_logic: true
  level_3_queue:
    storage: "local_queue"
    retry_after: "300s"
  level_4_alert:
    channel: "slack"
    include_context: true
    suggest_fix: true
  level_5_safe_stop:
    preserve_state: true
    notify_admin: true
    resume_on_restart: true

**ROI计算公式**：

```text
月度ROI = (节省小时数 x 时薪) - 自动化成本
# ...

  节省小时数 = 频率 x 单次时间 x 成功率
  时薪 = 员工成本 / 工作小时
  自动化成本 = 工具费用 + 维护小时 x 时薪
# ...
示例：
  流程：发票处理
  改造前：50张/周 x 12分钟 = 10小时/周 = 40小时/月

  节省：36.7小时/月
  按80元/小时：2,936元/月节省
  自动化成本：2小时/月维护 x 80 = 160元/月
  净ROI：2,776元/月 = 33,312元/年
```

## 性能优化策略

### 事件驱动优化

1. **事件分区**：按业务域分区事件流，避免单队列瓶颈
2. **批处理消费者**：低优先级事件批量消费，减少IO开销
3. **背压控制**：消费者过载时自动降速，避免雪崩
4. **死信队列**：无法处理的事件进入死信队列，人工介入

### 工作流优化

1. **并行化**：识别无依赖步骤，并行执行缩短P95延迟
2. **检查点**：每10-50步保存状态，中断后可恢复
3. **幂等设计**：每步设计为可重试，避免重复执行副作用
4. **超时调优**：根据历史P95设置合理超时，避免过长等待

### 同步优化

1. **增量同步**：仅同步变更部分，非全量同步
2. **压缩传输**：同步数据启用压缩，节省带宽
3. **批量写入**：目标系统支持批量API时，合并写入
4. **异步确认**：写入后异步确认，不阻塞主流程

### 成本控制

- 低优先级流程使用非高峰时段调度
- 事件消费者按需启停，非持续运行
- 缓存频繁访问的参考数据，减少API调用
- 定期审查自动化ROI，淘汰负ROI流程

## 边界情况与注意事项

- **时区混乱**：内部始终用UTC存储时间，仅在显示/通知时转换。测试夏令时切换前后.
- **速率限制**：跟踪API调用次数，实现退避，尽可能批量请求，缓存响应.
- **部分失败**：设计幂等性，确保5步中的第3步失败后可从第3步恢复.
- **数据增长**：100条记录能跑的自动化，1万条可能崩溃。规划分页、分块、归档.
- **凭据轮换**：API会更换密钥，构建认证失败告警，在全面崩溃前知晓.
- **模式变更**：外部API增删字段，防御性验证输入，不在意外数据上崩溃.
- **重复处理**：使用幂等键，处理前检查"已处理"，特别是付款和邮件.
- **测试自动化**：始终用真实（但安全）的数据测试。dry-run模式用于发邮件、扣款、改生产数据的场景.
- **金丝雀发布**：新自动化先对小范围数据运行，确认无误后全量.
- **回滚预案**：每个自动化都应有回滚方案，特别是修改生产数据的流程.

## FAQ

### Q1：免费版与专业版有什么区别？

免费版提供审计、设计、基础实现和监控四大核心能力。专业版新增五大高级功能：事件驱动架构（实时事件总线、扇出多消费者）、人机协同审批门（阈值审批、超时升级、自动批准）、五级容错降级（重试→回退→队列→告警→安全停止）、多系统同步策略（事件溯源、冲突解决、周对账）、ROI仪表盘（成功率、P95延迟、成本节省、准确率）。此外提供7种角色场景指南、性能优化策略、多平台集成示例与完整故障排查表.
### Q2：事件驱动架构与定时轮询有什么区别？

定时轮询按固定间隔检查是否有新数据，存在延迟和资源浪费。事件驱动架构在数据产生时立即触发，实现实时处理。例如，发票到达时立即触发处理，而非每5分钟检查一次。事件驱动还支持扇出（一个事件触发多个消费者）和内置审计追踪.
### Q3：人机协同审批门如何设计？

审批门通过条件触发（如金额>5000、新供应商、异常情况），发送包含上下文的审批请求（摘要、金额、原因、影响范围），设置截止时间，超时后自动升级或按规则处理。审批结果记录归档，支持追溯。核心原则是"高风险决策保留人工，低风险操作全自动".
### Q4：五级容错降级如何工作？

Level 1重试（瞬时错误，指数退避3次）→ Level 2回退（缓存数据、替代API、简化逻辑）→ Level 3队列（保存待恢复后处理）→ Level 4告警（通知人工，提供上下文与建议）→ Level 5安全停止（暂停工作流，保留状态，不丢数据）。每级自动升级，确保自动化在不稳定环境中仍尽可能完成任务.
### Q5：多系统同步如何避免数据不一致？

采用事件溯源模式：所有变更记录为事件，每个系统订阅相关事件，指定唯一数据源，单向同步优于双向，需双向时使用冲突解决策略（最后写入优先或手动合并），每周运行对账比较系统差异。核心是"每种数据类型只有一个权威来源".
### Q6：ROI仪表盘包含哪些指标？

可靠性指标（总运行数、成功数、失败数、成功率、平均耗时、P95耗时）、影响指标（节省小时数、自动化任务数、防止错误数、节省成本）、质量指标（误判数、遗漏数、人工修正数、准确率）、告警（本期问题）、优化建议（模式发现与改进建议）.
### Q7：如何测试自动化工作流？

建议三阶段测试：(1) dry-run模式模拟执行，不实际发送邮件或修改数据；(2) 小范围真实数据测试（如5条记录）；(3) 金丝雀发布，对小比例数据真实执行，确认无误后全量。每个阶段都应检查输出正确性与监控指标.
### Q8：如何处理自动化的回归问题？

建立自动化测试套件，每次修改工作流后运行回归测试。保留历史执行数据，对比修改前后的成功率与P95延迟。使用版本管理（workflow.version字段），出问题时可回滚到上一版本。定期审查自动化健康度，淘汰负ROI流程.
### Q9：事件驱动架构需要什么基础设施？

最小配置仅需Agent平台的内部事件队列。企业级配置可集成消息队列（如Redis Streams、RabbitMQ）。专业版支持内部事件总线，无需额外基础设施即可实现扇出与审计追踪。高吞吐场景建议外接消息队列.
### Q10：审批门可以配置多级审批吗？

可以。通过on_approve链式配置实现多级审批：第一级审批通过后触发第二级审批请求。每级可设置不同的审批人、条件与超时规则。例如：金额>1万需经理审批，>5万需总监审批，>10万需CEO审批.
### Q11：如何确保自动化的合规性？

启用compliance_logging记录所有操作（谁、何时、做了什么、结果如何），审批门保留审批人、审批时间、审批理由，事件溯源提供完整审计追踪，归档所有执行日志与输出，定期生成合规报表。所有日志不可篡改，支持外部审计.
