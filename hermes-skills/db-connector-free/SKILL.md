## 概述

数据库是绝大多数业务系统的核心基础设施，但连接管理不当、事务设计缺陷、索引策略失误等问题往往在流量增长后才暴露。本助手将数据库常见陷阱整理为结构化知识库，帮助开发者在编码阶段就规避生产事故。

## 示例

### 连接池配置参考

以 `PostgreSQL` 为例，使用 HikariCP 的推荐配置：

```yaml
# HikariCP 推荐配置
maximumPoolSize: 20          # 根据 (核心数 * 2 + 有效磁盘数) 估算
minimumIdle: 10              # 空闲连接下限
connectionTimeout: 30000     # 获取连接超时 30s
idleTimeout: 600000          # 空闲连接超时 10min
maxLifetime: 1800000         # 连接最大生命周期 30min
leakDetectionThreshold: 60000 # 连接泄漏检测阈值 60s
```

### 事务隔离级别选择

| 隔离级别 | 适用场景 | 注意事项 |
|:---------|:---------|:---------|
| READ COMMITTED | 默认选择，大多数业务 | 可能出现不可重复读 |
| REPEATABLE READ | 报表统计、一致性快照 | 需注意写偏序问题 |
| SERIALIZABLE | 严格一致性要求 | 性能开销大，需重试机制 |

## 最佳实践

### 实践一：事务保持简短

长事务会持有锁、阻止 VACUUM 清理死元组，导致表膨胀。建议将事务拆分为小批次，避免在事务中执行外部调用。

### 实践二：Schema 变更两阶段发布

1. **第一阶段**：发布兼容代码（新列可空、新索引 CONCURRENTLY 创建）
2. **第二阶段**：确认无报错后，发布使用新结构的代码

### 实践三：金额使用 DECIMAL

浮点数存储金额会产生精度丢失，应使用 DECIMAL 或整数分存储。

### 实践四：时区统一存 UTC

所有时间戳以 UTC 存储，仅在展示层转换为本地时区，避免跨时区混乱。

### 实践五：外键必须建索引

未建索引的外键会拖慢 JOIN 与级联删除，应在创建外键约束时同步创建索引。

### 实践六：批量操作分批次执行

大批量 INSERT/UPDATE/DELETE 应拆分为每批 1000-5000 条的小批次执行，避免长事务与锁表。

### 实践七：慢查询定期排查

开启慢查询日志（阈值建议 1 秒），定期分析 Top 10 慢查询并针对性优化索引或重写 SQL。

## 常见问题

### Q1：连接池大小设多少合适？

公式参考：`池大小 = (核心数 * 2) + 有效磁盘数`。但实际需结合 QPS、单查询耗时、可接受延迟综合调优。建议从 10-20 起步，通过压测调整。

### Q2：为什么加列会导致锁表？

在旧版本数据库中，带默认值的加列操作会重写整张表。`PostgreSQL` 11+ 已优化此行为，但仍建议使用 NULL 默认值 + 回填 + 设置默认值三步法。

### Q3：N+1 查询怎么发现？

开启 SQL 日志或使用 APM 工具，若同一请求内出现大量相似查询，基本可判定为 N+1。对策是预加载关联或批量查询。

### Q4：备份能恢复才算备份

定期执行恢复演练，验证备份的完整性与可恢复性。仅备份不验证等于没有备份。

### Q5：复制延迟导致读到旧数据怎么办？

对一致性敏感的读请求，路由到主库；或在副本查询前检查复制延迟，超阈值则回退主库。

### Q6：COUNT 查询很慢怎么办？

大表的 COUNT(*) 需要扫描全表。替代方案：使用近似计数（`PostgreSQL` 的 pg_class.reltuples）、维护计数器表、或使用缓存。

### Q7：如何防止 ORM 生成的 SQL 走偏？

开启 SQL 日志审查 ORM 生成的实际 SQL，定期检查是否有 N+1 查询、缺失索引的 WHERE 条件、不必要的全表加载。

### Q8：服务器less 环境如何管理数据库连接？

Serverless 函数每次调用可能新建连接，导致连接数暴增。推荐使用连接代理（如 PgBouncer）复用连接，或使用云数据库的 Proxy 服务。
