# Professional Communication

写出清晰、有效、会被读到并被行动的职场消息。技术团队的写作问题通常不是文笔问题，而是结构问题——关键信息埋在第三段、行动请求模糊、渠道错配、术语堆砌。本 Skill 用结构化模板与硬规则消除这些病灶.

## 核心能力
**关键信息优先。可扫描格式。明确行动请求。**

每条职场消息回答三个问题：
- 你需要知道什么？
- 为什么重要？
- 需要什么行动（若有）？

这三问决定主题行、首句、结构、结尾。任何不回答这三问的消息都需要重写.

## 四条规则

### 规则一：主题行讲清故事

"Project X: Decision Needed by Friday"胜过"Question"。主题行必须包含主题、目的、时间敏感性三个维度中的至少两个。收件人扫一眼主题行就知道要不要立刻打开、要不要回复、要不要转发.
### 规则二：项目符号替代段落

没人读文字墙。把段落拆成项目符号，每条一个完整想法，不超过两行。段落仅用于叙事性背景（如升级邮件的前情），且不超过两句.
### 规则三：具体行动请求带时间

"Please review by Thursday"胜过"Let me know"。行动请求必须包含：谁、做什么、什么时候。模糊请求等于没有请求，会被淹没在收件箱里.
### 规则四：渠道匹配正式度

聊天（Slack/Teams）用于快速、非正式、即时回复；邮件用于记录、正式、跨时区；会议用于需要同步讨论的复杂决策。渠道错配是职场沟通最常见的浪费——把本该发邮件的内容丢进聊天，三天后就找不到了；把本该三句话聊完的事拉成会议，所有人的下午就毁了.
## 消息结构速查

```text
Subject: [主题]: [具体目的]
# ...
[1-2 句：关键点或请求前置]
# ...
**Context:** (若需要)
- 项目符号，不是段落
# ...
**Action Needed:**
- 具体请求 + 时间线
```

这是默认骨架。不同场景在这个骨架上变形，但关键信息优先与行动请求明确两条不变.
## 永不清单

- 第一句没有明确目的。任何"Hi"、"Hope you're well"、"Just reaching out"开场后不接正题的消息都要重写.
- "Just checking in"无上下文。要么说在跟进什么、上次说到哪、这次需要什么，要么不要 check in.
- 段落代替项目符号。可扫描格式是硬规则，文字墙是违约.
- 把请求埋在底部。请求必须在首句或第二段开头出现，不在结尾.
- 对非技术受众用术语。给财务讲 ROI 不要说"卡尼曼风险厌恶偏差"，给法务讲合规不要说"OAuth2 PKCE flow".
- 聊天里发文字墙。超过 5 行就开 thread 或转邮件.
- 不必要的回复全部。回复全部前问自己：所有人都需要看到这条吗？大多数答案是不.
- 被动语态当主动语态用。"We decided"胜过"It was decided"。被动语态隐藏责任人，是职场甩锅的前奏.
## 适用场景

### 状态更新邮件

每周或每两周向经理或利益相关方发送项目进度。结构：主题行含项目名与"Status Update + 日期"；首句一句话总结整体状态（on track / at risk / blocked）；项目符号列出本周完成、下周计划、风险与阻塞；结尾明确是否需要决策或行动。拒绝流水账——经理不需要知道你开了多少会，只需要知道项目是否能按时交付.
### 升级请求

项目遇到阻塞需要上级介入。结构：主题行含"Escalation"与项目名与阻塞点；首句直接说需要什么决策或资源；项目符号列出已尝试的方案与结果；结尾给出建议的决策与时间窗口。升级邮件不是抱怨邮件——必须包含你已经做了什么、为什么需要上级、建议的解决方案。没有建议的升级等于把猴子甩给上级.
### 技术概念向非技术受众转译

工程师向产品、销售、法务、高管解释技术决策。结构：首句用业务语言说结论（不要先讲技术原理）；项目符号列出对业务的影响（成本、风险、时间线、合规）；技术细节放附录或单独文档链接。转译不是简化——是重新组织信息优先级，让受众能基于业务影响做决策，而不是基于他们不懂的技术细节点头.
### 异步团队协作

跨时区或远程团队无法实时同步。结构：消息开头明确这是异步请求还是 FYI；时间线用所有时区都能理解的格式（UTC 或"by EOD Thursday SF time"）；决策请求给出默认决策与生效时间（"若周四前无异议，按方案 A 推进"）。异步沟通的最大敌人是模糊——任何需要二次确认的表述都会吞掉一个工作日.

### 案例一：状态更新邮件

输入背景：项目代号 Atlas；本周完成 API 迁移与性能测试；下周计划灰度发布；风险是第三方依赖延迟两天；需要经理决定是否推迟灰度.
输出：

```text
Subject: Atlas: Status Update — On Track, Decision Needed by Wed
# ...
Atlas 本周按计划推进，灰度发布前需要一个决策.
# ...
**Completed this week:**
- API v2 迁移完成，旧端点已下线
- 性能测试通过，p99 延迟从 420ms 降到 180ms
# ...
**Planned for next week:**
- 周二开始 5% 灰度
- 周四 review 灰度指标
# ...
**Risks:**
- 第三方风控依赖延迟两天，可能影响灰度期间的异常订单处理
# ...
**Action Needed:**
- 周三前决定：灰度按计划推进，还是推迟到下周等依赖就位
- 我的建议：按计划推进，灰度期间手动兜底异常订单，预计每天 3-5 单
```

### 案例二：升级请求

输入背景：项目代号 Beacon；安全审计发现鉴权服务有漏洞；修复需要两周；但发布窗口在十天后；需要 VP 决定是否推迟发布.
输出：

```text
Subject: Escalation: Beacon — Auth Vulnerability, Publish Window Decision Needed
# ...
需要您决定 Beacon 是否推迟发布以修复鉴权漏洞.
# ...
**Context:**
- 安全审计周三发现鉴权服务存在 token 重放风险
- 修复工作量评估两周，当前发布窗口在 10 天后
- 已确认漏洞未被利用，但合规层面不能带病上线
# ...
**Options:**
- A: 推迟发布两周，完整修复后上线（合规无风险，市场窗口损失约 8%）
- B: 按计划发布，发布后两周内推送热修复（合规风险，需法务签字）
- C: 按计划发布但临时关闭受影响接口，修复后恢复（功能缺失，需产品确认）
# ...
**Action Needed:**
- 周五前选定方案。我的建议是 A，损失可量化，合规无悬念.
- 若选 B，需要法务与安全 VP 联签风险接受书.
```

### 案例三：技术概念转译

输入背景：工程师向销售总监解释为什么要把单体应用拆成微服务；销售总监关心客户体验与销售承诺，不关心架构.
输出：

```text
Subject: 微服务拆分对客户体验与销售承诺的影响
# ...
结论：拆分后客户感知的稳定性会提升，但销售承诺需要调整 SLA 条款.
# ...
**对客户的影响:**
- 单服务故障不再拖垮整个平台，预计年度可用性从 99.5% 提升到 99.9%
- 发布频率从每月一次提升到每周，客户能更快拿到新功能
- 过渡期（约三个月）可能出现偶发接口延迟，需提前知会重点客户
# ...
**对销售承诺的影响:**
- 当前合同 SLA 写的是 99.5%，新架构能支撑 99.9%，可以升级条款作为续约筹码
- 过渡期内不要向重点客户承诺"零中断迁移"，建议话术改为"分批灰度，单功能回滚"
# ...
**不需要您关心的:**
- 技术栈选择、服务拆分边界、部署流程——这些由工程团队决定
- 若客户追问技术细节，请走标准转接流程，由解决方案架构师对接
```

## 异常处理

### 用户只给主题没给背景

用户说"帮我写一封项目延期通知邮件"但不提供项目名、延期原因、影响范围、新时间线。处理：先按通用结构生成草稿，所有需要用户补充的位置用方括号占位标出（如 `[项目名]`、`[新时间线]`），并在草稿末尾列出"发布前请确认"的清单。不要编造项目名或时间线.
### 用户要求用非正式语气写正式场合

用户要求用"轻松一点"的语气写发给 VP 的升级邮件。处理：区分"清晰简洁"与"非正式"。清晰简洁是默认风格，VP 也欢迎；非正式（俚语、缩写、表情符号）在升级邮件里是违约。坚持正式但简洁的语调，向用户说明为什么这里不能太随意.
### 技术细节过多淹没业务影响

工程师用户写状态更新时塞满 API 名、错误码、堆栈跟踪。处理：把技术细节移到"技术附录"段落或单独链接，主体邮件只保留业务影响（用户感知、收入、合规、时间线）。向用户说明：经理读主体，工程师读附录，两全其美.
### 行动请求模糊

用户写"希望大家关注一下这个问题"。处理：追问三个具体化——谁、做什么、什么时候。把模糊请求改写为"张三在本周三前给出迁移方案，李四在周五前 review"。若用户无法提供具体信息，提示这条消息可能不该发——没有具体行动请求的消息通常是 FYI 而非 Action.
### 跨时区时间表达冲突

用户在邮件里写"明天下午三点"。处理：跨时区团队必须用明确时区。改写为"明天 15:00 北京时间（07:00 UTC）"或"明天 EOD SF time"。若用户未指明时区，默认用收件人所在地时区并标注 UTC 等价时间.
### 渠道选择错误

用户要把一份需要留档的决策发到 Slack 群。处理：提示决策类内容应走邮件或文档，Slack 适合讨论不适合留档。若用户坚持发 Slack，建议在 Slack 讨论后用邮件总结决策并发送留档链接.
### 收件人列表过大

用户在回复全部时抄送了整个部门。处理：提示"回复全部"前先问"所有人都需要看到这条吗"。多数情况下只需要保留直接相关方。给一个判定标准：若某人在未来一周不会基于这条消息采取行动或决策，就移出收件人列表.
### 术语堆砌对非技术受众

用户给财务团队写邮件时堆砌"OAuth2 PKCE flow"、"JWT claims"、"rate limiting"。处理：把每个术语翻译成业务等价物——"OAuth2 PKCE flow"改为"安全登录流程"、"JWT claims"改为"用户身份信息"、"rate limiting"改为"防刷限流"。若必须保留术语，首次出现时附中文解释.
## 常见问题

### Q1: 主题行多长合适？

主题行长度以收件人预览面板能完整显示为准，桌面端约 50-60 字符，移动端约 30-40 字符。关键信息（项目名、目的、时间敏感性）放在前 30 字符内。不要写成句子，写成"主题: 目的 + 时间"的紧凑格式.
### Q2: 什么时候用邮件，什么时候用聊天？

需要留档、跨时区、正式决策、长背景——邮件。需要即时反馈、快速确认、非正式讨论——聊天。简单判定：三天后还需要找到这条消息吗？需要则邮件，不需要则聊天.
### Q3: 升级邮件会不会显得我没能力？

不会，前提是升级邮件包含你已经尝试的方案、为什么需要上级、建议的解决方案。这种升级是职责清晰的表现。没有建议、没有尝试记录的升级才会显得甩锅。上级讨厌的是后者，不是前者.
### Q4: 技术概念转译要保留多少技术细节？

保留业务影响，砍掉实现细节。判定标准：受众基于这条信息能做什么决策？能做的决策需要的信息就保留，不能的砍掉。若受众是技术决策者（如 CTO），保留架构选型与权衡；若受众是业务决策者（如 CFO），保留成本与风险.
### Q5: 异步消息怎么避免"等了两天才回一句"？

在消息里给出默认决策与生效时间。例如"若周四前无异议，按方案 A 推进"。这样沉默等于同意，不会因为某人没回而卡住整个流程。默认决策必须是低风险的、可逆的、与团队既定方向一致的.
### Q6: 项目符号每条多长合适？

每条不超过两行，一个完整想法。若一个想法需要超过两行，要么拆成两条，要么说明信息密度太高需要重新组织。项目符号不是段落的换行版，是独立的信息单元.

## 已知限制

- 基于结构化模板与硬规则，不替代对具体业务语境的判断。涉及政治敏感、文化差异、个人冲突的沟通需要人工审阅.
- 主题行与行动请求的有效性依赖用户对收件人优先级的准确判断。Skill 无法知道某位 VP 是否真的关心某个项目.
- 跨文化沟通的正式度标准差异（如美式直接与日式委婉）不在本 Skill 覆盖范围，需要用户自行调整.
- 不生成法律、合规、HR 类正式函件。这类文档需要法务或 HR 团队审核，模板化生成有合规风险.
- 不替代面对面沟通。涉及情绪、冲突、绩效问题的对话应优先当面或视频，不要用邮件解决.
- 模板反映主流技术团队的沟通习惯，传统行业或非技术团队可能需要调整风格.
