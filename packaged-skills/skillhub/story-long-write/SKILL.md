# 长篇网文写作

从零开始写一本长篇网络小说,从选题确认到大纲搭建再到正文输出。先抓情绪,再用验证过的方法可靠地交付这个情绪,灵感只做素材来源.

## 核心能力

### 1. 核心方法
- 先定情绪,再定故事:每个场景必须服务于明确的情绪目标,说不清交付什么情绪的场景不该存在
- 从验证过的模式出发:扫榜找方向,拆文找模块,对标找节奏
- 用模块组装:把对标书的具体角色看成功能位(对手/盟友/催化剂),映射到自己的角色,用自己素材填充
- 只加载必需信息:写每章时只加载"不知道就会写错"的信息
- 契约与推进决策走权威参考:涉及读者契约、主角代理权、期待债、终局储备时,按 `references/reader-contract-and-progression.md` 校准

**输入**: 用户提供核心方法所需的指令和必要参数.
**处理**: 解析核心方法的输入参数,执行核心处理逻辑,返回结构化结果和执行状态.
**输出**: 返回核心方法的处理结果,包含执行状态码、结果数据和执行日志.

### 3. 场景路由
| 场景 | 触发条件 | 执行流程 |
|:------|------:|:------|
| 开书 | "帮我开书"/项目目录为空 | Phase 1→2→3,默认停在细纲交付,不自动写正文 |
| 写指定章 | "写第N章" | Phase 4 单章写作,写完Phase 5检查后停止 |
| 补纲/扩纲 | "出细纲/补细纲/规划剧情"且已有大纲 | Phase 3 中途补纲小流程,默认停在细纲交付 |
| 日更续写 | "日更/续写/继续写"且已有正文+追踪 | 加载 `references/workflow-daily.md` |
| 大修 | "修改第X章/回炉/重写" | 加载 `references/workflow-revision.md` |

匹配优先级:大修 → 写指定章 → 补纲/扩纲 → 日更续写 → 开书.
**输出**: 返回场景路由的处理结果,包含执行状态码、结果数据和执行日志.
### 4. 裸调用与停靠点(防失控)
裸调用(无明确意图)时只做项目状态诊断并列出选项,不自动进入正文写作:
- 空项目 → 建议"帮我开书"
- 已有设定/大纲但无正文 → 建议"写第1章"
- 已有正文+追踪 → 展示最后章节与下一章细纲状态

正文批量上限:未给数量时单章默认1章;日更默认2-3章;用户给N时单轮最多3章,超过拆成本轮3章.
- 参考`裸调用与停靠点(防失控)`的配置文档进行参数调优

**输出**: 返回裸调用与停靠点(防失控)的处理结果,包含执行状态码、结果数据和执行日志.
### 5. Phase 1:确认选题方向
- 先查 `选题决策.md`:取可行性最高的推荐选题作为起点
- 用户无方向时问:"想让读者什么感觉?有喜欢的书对标?优势是什么?"
- 脑洞好→系统文/诸天流/无限流;文笔好→仙侠/历史/文艺都市;节奏感好→都市爽文/重生文;生活经验丰富→行业文/都市日常/种田文

**处理**: 解析Phase 1:确认选题方向的输入参数,执行核心处理逻辑,返回结构化结果和执行状态.
**输出**: 返回Phase 1:确认选题方向的处理结果,包含执行状态码、结果数据和执行日志.
### 6. 对标上下文加载
- 检查 `对标/` 目录的 `剧情/情绪模块.md` 与 `剧情/节奏.md`
- 任一主产物缺失必须停止,设置 `missing_primary_contract: true`,要求重跑分析流程
- 两个主产物都存在时:先读情绪模块的读者需求/情绪引擎/可复现模块,再读节奏的关键信息推进/情绪触动点/爆发节奏

**输入**: 用户提供对标上下文加载所需的指令和必要参数.
**输出**: 返回对标上下文加载的处理结果,包含执行状态码、结果数据和执行日志.
### 7. Phase 2:核心设定
从目标情绪出发,在题材框架中找到剧情模式,从对标书提取可复用模块,用用户角色填充。产出:
- `设定/关系.md`:角色关系映射(四种关系类型)
- `设定/题材定位.md`:题材核心梗三分法+对标分析,终局底牌与升级台阶
- `设定/题材正文提示卡.md`:题材边界、核心逻辑、读者期待、核心爽点、正文落点、前中后期打法、节奏密度、场景颗粒、禁止漂移

**输出**: 返回Phase 2:核心设定的处理结果,包含执行状态码、结果数据和执行日志.
### 8. Phase 3:大纲搭建
- 全书体量与阶段边界:作为卷纲、细纲、日更补纲的共同约束
- 卷级大纲:功能、所属阶段、卷契约、终局储备、剧情单元、阶段边界、核心事件、起始→结束状态
- 细纲:每章一个文件 `大纲/细纲_第XXX章.md`,不允许跳章

**输入**: 用户提供Phase 3:大纲搭建所需的指令和必要参数.
**处理**: 解析Phase 3:大纲搭建的输入参数,执行核心处理逻辑,返回结构化结果和执行状态.
**输出**: 返回Phase 3:大纲搭建的处理结果,包含执行状态码、结果数据和执行日志.
### 9. 大纲安全七检
每卷/每批细纲设计前答一遍:
1. 本卷交付什么情绪?什么剧情模式能可靠交付?
2. 本卷核心冲突是什么?
3. 卷节奏哪段加速哪段减速?
4. 本卷需新埋设的伏笔?上一卷待回收伏笔如何处理?
5. 章节定位分布是否有高低层次?低压+过场合计不超约15%?
6. 本卷/本章处在全书哪个阶段,可释放什么,必须压住什么?
7. 读者契约、主角代理权、期待债、终局储备是否评为 契约安全/需补强/契约破坏?

**输入**: 用户提供大纲安全七检所需的指令和必要参数.
**输出**: 返回大纲安全七检的处理结果,包含执行状态码、结果数据和执行日志.
### 10. 大纲安全审查
顺序不可倒置:
1. 先按 `references/emotional-methods.md` 设计正向情绪发动机与题材核心兑现
2. 用权威文件的主角代理权/所有权/成长负向风险护栏诊断
3. 检查每个剧情单元的主角目标、关键选择、兑现归属、核心资产交换、机构/势力边界
4. 契约破坏必须修纲,需补强必须补交换/铺垫/成本
5. 两个可证伪降级检查:删除题材核心后是否退化为通用升级?删除主角后情绪兑现是否不变?

**输入**: 用户提供大纲安全审查所需的指令和必要参数.
**输出**: 返回大纲安全审查的处理结果,包含执行状态码、结果数据和执行日志.
### 11. 批次补纲规则
- 批次边界:一批≈一个剧情单元(5-15章);>10章拆两批;首批前10章封顶
- 批次定位:当前章节区间属于哪个阶段与剧情单元、本批推进目标、可释放信息、严禁提前释放信息
- 剧情单元消费:建卡批读本批剧情单元卡(≤3个文件),按节拍→章映射铺章
- 批末预算核对:各章预算合计 Σ∈[章目标, ×1.1],不合格补密点/压疏点

**处理**: 解析批次补纲规则的输入参数,执行核心处理逻辑,返回结构化结果和执行状态.
**输出**: 返回批次补纲规则的处理结果,包含执行状态码、结果数据和执行日志.
### 12. 大纲锁定
- 已进入正文写作的前10章细纲锁定,未经用户确认不得修改
- 卷纲锁定:某卷一旦已有正文章节,该卷已写区间对应的剧情单元卡与对标结构坐标视为锁定
- 允许在卷尾追加新剧情单元卡(只增不改),追加只在用户明确要求补纲/扩纲时

**处理**: 解析大纲锁定的输入参数,执行核心处理逻辑,返回结构化结果和执行状态.
**输出**: 返回大纲锁定的处理结果,包含执行状态码、结果数据和执行日志.
### 13. 中途补纲/扩纲小流程
1. 定位当前卷、已写进度与既有剧情单元卡
2. 选同类剧情单元,设计新剧情单元卡,追加到当前卷卷纲卷尾
3. 批次级答大纲安全七检并执行大纲安全审查
4. 按剧情批滚动补细纲(一批≈一个剧情单元)
5. 更新 `追踪/上下文.md`

**输出**: 返回中途补纲/扩纲小流程的处理结果,包含执行状态码、结果数据和执行日志.
### 14. Phase 4:正文写作
- 只加载"不知道就会写错"的信息:角色状态、待回收伏笔、相关设定
- 按细纲直接指导正文:阶段位置、结构公式、禁止提前释放、内容概括、情节安排、人物关系、情节细化、结尾设定
- 强度按章节定位分配:高压章配齐钩子+爽点+悬念;低压章允许无显性爽点,重点写功能(喘息、关系、铺垫、转场)
- 底线:每章给读者一个往下看的理由,相邻章不情绪趋同

**输入**: 用户提供Phase 4:正文写作所需的指令和必要参数.
**处理**: 解析Phase 4:正文写作的输入参数,执行核心处理逻辑,返回结构化结果和执行状态.
**输出**: 返回Phase 4:正文写作的处理结果,包含执行状态码、结果数据和执行日志.
### 15. Phase 5:质量检查
三维度检查:
- 情绪交付:每章是否交付细纲中规划的目标情绪?
- 契约风险:按 `references/reader-contract-and-progression.md` 检查因果权+结算权、关键节点四问、期待所有权、期待债、终局储备;契约破坏先修正文或修后续纲
- 技术质量:一致性、格式、禁用词

**输入**: 用户提供Phase 5:质量检查所需的指令和必要参数.
**输出**: 返回Phase 5:质量检查的处理结果,包含执行状态码、结果数据和执行日志.
### 16. 正文元信息扫描
质量检查必须覆盖标题行以外的正文,发现 `第X章|上一章|前一章|本章|前文|后文|伏笔|细纲|读者` 等写作工程词时,改成角色当下可感知的事件、物件、动作或相对时间。故事内真实阅读/讨论"第X章"或真实读者身份语境除外。- 验证返回数据的完整性和格式正确性
- 参考`正文元信息扫描`的配置文档进行参数调优
### 17. 写后同轮清零
每章落盘后必须在同一轮内跑完Phase 4步骤10-11扫描、确定性收尾脚本与叙事审查代理审查,blocking清零才算本章完成。唯一豁免:用户显式说"本章不去味/跳过检查",在标题行下加 `<!-- 去味:跳过 -->`。- 验证返回数据的完整性和格式正确性
- 参考`写后同轮清零`的配置文档进行参数调优
### 18. 确定性收尾
- `node （请参考skill目录中的脚本文件） --check --fail-on=blocking 正文/第XXX章_*.md`:blocking命中先回正文改写并复扫,advisory作读感提示
- `node （请参考skill目录中的脚本文件） 正文/第XXX章_*.md`:清理无功能省略号、破折号、双连字符和独立分隔线
- `node （请参考skill目录中的脚本文件） --check 正文/第XXX章_*.md`:blocking(复读、截断、拒绝语、tier1工程词泄漏)重写受影响章节，最多2次

**输入**: 用户提供确定性收尾所需的指令和必要参数.
**处理**: 解析确定性收尾的输入参数,执行核心处理逻辑,返回结构化结果和执行状态.
**输出**: 返回确定性收尾的处理结果,包含执行状态码、结果数据和执行日志.
### 19. 代理调用
- 结构设计代理:辅助题材分析、世界观构建、核心冲突设计、卷级结构

**输入**: 用户提供代理调用所需的指令和必要参数.
**输出**: 返回代理调用的处理结果,包含执行状态码、结果数据和执行日志.

## 适用场景

| 场景 | 输入 | 输出 |
|---:|:---|---:|
| 开书 | 选题方向或对标书 | 项目结构+核心设定+卷纲+前10章细纲 |
| 写指定章 | 章节号 | 该章正文+质量检查报告 |

不适用于:短篇创作、纯技术文档、非虚构写作、学术写作.
## 使用流程

1. 识别场景:开书/写指定章/补纲/日更/大修,按匹配优先级
2. 裸调用时只做项目状态诊断,不自动进入正文
3. 开书走Phase 1→2→3,默认停在细纲交付
4. 写正文走Phase 4,写完Phase 5同轮清零
5. 补纲走中途补纲小流程,默认停在细纲交付
6. 日更保持在 `references/workflow-daily.md` 内,不跳出

**结果验证**: 任务完成后,查看输出确认状态。成功时返回摘要和数据;失败时根据错误信息排查,参考恢复章节获取修复步骤.

## 错误处理

| 错误场景 | 原因 | 处理方式 |
|:------:|--------|:-------|
| 对标主产物缺失 | `剧情/情绪模块.md`或`剧情/节奏.md`不存在 | 停止,设置 `missing_primary_contract: true`,要求重跑分析流程或重新导入 |
| 细纲缺失无法写正文 | 未建对应章细纲 | 先补Phase 1→3再写点名章,不跳章 |
| 章节号冲突 | 细纲与正文文件名编号不一致 | 按细纲编号修正正文文件名,保持一致 |
| 大纲安全审查契约破坏 | 剧情违反读者契约或主角代理权 | 必须修纲,补交换/铺垫/成本后再继续 |
| 正文元信息扫描发现工程词 | 正文含"第X章/前文/伏笔"等词 | 改成角色当下可感知的事件、物件、动作或相对时间 |
| 退化防护检测到复读/截断 | blocking级退化(复读、截断、拒绝语) | 重写受影响章节，最多2次;仍失败报告证据让用户定夺 |
| 预算合计超出[章目标,×1.1] | 章节字数过少或过多 | 补密点(增加情节)或压疏点(精简过渡)至预算范围 |

## 常见问题

### Q1: 首批细纲为何封顶10章?
A: 避免单次对话强行产出30章完整细纲导致质量下降。首批前10章封顶,可跨剧情单元。写到单元末段(剩余未建细纲≤2章)即滚动补下一单元整批。用户显式给章数时按要求执行,单批仍建议≤10章.
### Q2: 低压+过场合计不超约15%如何把握?
A: 章节定位分高压/推进/修炼试错/关系回收/低压生活/信息整理。低压章允许无显性爽点,重点写功能(喘息、关系、铺垫、转场)。但低压+过场合计不超约15%,否则读者失去往下看的理由。底线是每章给读者一个往下看的理由,相邻章不情绪趋同.
### Q3: 批末预算核对如何执行?
A: 每批建完必做。统一核对各章「预算合计」行 Σ∈[章目标, ×1.1]。不合格章补密点(增加情节)或压疏点(精简过渡)后再交付。不要求逐章写作时反复心算,批次级核对即可.
### Q4: 大纲锁定的具体定义是什么?
A: 已进入正文写作的前10章细纲锁定,未经用户确认不得修改。卷纲锁定:某卷一旦已有正文章节,该卷已写区间对应的剧情单元卡与对标结构坐标视为锁定。允许在卷尾追加新剧情单元卡(只增不改),追加只在用户明确要求补纲/扩纲时.
### Q5: 写后同轮清零的豁免如何标记?
A: 用户显式说"本章不去味/跳过检查"时,在该章标题行下加一行 `<!-- 去味:跳过 -->`。写后hook的毒句式推回与写下一章前的欠账拦截都认这个标记。其余网(一致性检查、退化防护)照常运行.
### Q6: `check-ai-patterns.js` 的blocking与advisory有何区别?
A: blocking命中必须当轮回正文改写并复扫,不清零不算本章完成;advisory只作读感提示,确属问题才改,功能性写法标 `[需复核]`。退化防护的blocking(复读、截断、拒绝语、tier1工程词泄漏)重写受影响章节最多2次,仍失败报告证据让用户定夺.
