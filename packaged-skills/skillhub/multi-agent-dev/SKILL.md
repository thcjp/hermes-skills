# 多代理开发框架（Multi-Agent Dev）

通过为每个任务派发新鲜子代理执行实现计划，并在每步后进行两阶段评审（先规格合规，后代码质量），实现高质量快速迭代。

**核心原则：** 每任务新鲜子代理 + 两阶段评审（规格→质量）= 高质量、快迭代

## 核心能力

1. **智能任务分解与依赖图构建**
   - 读取实现计划文件，一次性提取所有任务的完整文本与上下文
   - 构建任务依赖图，识别完全独立任务、共享文件任务、有逻辑依赖任务
   - 创建TodoWrite跟踪所有任务状态，确保进度可见
   - 控制器精确策展每个子代理所需的上下文，避免子代理读取计划文件
   - 输出：任务依赖图、TodoWrite任务列表、每个任务的完整上下文包

2. **选择性并行执行策略**
   - 完全独立任务（无共享文件）：可并行派发实现子代理，加速开发
   - 共享文件但无逻辑依赖：串行实现，可并行评审，避免文件冲突
   - 有逻辑依赖：严格串行，等依赖任务完成后再开始
   - 不确定时默认串行，安全优先
   - 并行安全规则：永远不并行派发操作同一文件的实现子代理；并行实现完成后评审必须串行；出现冲突立即回退串行

3. **分层评审机制（L0-L3）**
   - L0自评审：每次实现后，实现子代理自检，成本最低
   - L1规格合规：L0通过后，对照规格检查完整性与多余内容
   - L2代码质量：L1通过后，检查代码质量、最佳实践、魔术数字等问题
   - L3全局评审：所有任务完成后，检查整体一致性与集成问题
   - 严格评审顺序：L0→L1→L2→（所有任务完成）→L3，绝不在L1通过前开始L2
   - 评审循环：评审发现问题→实现者修复→评审者重新评审，重复直到通过

4. **上下文隔离与新鲜子代理**
   - 每个任务派发全新子代理，避免上下文污染
   - 控制器提供完整任务文本+场景设定+相关文件内容
   - 子代理预先获得完整信息，问题在工作开始前浮现而非之后
   - 子代理可提问，控制器清晰完整地回答并提供额外上下文
   - 子代理遵循TDD流程：实现、测试、提交、自评审

5. **应急处理与红旗防护**
   - 子代理提问时：清晰完整回答，必要时提供额外上下文，不催促进入实现
   - 评审发现问题：同一实现子代理修复，评审者重新评审，不跳过重新评审
   - 子代理任务失败：派发修复子代理并附具体指令，不手动修复（避免上下文污染）
   - 子代理上下文不足：控制器补充缺失上下文，重新派发
   - 并行任务冲突：立即停止冲突任务，回退串行执行
   - 红旗清单：未经同意不在main/master分支开始、不跳过评审、不带未修复问题继续、不让子代理读取计划文件

6. **集成与工作流协同**
   - 必需集成：git-worktrees（隔离工作空间）、writing-plans（创建计划）、finishing-a-development-branch（完成开发）
   - 子代理使用：test-driven-development（每任务遵循TDD）
   - 替代工作流：executing-plans（用于并行会话而非同会话执行）
   - 评审模板：implementer-prompt.md、spec-reviewer-prompt.md、code-quality-reviewer-prompt.md

## 使用流程

### 第一步：确认前置条件并读取计划

确认满足以下条件后开始：
- 有明确的实现计划文件（如 `docs/plans/feature-plan.md`）
- 任务相对独立（紧耦合任务应先头脑风暴或手动执行）
- 在当前会话中执行（如需跨会话则使用executing-plans替代）

读取计划文件一次，提取所有任务的完整文本和上下文，创建TodoWrite包含所有任务。

### 第二步：构建依赖图并选择执行策略

根据任务间的文件操作和逻辑依赖关系，构建依赖图：
- 识别完全独立任务（无共享文件）→ 可并行
- 识别共享文件但无逻辑依赖 → 串行实现，可并行评审
- 识别有逻辑依赖 → 严格串行
- 不确定时默认串行

### 第三步：逐任务派发子代理执行

对每个任务：
1. 获取任务文本和上下文（已提取）
2. 派发实现子代理，提供完整任务文本+场景设定+相关文件内容
3. 如子代理有疑问，回答问题并提供上下文后重新派发
4. 子代理实现、测试、提交、自评审（L0）
5. 派发规格合规评审子代理（L1），不合规则实现者修复后重新评审
6. 派发代码质量评审子代理（L2），不通过则实现者修复后重新评审
7. 在TodoWrite中标记任务完成

### 第四步：执行全局评审

所有任务完成后，派发最终全局代码评审子代理（L3），检查整体一致性、集成问题、接口对齐。

### 第五步：完成开发分支

使用"完成开发分支"流程，决定合并、PR或保留分支。

## 错误处理

| 错误类型 | 原因 | 处理方式 |
|---:|---:|---:|
| 子代理产出与规格不符 | 上下文不足或规格模糊 | 补充完整任务文本+场景设定+相关文件内容，重新派发子代理 |
| 评审循环超过3次 | 实现质量低或规格不清晰 | 检查规格明确性，考虑重新分解任务为更小单元 |
| 子代理上下文污染 | 复用了非新鲜子代理 | 确保每任务派发全新子代理，不复用历史子代理实例 |
| 并行任务文件冲突 | 并行子代理操作了相同文件 | 立即停止冲突任务，回退到串行执行模式，记录冲突文件 |
| TodoWrite状态不同步 | 未及时更新任务完成状态 | 每完成一个任务立即更新TodoWrite，定期检查状态一致性 |
| 最终评审发现集成问题 | 任务间接口未对齐 | 在计划阶段明确接口契约，补充接口文档后重新评审 |
| 子代理反复提问影响效率 | 计划不够详细或上下文不完整 | 完善计划细节，控制器在派发时提供更完整的上下文 |
| 子代理任务执行失败 | 实现复杂度超出子代理能力 | 派发修复子代理并附具体指令，或将任务拆分为更小子任务 |

## FAQ

**Q1：任务之间有依赖怎么办？**
严格串行执行依赖任务。使用依赖图识别哪些任务可并行（完全独立）、哪些必须串行（有逻辑依赖）。不确定时默认串行。依赖任务需等待前序任务完成后再开始，前序任务的输出作为后续任务的上下文。

**Q2：评审循环太多导致成本过高怎么办？**
使用分层评审。L0自评审可过滤明显问题，减少L1/L2评审循环。确保实现子代理在提交前充分自检。规格清晰的计划能减少规格合规循环。如果评审循环超过3次，应检查规格明确性，考虑重新分解任务为更小单元。

**Q3：子代理反复提问影响效率怎么办？**
确保控制器在派发时提供完整上下文（任务全文+场景设定+相关文件内容）。如子代理仍反复提问，可能是计划不够详细，应先完善计划。回答子代理问题时清晰完整，必要时提供额外上下文，不要催促子代理进入实现。

**Q4：如何判断任务是否"相对独立"？**
检查三个方面：(1) 是否操作相同文件；(2) 是否有数据依赖（需要前序任务的输出）；(3) 是否有逻辑依赖（前序任务的结果影响后续任务的实现方式）。如都不涉及，则独立。不确定时按串行处理。

**Q5：并行执行真的安全吗？**
仅当任务操作完全不同的文件且无逻辑依赖时安全。并行安全规则：永远不并行派发操作同一文件的实现子代理；并行实现完成后评审仍需串行；出现任何冲突立即回退串行。并行任务出现冲突时，立即停止冲突任务并记录冲突文件。

**Q6：为什么不让子代理读取计划文件？**
控制器提供完整任务文本而非让子代理读取计划文件，有两个好处：(1) 无文件读取开销，子代理预先获得完整信息；(2) 控制器精确策展所需上下文，避免子代理被无关任务信息干扰。这样问题在工作开始前浮现而非之后。

**Q7：与executing-plans有什么区别？**
本技能在同一会话内执行（无上下文切换），每任务新鲜子代理，每任务后两阶段评审，更快迭代。executing-plans用于并行会话执行（跨会话），适合需要完全隔离的场景。如有实现计划且任务相对独立且留在当前会话，使用本技能。

## 已知限制

1. **子代理调用成本较高**：每个任务需要实现子代理+2个评审子代理（规格合规+代码质量），控制器还需更多准备工作（预先提取所有任务），评审循环增加迭代次数。但早期捕获问题比后期调试更便宜。

2. **依赖Agent平台子代理能力**：需要Agent平台支持子代理派发功能，不支持子代理的平台无法使用本技能。子代理能力质量直接影响执行效果。

3. **紧耦合任务不适用**：如果任务之间高度耦合（共享大量文件或有复杂逻辑依赖），并行执行优势无法发挥，应考虑手动执行或先头脑风暴重新分解任务。

4. **需要明确的实现计划**：本技能执行已有计划，不负责计划创建。如果没有实现计划，需先使用writing-plans等技能创建计划。模糊需求不适用。

5. **评审顺序不可跳过**：必须严格遵守L0→L1→L2→L3的评审顺序，绝不能在L1规格合规通过前开始L2代码质量评审。跳过评审或错误顺序会导致质量问题遗漏。
