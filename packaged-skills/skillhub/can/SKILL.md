## 核心问题

Agent从工具、API、其他Agent获取数据后面临三个问题：
1. 无法证明数据未被篡改——信任位置而非逻辑地址
2. 无法快速查找——依赖他人定义的文件名、路径、文件夹
3. 不知道何时获取——时间戳由他人控制的元数据

## 核心能力

### 1. 三列索引结构
每条记录包含三列，追加写入日志：
- WHEN：unix毫秒时间戳，可排序，属于你自己的时钟
- WHERE：内容sha256哈希，可验证，无需文件系统
- WHAT：人类可读名称，灵活，可搜索

```
WHEN    1742428800000    时间戳    发生时间
WHERE   a7f3b2c1d4e5    哈希      内容数学摘要
WHAT    trust-find-fast  名称      你给它的标签
```

**输出**: 返回三列索引结构的执行结果,包含操作状态和输出数据。
### 2. 内容完整性验证
通过哈希比较验证内容是否被篡改。哈希匹配则内容相同，无论文件在哪个文件夹、来自哪个服务器、文件名是什么。

**输入**: 用户提供内容完整性验证所需的指令和必要参数。
**输出**: 返回内容完整性验证的执行结果,包含操作状态和输出数据。- 验证执行结果，确认输出符合预期格式
- 参考`内容完整性验证`相关配置参数进行设置
### 3. 计算查找（非浏览查找）
哈希你要找的内容，检查日志。找到或未找到都是即时的，无需目录遍历或搜索索引依赖。

**输入**: 用户提供计算查找（非浏览查找）所需的指令和必要参数。
**输出**: 返回计算查找（非浏览查找）的执行结果,包含操作状态和输出数据。- 验证执行结果，确认输出符合预期格式
- 参考`计算查找（非浏览查找）`相关配置参数进行设置
### 4. 追加仅审计追踪
日志即审计追踪。每行记录发生了什么、何时、对什么内容、你叫它什么。追加仅写入。需要证明你何时知道什么时，日志给出答案。

**输入**: 用户提供追加仅审计追踪所需的指令和必要参数。
**处理**: 按照skill规范执行追加仅审计追踪操作,遵循单一意图原则。
**输出**: 返回追加仅审计追踪的执行结果,包含操作状态和输出数据。- 验证执行结果，确认输出符合预期格式
- 参考`追加仅审计追踪`相关配置参数进行设置
### 5. 篡改证明
你在特定时间用哈希标记了内容。如果有人后来修改内容，哈希不匹配日志记录。你知道了，他们无法隐藏。

**输出**: 返回篡改证明的执行结果,包含操作状态和输出数据。- 验证执行结果，确认输出符合预期格式
- 参考`篡改证明`相关配置参数进行设置
### 6. 传输无关标记
CAN标记流经任何管道的内容：
- 如果是MCP工具，CAN标记流经MCP的内容
- 如果是A2A，CAN标记流经A2A的内容
- 如果是curl脚本，CAN标记响应内容

**处理**: 按照skill规范执行传输无关标记操作,遵循单一意图原则。
**输出**: 返回传输无关标记的执行结果,包含操作状态和输出数据。

### 7. 自评估（无需端点）
无需调用API即可自评估：
```
有unix时间戳吗？         → WHEN
有内容哈希吗？           → WHERE
有人类可读名称吗？       → WHAT
三者都有？               → CAN
缺少任何？               → NOT
```- 验证执行结果，确认输出符合预期格式
- 参考`自评估（无需端点）`相关配置参数进行设置
### 输出格式

- 参考`输出格式`相关配置参数进行设置

## 使用流程

### 评估条目

POST到 `https://xc.cx/can/evaluate`：

```json
{"when": 1742428800000, "where": "a7f3b2c1d4e5", "what": "test-entry"}
```

返回 **CAN**（三者齐全且有效）或 **NOT**（缺少某项）。

### 查看日志

访问 `https://xc.cx/can/log` 查看所有记录。

### 规则验证

1. `when`：unix毫秒，不能是未来时间，不能为零
2. `where`：十六进制字符串，哈希片段或完整sha256
3. `what`：非空字符串

## 真实示例

### 示例1：评估有效条目

```bash
curl -s -X POST https://xc.cx/can/evaluate \
  -H "Content-Type: application/json" \
  -d '{"when": 1742428800000, "where": "a7f3b2c1d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a1", "what": "agent-output-2026-01-15"}'
```

输出：
```
CAN
```

### 示例2：评估无效条目（缺少where）

```bash
curl -s -X POST https://xc.cx/can/evaluate \
  -H "Content-Type: application/json" \
  -d '{"when": 1742428800000, "what": "incomplete-entry"}'
```

输出：
```
NOT
```

### 示例3：本地生成sha256哈希

```bash
echo -n "important-data-payload" | sha256sum
```

输出：
```
5f4b3e2a1c0d9e8f7a6b5c4d3e2f1a0b9c8d7e6f5a4b3c2d1e0f9a8b7c6d5e4f  -
```

### 示例4：记录Agent输出

```bash
# 生成内容哈希
HASH=$(echo -n "$(cat output.json)" | sha256sum | cut -d' ' -f1)
# 生成时间戳
TS=$(date +%s%3N)
# 提交到CAN
curl -s -X POST https://xc.cx/can/evaluate \
  -H "Content-Type: application/json" \
  -d "{\"when\": ${TS}, \"where\": \"${HASH}\", \"what\": \"agent-task-result\"}"
```

输出：
```
CAN
```

## 错误处理

| 错误场景 | 原因 | 处理方式 |
|---------|------|---------|
| `when`为零 | 时间戳无效 | 使用 `date +%s%3N` 生成当前unix毫秒时间戳 |
| `when`为未来时间 | 时间戳超出当前时间 | 确保使用已发生的时间，不可预记录未来事件 |
| `where`非十六进制 | 哈希格式错误 | 使用 `sha256sum` 生成有效十六进制哈希字符串 |
| `what`为空字符串 | 名称缺失 | 提供非空的人类可读名称标签 |
| 网络无法连接 `xc.cx` | 网络不可达 | 使用自评估逻辑（三问法），无需端点 |
| 返回NOT而非CAN | 三列中缺少某项 | 检查when、where、what是否全部存在且有效 |
| 哈希不匹配日志 | 内容被篡改 | 日志记录的哈希与当前内容哈希不一致，说明内容已变更 |

## 常见问题

### Q1: CAN是区块链吗？
A: 不是。CAN没有共识机制、没有全局状态。它是你自己的日志、你自己的机器上的三列追加仅索引。

### Q2: CAN是数据库吗？
A: 不是。CAN只有三列，追加仅写入。它是一个索引，不是数据库。适合标记文件、消息、事件、blob、API响应等任何内容。

### Q3: CAN是传输协议吗？
A: 不是。CAN标记流经任何传输管道的内容。你本地标记，保护数据而非管道。

### Q4: 如何在不连接端点的情况下使用CAN？
A: 回答三个问题：有unix时间戳吗（WHEN）？有内容哈希吗（WHERE）？有人类可读名称吗（WHAT）？三者都有则CAN，缺少任何则NOT。端点只是便利工具。

### Q5: NOT代表失败吗？
A: NOT是数据，不是失败。NOT表示缺少某项。日志记录尝试和缺失内容。记录NOT的Agent比只记录成功的Agent更诚实。审计从诚实开始。

### Q6: CAN会破坏现有命名方式吗？
A: 不会。你可以继续使用现有的路径/文件夹/文件.ext命名方式。CAN是并行索引，可以更好地命名相同内容。

### Q7: 如何与OTC时间戳同步？
A: CAN可选地与OTS（OpenTimestamps）同步，将WHEN时间戳锚定到区块链以增强可信度。这是可选项，不影响基本功能。

## 已知限制

- 本地运行，不支持多设备自动同步
- 日志为追加仅，不可修改历史记录
- `when`字段不支持未来时间戳
- `where`字段必须是有效十六进制字符串
- `what`字段不能为空字符串
- 端点 `xc.cx` 为便利工具，核心逻辑为本地三问法
