# 数据库连接助手(专业版)

## 核心能力

### 模块一：分库分表策略设计（专业版独有）

| 决策维度 | 评估要点 | 输出产物 |
|:---------|:---------|:---------|
| 分片键选择 | 访问模式、数据分布、跨片查询频率 | 分片键推荐 + 反例说明 |
| 分片算法 | 范围分片 vs 哈希分片 vs 一致性哈希 | 算法对比 + 迁移成本评估 |
| 分片数量 | 当前数据量增速、单片容量上限 | 分片数计算公式 + 扩容预案 |
| 跨片查询 | JOIN、聚合、分页的处理方式 | 查询路由方案 + 结果合并策略 |

**分片键决策树**：

```
开始
 ├─ 是否有高频等值查询字段？ → 是 → 该字段是否分布均匀？
 │                                        ├─ 是 → 选为分片键
 │                                        └─ 否 → 考虑组合分片键
 └─ 否 → 选择主键或时间字段范围分片
```

### 模块二：复制高可用与故障转移（专业版独有）

提供主从复制、半同步复制、多主架构的对比选型，以及自动故障转移的完整流程设计。

| 复制模式 | 数据一致性 | 故障切换速度 | 适用场景 |
|:---------|:-----------|:-------------|:---------|
| 异步复制 | 最终一致 | 秒级 | 读多写少、可容忍少量丢失 |
| 半同步复制 | 强一致（至少一个副本确认） | 秒级 | 金融交易、订单系统 |
| 多主复制 | 最终一致 | 无需切换 | 多地域写入 |

**故障转移检查清单**：
1. 主库心跳检测连续失败 N 次触发转移
2. 确认副本复制延迟在可接受范围
3. 提升副本为新主库
4. 更新连接配置或依赖服务发现自动重连
5. 旧主恢复后以副本身份重新加入

### 模块三：深度性能调优（专业版独有）

提供 EXPLAIN 执行计划解读方法论、慢查询根因分析框架、索引优化决策矩阵。

**执行计划解读四步法**：
1. **看扫描方式**：Seq Scan（全表）vs Index Scan（索引）vs Bitmap Scan（位图）
2. **看预估行数**：rows 估值与实际行数偏差大说明统计信息过期
3. **看连接策略**：Nested Loop（小表驱动）vs Hash Join（大表等值）vs Merge Join（有序数据）
4. **看额外信息**：Filter、Sort、Materialize 等节点揭示潜在优化点

**索引优化决策矩阵**：

| 查询特征 | 推荐索引类型 | 示例 |
|:---------|:-------------|:-----|
| 等值查询 | B-Tree | `WHERE id = 100` |
| 范围查询 | B-Tree | `WHERE created_at > '2025-01-01'` |
| 模糊匹配 | GIN / trigram | `WHERE content LIKE '%keyword%'` |
| 多列组合 | 复合索引（按选择性排序） | `(user_id, created_at)` |
| JSON 字段 | GIN | `WHERE data @> '{"key":"val"}'` |

## 常见问题

### Q1：分片键选错了怎么办？

分片键变更需要全量数据重分布，代价高昂。缓解方案：在应用层增加双写，新数据按新分片键写入，老数据异步迁移，迁移完成后切换读流量。

### Q2：半同步复制降级为异步怎么办？

当副本不可用时，半同步会等待超时后降级为异步。应配置监控告警，降级时立即通知 DBA 评估数据安全风险。

### Q3：执行计划显示索引可用但仍走全表扫描？

常见原因：统计信息过期、隐式类型转换、OR 条件中断索引使用、LIKE 前缀通配符。逐一排查并修正。

### Q4：VACUUM 跟不上怎么办？

长事务阻止 VACUUM 清理死元组。排查并终止长事务，调大 autovacuum 频率，必要时手动 VACUUM。

### Q5：主从切换后应用连不上新主？

应用连接配置未更新或 DNS 缓存未过期。方案：使用服务发现（Consul / etcd）管理数据库地址，切换时自动更新。

### Q6：PITR 恢复到哪个时间点？

根据误操作发生时间确定恢复目标。恢复时使用 recovery_target_time 参数指定精确时间点，恢复完成后立即验证数据正确性。

### Q7：如何实现零停机大版本升级？

使用逻辑复制：新版本数据库作为订阅者，同步完成后切换应用连接，验证无误后下线旧库。

### Q8：监控告警太多怎么办？

按严重程度分级（P0 立即处理 / P1 工作时间内处理 / P2 记录观察），合并相关告警，设置告静默窗口避免风暴。
