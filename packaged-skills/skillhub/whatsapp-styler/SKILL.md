# WhatsApp Styler

将标准Markdown转换为WhatsApp原生排版语法,并验证消息格式合规性。确保接收方看到干净的加粗、斜体、删除线等样式,而非原始的`**`、`#`等符号。

## 核心能力

### 1. Markdown语法转换
将标准Markdown格式映射为WhatsApp原生排版语法,确保接收端正确渲染样式。

**转换映射规则**:
- 加粗: `**text**` 转为 `*text*`(双星号转单星号)
- 斜体: Markdown `*text*` 转为 WhatsApp `_text_`(星号转下划线)
- 删除线: `~~text~~` 转为 `~text~`(双波浪线转单波浪线)
- 行内代码: 单反引号 `` `text` `` 转为三反引号 ``` ```text``` ```
- 标题: `# Header` 转为 `*HEADER*`(井号转为大写加粗)
- 二级标题: `## Header` 转为 `*Header*`(井号转为加粗,保留大小写)
- 分割线: `---` 转为 `__________`(三条横线转为下划线行)
- 表格: 转为项目符号列表,每行格式为 `* 列1: 列2`

**关键差异**: WhatsApp用单星号表示加粗(非双星号),单下划线表示斜体,单波浪线表示删除线。这与标准Markdown完全不同,直接发送Markdown会导致接收方看到原始符号而非样式效果。

**输入**: 用户提供Markdown语法转换所需的指令和必要参数。
**输出**: 返回Markdown语法转换的执行结果,包含操作状态和输出数据。

### 2. 格式合规验证
检测消息中包含WhatsApp不支持的Markdown元素,在发送前拦截问题。

**验证检查项**:
- 井号标题(`#`/`##`/`###`): WhatsApp不渲染,接收方看到原始井号
- Markdown表格(`| col |`): WhatsApp不渲染表格,显示为混乱的竖线文本
- 水平分割线(`---`): 可能被误判为删除线语法
- 双星号加粗(`**text**`): WhatsApp会显示原始星号而非加粗效果
- 双波浪线删除线(`~~text~~`): WhatsApp会显示原始波浪线
- 嵌套格式(`***text***`): 同时加粗加斜体在WhatsApp中不可靠,可能显示原始符号
- HTML标签(`<b>text</b>`): WhatsApp不支持HTML标签渲染

**输入**: 用户提供格式合规验证所需的指令和必要参数。
**输出**: 返回格式合规验证的执行结果,包含操作状态和输出数据。

### 3. 批量消息处理
对多条消息执行统一格式转换,适用于广播通知、批量通知推送等场景。

**处理策略**:
- 逐条转换: 对每条消息独立应用转换规则
- 一致性校验: 确保同批次消息格式风格统一
- 长度检查: WhatsApp单条消息文本上限65536字符,超长需拆分
- 变量保留: 转换时保留`{name}`、`{order_id}`等模板变量不被破坏

**输入**: 用户提供批量消息处理所需的指令和必要参数。
**输出**: 返回批量消息处理的执行结果,包含操作状态和输出数据。

### 4. 格式自动修复
对已检测到的格式问题自动应用修复规则,无需人工逐条修改。

**修复规则**:
- `**text**` 修复为 `*text*`(双星号修复为单星号)
- `~~text~~` 修复为 `~text~`(双波浪线修复为单波浪线)
- `# Header` 修复为 `*HEADER*`(标题修复为大写加粗)
- `| A | B |` 表格修复为 `* A: B`(表格转为列表)
- `---` 修复为 `__________`(分割线修复为下划线行)
- `<b>text</b>` 修复为 `*text*`(HTML标签转为WhatsApp加粗)

**输入**: 用户提供格式自动修复所需的指令和必要参数。
**处理**: 按照skill规范执行格式自动修复操作,遵循单一意图原则。
**输出**: 返回格式自动修复的执行结果,包含操作状态和输出数据。

### 5. 样式意图保持
转换时识别原文的强调层次(主标题、副标题、正文、强调),在WhatsApp语法中找到对应的表达方式,而非机械替换符号。

**意图映射**:
- 文档主标题: 转为 `*全大写加粗*`(WhatsApp中视觉强调最为显著)
- 文档副标题: 转为 `*加粗*`(正常大小写)
- 重要警告: 转为 `*全大写加粗*` 加换行加正文
- 正文强调: 转为 `*加粗*` 或 `_斜体_`
- 代码或技术ID: 转为 ``` ```等宽字体``` ```
- 引用说明: 转为 `> 引用文本`

**输入**: 用户提供样式意图保持所需的指令和必要参数。
**输出**: 返回样式意图保持的执行结果,包含操作状态和输出数据。

### 6. 特殊字符冲突处理
处理原文中合法包含的星号、下划线等字符与WhatsApp格式语法的冲突。

**冲突场景与处理**:
- 数学表达式 `3 * 4 = 12` 中的星号会被WhatsApp误判为加粗语法: 在星号前后加空格或用等宽字体包裹
- 文件名 `my_file.txt` 中的下划线会被误判为斜体: 用等宽字体包裹
- 日期 `2025_01_18` 中的下划线: 用等宽字体包裹
- emoji紧贴星号 `*促销*` 可能导致格式不生效: 在emoji与星号间加空格

**输入**: 用户提供特殊字符冲突处理所需的指令和必要参数。
**输出**: 返回特殊字符冲突处理的执行结果,包含操作状态和输出数据。

**输入**(Markdown):
```
## 订单确认
您的订单 **#20250118** 已确认!

### 商品明细
| 商品 | 数量 | 金额 |
|------|------|------|

| 手机壳 | 2 | ¥78 |

**合计: ¥377**
---
预计 *3-5个工作日* 送达。如有疑问请回复此消息。
```

**输出**(WhatsApp):
```
*订单确认*
您的订单 *#20250118* 已确认!

*商品明细*
* 蓝牙耳机: 1件 ¥299
* 手机壳: 2件 ¥78

*合计: ¥377*
__________
```

### 示例2: 检测并修复格式问题

**输入**(待发送的WhatsApp消息):
```
**限时优惠**
~~原价~~ ¥999 到 **现价** ¥699

活动规则:
1. 每人限购1件
2. `PROMO2025` 优惠码
3. 有效期至 #2025-01-31#
```

**检测到的问题**:
- `**限时优惠**`: 使用了双星号,WhatsApp会显示原始星号
- `~~原价~~`: 使用了双波浪线,WhatsApp会显示原始波浪线
- `**现价**`: 同上双星号问题
- `#2025-01-31#`: 井号日期格式会被WhatsApp忽略

**修复后输出**:
```
*限时优惠*
~原价~ ¥999 到 *现价* ¥699

活动规则:
1. 每人限购1件
2. ```PROMO2025``` 优惠码
3. 有效期至 *2025-01-31*
```

**修复说明**: 双星号转为单星号;双波浪线转为单波浪线;井号日期转为加粗;行内代码单反引号转为三反引号。优惠码用等宽字体显示,避免与格式符号冲突。

## 异常处理

| 错误场景 | 错误信息/现象 | 原因分析 | 处理方式 |
|---------|-------------|---------|---------|
| 双星号未转换 | 接收方看到`**text**`而非加粗 | 消息中残留Markdown双星号语法 | 将`**text**`替换为`*text*`,扫描时用正则`\*\*(.+?)\*\*`匹配所有双星号对 |
| 井号标题显示异常 | 接收方看到`# 标题`原始井号 | WhatsApp不渲染Markdown标题语法 | 将`# Header`替换为`*HEADER*`(大写加粗),二级标题`## Header`替换为`*Header*` |
| 表格显示混乱 | 接收方看到竖线和横线混乱排列 | WhatsApp不支持Markdown表格渲染 | 将表格转为项目符号列表,每行格式为`* 列1: 列2`,表头行作为列表标题 |
| 分割线被误判 | `---`后紧跟的文字变成删除线 | WhatsApp将`~`视为删除线语法,连续横线可能触发误判 | 将`---`替换为`__________`(下划线行),确保前后各空一行 |
| 星号与数学符号冲突 | `3 * 4 * 5`中的星号被渲染为加粗 | WhatsApp将成对星号间的内容识别为加粗 | 在运算符星号前后确保有空格,或用三反引号包裹整个表达式 |
| 下划线与文件名冲突 | `my_file_name`显示为斜体 | WhatsApp将成对下划线间内容识别为斜体 | 用三反引号包裹文件名: ``` ```my_file_name``` ``` |
| 嵌套格式失效 | `*_text_*`显示原始符号而非加粗斜体 | WhatsApp嵌套格式支持不稳定 | 避免嵌套,选择单一强调方式;或拆分为两行分别加粗和斜体 |
| emoji紧贴格式符号 | 优惠消息中加粗不生效 | emoji与星号间无空格导致WhatsApp解析失败 | 在emoji与格式符号间插入空格,如 `促销 *限时特惠*` |

## 常见问题

### Q1: WhatsApp为什么用单星号表示加粗,而Markdown用双星号?
A: WhatsApp的格式语法是平台自定义的,与标准Markdown不同。WhatsApp用`*text*`(单星号)表示加粗,`_text_`(单下划线)表示斜体,`~text~`(单波浪线)表示删除线。如果直接发送Markdown的`**text**`,接收方会看到原始的双星号符号而非加粗效果。这是WhatsApp Styler需要存在的核心原因:在发送前完成Markdown到WhatsApp语法的转换。

### Q2: 消息中需要包含数学公式(含星号)怎么办?
A: 数学表达式中的乘号星号(如`3 * 4 = 12`)会被WhatsApp误判为加粗语法的起始或结束符。三种处理方式:(1)确保运算符星号前后都有空格,WhatsApp只将紧贴文字的星号识别为格式符;(2)用三反引号将整个表达式包裹为等宽字体;(3)将乘号替换为`×`符号避免冲突。推荐方式2,等宽字体还能保持公式对齐。

### Q3: 可以在WhatsApp消息中使用标题吗?
A: WhatsApp不支持Markdown的`#`/`##`/`###`标题语法,发送`# 标题`接收方会看到原始井号。替代方案:用`*全大写加粗*`作为主标题(WhatsApp中视觉强调最为显著),用`*加粗*`(正常大小写)作为副标题。例如文档主标题用`*订单确认*`,段落小标题用`*商品明细*`。这样既保持视觉层次,又符合WhatsApp语法。

### Q4: WhatsApp如何显示代码或技术ID?
A: WhatsApp用三反引号表示等宽字体(monospace),适合显示代码片段、订单号、验证码等技术内容。注意:标准Markdown用单反引号表示行内代码,但WhatsApp需要三反引号。转换时将单反引号替换为三反引号。例如Markdown的`` `PROMO2025` ``转为WhatsApp的``` ```PROMO2025``` ```。等宽字体还能避免技术内容中的下划线被误判为斜体。

### Q5: 批量处理多条消息时如何保证模板变量不被破坏?
A: 转换前先识别模板变量模式(如`{name}`、`{order_id}`、`%user%`),在转换规则中将这些模式加入白名单,不对其内部内容应用格式转换。具体做法:用正则匹配常见变量模式,匹配到的内容原样保留。如果变量名中包含下划线(如`{user_name}`),需特别注意下划线不要被误判为斜体语法,必要时用等宽字体包裹变量。

### Q6: WhatsApp消息有长度限制吗?超长消息怎么处理?
A: WhatsApp单条文本消息上限65536字符,超过会被截断。实际建议单条消息控制在1000字符以内保证可读性。超长内容处理:(1)按语义段落拆分为多条消息;(2)用`__________`(下划线行)作为分页分隔符;(3)长列表按10至15项一组拆分,每组末尾加"更多请回复N"引导。批量处理时Agent会自动检查每条消息长度并提示拆分建议。

## 已知限制

- **不支持富媒体**: 本Skill仅处理文本格式转换,不涉及图片、视频、文档、按钮等WhatsApp Business模板消息元素
- **格式渲染依赖客户端**: WhatsApp格式渲染效果因客户端版本而异,旧版客户端可能不支持部分格式
- **嵌套格式不可靠**: WhatsApp对嵌套格式(如同时加粗加斜体)支持不稳定,建议避免使用
- **特殊字符需人工判断**: 含大量星号、下划线的技术文本(如正则表达式、代码)需人工确认转换结果
- **不处理消息发送**: 本Skill负责格式转换与验证,不涉及WhatsApp API调用与消息发送
- **语言无关但示例偏中文**: 转换规则适用于所有语言,但示例以中文场景为主,其他语言场景可能需额外适配
