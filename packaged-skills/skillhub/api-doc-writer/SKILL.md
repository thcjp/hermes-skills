# API 接口文档编写器

API 接口文档编写助手。用于编写 REST API 文档、定义接口规范、生成接口说明。提供完整文档模板、RESTful 设计规范与安全建议。

## RESTful 设计规范

### HTTP 方法语义
| 方法 | 用途 | 示例 |
|------|------|------|
| GET | 查询资源 | `GET /api/v1/users` |
| POST | 创建资源 | `POST /api/v1/users` |
| PUT | 完整更新 | `PUT /api/v1/users/1` |
| PATCH | 部分更新 | `PATCH /api/v1/users/1` |
| DELETE | 删除资源 | `DELETE /api/v1/users/1` |

### URL 命名规范
- 使用名词复数: `/api/v1/users`
- 使用小写: `/api/v1/user-info`
- 使用连字符分隔: `/api/v1/order-details`
- 避免动词: 不用 `/api/v1/getUser`

### HTTP 状态码分类
| 类别 | 状态码 | 说明 |
| --- | --- | --- |
| 1xx | 100-101 | 信息,请求正在处理 |
| 2xx | 200-206 | 成功,请求正常处理完毕 |
| 3xx | 300-305 | 重定向,需附加操作完成请求 |
| 4xx | 400-415 | 客户端错误,请求有语法错误 |
| 5xx | 500-505 | 服务器错误,服务器处理出错 |

## 安全建议

- **敏感信息加密传输**: 密码、Token 等字段使用 HTTPS + 字段级加密
- **身份验证 Token 过期机制**: `access_token` 有效期 2 小时,`refresh_token` 有效期 7 天
- **接口调用频率限制**: 单 IP 60 次/分钟,单用户 1000 次/小时
- **参数校验过滤**: 服务端校验必填项、类型、长度、格式,拒绝非法输入

## 案例展示

### 案例1: 用户信息接口文档
**场景**: 为获取用户信息接口编写文档

**接口地址**:
```
GET /api/v1/users/{id}
```

**请求参数**:
| 参数名 | 类型 | 位置 | 必填 | 说明 |
| --- | --- | --- | --- | --- |
| id | long | path | 是 | 用户ID |

**请求示例**:
```
GET /api/v1/users/123
```

**响应示例**:
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "id": 123,
    "name": "张三",
    "email": "zhangsan@example.com",
    "phone": "13800138000",
    "created_at": "2024-01-01 10:00:00"
  }
}
```

**错误示例**:
```json
{
  "code": 3001,
  "message": "用户不存在",
  "data": null
}
```

**说明**: GET 请求使用 path 参数传递资源 ID,响应使用统一格式 `code`/`message`/`data`。用户不存在时返回业务码 `3001`。

### 案例2: 订单列表接口文档（含分页）
**场景**: 为订单列表接口编写文档,含分页参数

**接口地址**:
```
GET /api/v1/orders
```

**请求参数**:
| 参数名 | 类型 | 位置 | 必填 | 说明 |
| --- | --- | --- | --- | --- |
| page | int | query | 否 | 页码,默认 1 |
| page_size | int | query | 否 | 每页数量,默认 20 |
| status | string | query | 否 | 订单状态 |

**请求示例**:
```
GET /api/v1/orders?page=1&page_size=10&status=paid
```

**响应示例**:
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "total": 100,
    "page": 1,
    "page_size": 10,
    "list": [
      {
        "id": "ORD202401010001",
        "user_id": 123,
        "amount": 100.00,
        "status": "paid",
        "created_at": "2024-01-01 10:00:00"
      }
    ]
  }
}
```

**说明**: 列表接口使用 query 参数分页,默认 `page=1`、`page_size=20`。响应含 `total`/`page`/`page_size`/`list` 分页元数据。

### 案例3: 创建用户接口文档
**场景**: 为创建用户接口编写文档

**接口地址**:

**请求参数**:
| 参数名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| name | string | 是 | 用户名 |
| email | string | 是 | 邮箱 |
| phone | string | 否 | 手机号 |
| password | string | 是 | 密码 |

## 错误处理


| 错误场景 | 业务码 | 原因分析 | 处理方式 |
|---------|--------|---------|---------|
| 参数缺失 | `1001` | 必填参数未传 | 检查请求参数表,补全必填项 |
| 参数类型错误 | `1001` | 传入了错误类型（如 string 传 int） | 按参数表要求修正类型 |
| 未授权 | `2001` | `Authorization: Bearer` 头缺失或 token 过期 | 重新获取 access_token 后执行ping命令测试网络连通性,检查防火墙和代理设置连接后重新执行命令 |
| 资源不存在 | `3001` | 请求的资源 ID 不存在 | 核实资源 ID 是否正确 |
| 服务器错误 | `5001` | 服务端处理异常 | 联系后端排查日志,附带请求 ID |
| 重复创建 | `1001` | 唯一约束冲突（如邮箱已注册） | 先查询资源是否存在再创建 |
| 频率超限 | `2001` | 超过 60 次/分钟限制 | 等待 60 秒后执行ping命令测试网络连通性,检查防火墙和代理设置连接后重新执行命令,或申请提升配额 |

## 常见问题

### Q1: 业务状态码和 HTTP 状态码有什么区别?
A: HTTP 状态码（200/400/500 等）由协议层返回,业务状态码（0/1001/2001/3001/5001）在响应 body 的 `code` 字段中返回。推荐: HTTP 层返回 200,业务层通过 `code` 区分成功与失败,便于前端统一处理。

### Q2: URL 应该用动词还是名词?
A: RESTful 规范使用名词复数,动词通过 HTTP 方法表达。用 `GET /api/v1/users` 而非 `GET /api/v1/getUsers`;用 `POST /api/v1/users` 而非 `POST /api/v1/createUser`。

### Q3: 分页参数默认值怎么设?
A: 推荐 `page` 默认 1（首页）,`page_size` 默认 20（每页 20 条）。同时设置最大值（如 `page_size` 不超过 100）防止一次拉取过多数据。

### Q4: 敏感字段如密码怎么处理?
A: 请求中密码字段使用 HTTPS 加密传输,响应中永不返回密码字段。建议密码使用 bcrypt 等算法哈希存储,不存明文。

### Q5: 接口版本怎么管理?
A: 在 URL 路径中标注版本: `/api/v1/users`、`/api/v2/users`。版本升级时保留旧版本一段时间,在变更记录中标注废弃时间与新版本迁移指南。

### Q6: 如何记录接口变更?
A: 使用变更记录表: 版本号、日期、变更内容、变更人。如 `V1.1 | 2024-03-01 | 新增支付回调接口 | @zhangsan`。每次接口变更都需更新版本号与记录。

## 已知限制

1. **纯文档生成**: 生成 Markdown 文档,不生成代码框架或 mock 服务
2. **需人工补充业务逻辑**: 模板提供结构,具体业务参数需开发者填写
3. **不含 API 测试**: 不提供接口测试执行能力,仅编写文档
4. **模板为通用型**: 特殊行业（如银行、医疗）可能需自定义状态码体系
5. **不含 OpenAPI/Swagger 格式**: 输出为 Markdown 文档,不生成 YAML/JSON 规范文件
6. **单语言示例**: 示例以 curl/JSON 为主,不提供多语言 SDK 示例
