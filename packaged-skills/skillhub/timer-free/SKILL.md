# timer-free

在后台运行基础定时器。当定时器完成时，你会收到一条系统通知，必须以提醒形式告知用户。免费版支持秒、分、时三种时间格式与自定义提醒文案，满足日常烹饪、休息、短时提醒等基础计时需求。

## Quick Start

```bash
bash background:true command:"node {baseDir}/timer.js 5m"
bash background:true command:"node {baseDir}/timer.js 10m 'Check the oven'"
bash background:true command:"node {baseDir}/timer.js 30s"
```

第一个参数为时长，第二个可选参数为提醒文案。提醒文案会在定时器完成时回传给 Agent，再由 Agent 转达给用户。

## Time Formats

| Format | Description | Example |
| --- | --- | --- |
| `Ns` | N 秒 | `30s`, `90s` |
| `Nm` | N 分钟 | `5m`, `15m` |
| `Nh` | N 小时 | `1h`, `2h` |
| `N` | N 分钟（默认单位） | `5` 等价于 5 分钟 |

免费版支持以上四种基础格式。带冒号的复合格式（如 `MM:SS`、`HH:MM:SS`）属于付费版能力。

## 适用场景

### 烹饪与厨房计时
用户在做饭时需要简单的倒计时提醒，例如意面 12 分钟、蒸蛋 8 分钟、泡面 3 分钟。启动一个定时器，到期后通过系统通知提醒用户关火或取出食物，避免煮过头或忘记关火。提醒文案可写明食材名称，到期时直接告知用户该处理哪一道菜。

### 休息与短时提醒
工作一段时间后设置 5 分钟或 10 分钟的休息提醒，或为短暂的待办事项设置倒计时，到期提醒帮助及时切换上下文。适合不需要并行管理的单次计时场景，例如起身活动、眼睛休息、短暂离开座位等。

### 短时任务节奏控制
为单一任务设置一个倒计时节点，例如 30 分钟内完成一份草稿、15 分钟内回复邮件。到期提醒帮助控制任务节奏，避免在单一任务上投入过多时间。完成后由用户决定是否启动下一个计时。

## 使用案例

### 案例一：烹饪计时

用户煮意面需要 12 分钟，希望到期提醒。

```bash
bash background:true command:"node {baseDir}/timer.js 12m 'Pasta is ready!'"
```

定时器在后台运行，12 分钟后收到系统通知。Agent 收到通知后立即向用户转达提醒："⏰ 您的 12 分钟计时已到：Pasta is ready!"。回复必须直接以提醒文案开头，不要以 `HEARTBEAT_OK` 开头，否则提醒会被过滤。

### 案例二：休息提醒

用户工作一段时间后希望 5 分钟后提醒休息。

```bash
bash background:true command:"node {baseDir}/timer.js 5m 'Take a break'"
```

5 分钟后到期，Agent 转达提醒文案。需要继续工作时，可由用户再次请求启动新的定时器。

## Timer Completion Notification

当定时器完成时，会收到一条 `System:` 消息：

```text
System: [2026-01-24 21:27:13] Exec completed (swift-me, code 0) :: ⏰ Timer complete! Check the pasta!
```

收到完成通知后，回复必须直接以提醒内容开头，例如：

```text
⏰ Timer Alert! Your timer is complete: Check the pasta!
```

不要以 `HEARTBEAT_OK` 开头。以 `HEARTBEAT_OK` 开头且后续内容少于 300 字符的回复会被自动过滤，用户将收不到提醒。

退出码含义：
- `code 0`：定时器正常完成，应向用户转达提醒文案
- `code 130`：定时器被终止，属于用户主动取消，无需再提示到期

## Notes

- 定时器以独立后台进程运行，不阻塞当前会话
- 完成时退出码为 0，被 kill 终止时退出码为 130
- 免费版每次启动一个定时器，到期后由用户决定是否再次启动
- macOS 上完成时会尝试调用 `afplay` 播放提示音，其他系统仅依赖文本通知

## 异常处理

### 启动命令未识别 time 参数
当第一个参数为空或无法解析为已知时间格式时，timer.js 会立即退出并输出错误。处理：确认参数使用了支持的格式（`Ns`/`Nm`/`Nh`/`N`），不要传入纯字符串如 "ten minutes"。

### 提醒文案含引号导致命令解析失败
当提醒文案中包含单引号或双引号时，可能导致 shell 解析截断。处理：用与外层不同的引号包裹文案，或将文案中的引号转义；复杂文案建议仅用字母、数字、空格与常见标点。

### 完成通知未送达用户
若 Agent 回复以 `HEARTBEAT_OK` 开头且字符数不足 300，回复会被自动过滤。处理：回复直接以提醒文案开头（如 `⏰ ...`），不要以 `HEARTBEAT_OK` 开头。

### kill 后仍收到完成通知
通过 `process action:kill` 终止定时器后，仍会收到一条退出码为 130 的完成通知。这是预期行为而非错误。处理：将退出码 130 识别为"用户已取消"，不必再向用户提示到期。

### 定时器在会话结束后失效
免费版定时器依赖 Agent 后台进程，会话结束或重启后进程会被清理。处理：需要跨会话提醒时，改用外部日历或系统级定时任务。

## FAQ

### 能否设置循环定时器？
不能。timer.js 每次只执行一次倒计时。需要循环时，在收到完成通知后由用户再次请求启动下一个定时器。

### 提醒文案是否支持中文？
支持。提醒文案作为字符串参数传入，会在完成通知中原样回传，中文、英文、emoji 与常见标点均可使用。注意文案中的引号需正确转义。

### 如何取消正在运行的定时器？
执行 `process action:list` 找到目标定时器的 sessionId，再执行 `process action:kill sessionId:XXX` 终止。终止后会收到退出码 130 的完成通知，Agent 应识别为取消。

### 免费版支持哪些时间格式？
支持 `Ns`（秒）、`Nm`（分钟）、`Nh`（小时）、`N`（默认分钟）四种基础格式。带冒号的复合格式（`MM:SS`、`HH:MM:SS`）属于付费版能力。

## 已知限制

- 仅支持秒、分、时及纯数字四种基础时间格式，不支持 `MM:SS`、`HH:MM:SS` 复合格式
- 不支持多定时器并行管理的进阶能力（如日志查看、批量轮询）
- 定时器仅在当前 Agent 会话存活，会话结束即失效
- 不支持循环/重复定时，需要循环时须由用户手动重启
- 不支持指定绝对触发时刻，只支持从当前时刻开始的相对时长
- 声音提示仅在 macOS 上播放，其他系统仅依赖文本通知

## 升级提示

本免费版仅提供基础倒计时能力。如需复合时间格式（`MM:SS`/`HH:MM:SS`）、多定时器并行管理、日志查看、剩余时间轮询、番茄工作法循环等进阶能力，请升级到付费版 timer，解锁完整的后台定时器与提醒功能。
