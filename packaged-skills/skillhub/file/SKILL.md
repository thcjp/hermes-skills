# File 文件管理整理

文件与文件夹管理的完整工具，从命名规范到文件夹结构到关键文档归档，帮助建立可持续的文件管理体系。

## 核心能力

### 文件命名系统构建（Naming Conventions）

大多数文件混乱始于命名问题。本能力根据用户的实际文件类型构建个性化命名规范：

- **命名原则**：名称足够具体，不打开文件就能知道内容；日期使用可排序格式（`2026-04-17`）；版本标记让当前版本一目了然
- **搜索友好**：使用你实际会输入的搜索词命名，而非保存时觉得合理的词
- **版本管理**：用 `v1`、`v2`、`FINAL` 区分版本，避免 `final`、`final2`、`final_USE_THIS_ONE` 的混乱
- **截图命名**：将 `Screen Shot 2023-04-17 at 11.43.22 AM` 重命名为内容描述格式
- **批量重命名**：根据实际文件生成命名规则，通过实际文件教学而非抽象示例

### 文件夹结构设计（Folder Structure）
围绕文件检索方式而非创建方式组织文件夹结构：

- **浅层结构**：不超过 3-4 层深度，任何文件 3-4 次点击内可找到
- **一致性**：结构足够一致，总是大致知道某个文件应该在哪里
- **从零设计或重组**：支持从零设计新结构或重组现有结构，重组前先展示完整计划，经用户审批后才移动任何文件
- **项目文件夹模板**：每个项目统一分类 — 工作文档、参考材料、assets、deliverables、communications
- **归档机制**：项目完成后干净归档 — 压缩需保留的、删除不需要的，确保一年后客户提问时能快速找到

**处理**: 按照skill规范执行文件夹结构设计（Folder Structure）操作,遵循单一意图原则。
### 文件检索与找回（Finding Lost Files）
通过自然语言描述检索找不到的文件：

- **描述式搜索**：描述你记得的文件信息 — 大致内容、创建或接收时间、关联项目、文件格式、名称片段
- **构建精准搜索**：从描述细节构建最精准的搜索条件
- **根因分析**：危机过后分析文件为什么难找 — 检索问题几乎总是命名或组织问题的伪装
- **预防措施**：确保类似文件下次可在秒级而非分钟级找到

**输入**: 用户提供文件检索与找回（Finding Lost Files）所需的指令和必要参数。
### Downloads 与 Desktop 清理（Inbox Zero for Files）
像邮件 Inbox Zero 一样处理文件堆积点：

- **Downloads 处理**：Downloads 是文件被遗忘的地方。每个文件要么归档、要么删除、要么有意延期并标注原因
- **Desktop 清理**：Desktop 是没时间放好时文件的去处。清理到每个剩余文件都是有意放置的
- **定期处理**：每周 5 分钟检查 Downloads 文件夹，项目结束时 15 分钟收尾清理
- **系统 vs 积累**：区分运行的系统和积累债务的系统 — 定期处理的文件环境保持可控

**输入**: 用户提供Downloads 与 Desktop 清理（Inbox Zero for Files）所需的指令和必要参数。
**输出**: 返回Downloads 与 Desktop 清理（Inbox Zero for Files）的执行结果,包含操作状态和输出数据。### 关键文档 inventory 管理（Critical Documents）
识别和管理有实际后果的重要文档：

- **文档类型**：合同、税务文件、医疗记录、财务报表、法律信函 — 丢失或无法访问会有实际后果
- **存储与命名**：确保关键文档以可访问的方式存储、命名和备份，因为需要时通常很紧急
- **文档清单**：建立关键文档的简单 inventory — 是什么、存在哪里、何时到期或需更新
- **到期追踪**：追踪文档的有效期和更新时间，避免需要时不知道是否有这份文件
- **备份策略**：关键文档多重备份，确保可访问性

**输入**: 用户提供关键文档 inventory 管理（Critical Documents）所需的指令和必要参数。
**处理**: 按照skill规范执行关键文档 inventory 管理（Critical Documents）操作,遵循单一意图原则。
**输出**: 返回关键文档 inventory 管理（Critical Documents）的执行结果,包含操作状态和输出数据。### 文件维护习惯养成（Maintenance Without Effort）
建立足够轻量可持续的文件管理习惯：

- **低摩擦原则**：正确归档只需 10 秒而非 10 分钟，结构清晰到不存在"该放哪"的问题
- **周度检查**：每周 5 分钟检查 Downloads 文件夹
- **项目收尾**：项目结束 15 分钟收尾流程，不遗留任何未处理文件
- **季度回顾**：季度检查容易积累杂物的文件夹
- **目标**：不是完美系统，而是维护成本足够低以至于你实际会维护的系统

**输入**: 用户提供文件维护习惯养成（Maintenance Without Effort）所需的指令和必要参数。

## 错误处理

| 错误场景 | 原因 | 处理方式 |
|---------|------|---------|
| 文件移动后无法找到 | 新位置不符合用户记忆习惯 | 移动前展示完整计划，经用户审批后再执行，保留移动日志 |
| 批量重命名冲突 | 目标名称已存在或格式冲突 | 重命名前检查冲突，自动添加序号后缀或提示用户手动决定 |
| 文件夹结构过深 | 层级超过 3-4 层导致检索困难 | 限制最大深度，将深层内容提升到浅层或拆分子项目 |
| 关键文档误删 | 清理时未识别为重要文件 | 关键文档 inventory 预先标记，删除前与 inventory 比对 |
| 搜索无结果 | 文件名不含搜索关键词或已被移动 | 使用内容描述搜索，检查 archive 文件夹和 trash |
| 权限不足无法操作 | 系统文件或受保护目录 | 确认操作权限，跳过系统文件，仅处理用户目录 |
| 备份失败 | 存储空间不足或路径不可达 | 检查目标存储空间和路径可访问性，多重备份策略容错 |
| 项目归档不完整 | 未清理临时文件或遗漏交付物 | 使用项目收尾检查清单，逐项确认后再归档压缩 |

## 常见问题

### Q1: 如何开始整理一个混乱的桌面？
A: 先扫描所有文件，按类型分类（截图、文档、文件夹、临时文件）。生成命名规范和目标文件夹结构，展示完整移动计划，用户审批后批量执行。每周 5 分钟检查保持整洁。

### Q2: 文件命名规范应该怎么定？
A: 根据你实际搜索时输入的关键词命名。日期用 `2026-04-17` 格式可排序。版本用 `v1/v2/FINAL` 标记。名称要足够具体，不打开文件就知道内容。

### Q3: 关键文档如何确保不丢失？
A: 建立文档 inventory — 记录是什么、存在哪里、何时到期。多重备份（本地 + 云端）。合同、税务、医疗等文件单独标记，清理时自动跳过。

## 已知限制

- 无法处理加密文件或受密码保护的文档
- 文件操作需要 Agent 具备文件系统 exec 权限
- 重组大量文件可能耗时较长，建议分批处理
- 无法恢复已被永久删除的文件（需数据恢复工具）
