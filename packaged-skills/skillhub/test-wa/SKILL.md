# WhatsApp 消息助手

基于 wacli CLI 实现 WhatsApp 消息发送、聊天历史检索与同步。仅当用户明确要求向他人发送 WhatsApp 消息或同步/检索历史时使用。用户与 Agent 的常规聊天不应使用此工具。

## 核心能力

- **多类型消息发送**：向个人发送文本消息、向群组发送文本、向个人发送文件（含 caption 说明）
- **聊天列表查询**：按名称或号码查询聊天列表，支持 limit 限制返回数量
- **消息检索**：按关键词检索消息，支持按聊天 JID 过滤、按日期范围（after/before）过滤
- **历史回填**：对指定聊天回填历史消息，支持控制请求数与每请求消息数
- **持续同步**：通过 `sync --follow` 持续同步新消息
- **认证与诊断**：二维码登录认证、环境诊断（doctor）
- **JID 格式识别**：直聊 JID 格式为 `<number>@s.whatsapp.net`，群组 JID 格式为 `<id>@g.us`
- **JSON 输出**：支持 `--json` 参数输出机器可读格式，便于解析与后续处理
### 多类型消息发送

执行多类型消息发送操作,处理用户输入并返回结果。

**输入**: 用户提供多类型消息发送所需的参数和指令。

**输出**: 返回多类型消息发送的处理结果。
### 聊天列表查询

执行聊天列表查询操作,处理用户输入并返回结果。

**输入**: 用户提供聊天列表查询所需的参数和指令。

**输出**: 返回聊天列表查询的处理结果。
### 消息检索

执行消息检索操作,处理用户输入并返回结果。

**输入**: 用户提供消息检索所需的参数和指令。

**输出**: 返回消息检索的处理结果。


## 工具命令

### 认证与同步
- `wacli auth`：二维码登录与初始同步
- `wacli sync --follow`：持续同步新消息
- `wacli doctor`：诊断环境与连接状态

### 聊天与消息查询
- `wacli chats list --limit 20 --query "名称或号码"`：查询聊天列表
- `wacli messages search "关键词" --limit 20 --chat <jid>`：按关键词与聊天检索消息
- `wacli messages search "关键词" --after 2025-01-01 --before 2025-12-31`：按日期范围检索

### 历史回填
- `wacli history backfill --chat <jid> --requests 2 --count 50`：回填指定聊天历史

### 消息发送
- `wacli send text --to "+14155551212" --message "文本内容"`：发送文本消息
- `wacli send text --to "1234567890-123456789@g.us" --message "群组消息"`：发送群组消息
- `wacli send file --to "+14155551212" --file /path/agenda.pdf --caption "议程"`：发送文件

## 适用场景

| 场景 | 输入 | 输出 |
|------|------|------|
| 发送会议提醒消息 | 收件人号码与提醒文本 | 向收件人发送 WhatsApp 文本消息 |
| 群组文件分发 | 群组 JID 与文件路径 | 向群组发送文件并附带 caption |
| 历史消息检索 | 关键词与日期范围 | 返回匹配的消息记录列表 |
| 聊天历史回填 | 聊天 JID 与回填深度 | 从服务器拉取历史消息并存储到本地 |

**不适用于**：用户与 Agent 的常规聊天（自动路由无需此工具）、需要端到端加密保障的敏感通信、大规模营销群发。

## 使用流程

1. **确认认证状态**：首次使用运行 `wacli auth` 扫码登录，完成后运行 `wacli doctor` 确认连接正常
2. **识别收件人 JID**：通过 `wacli chats list --query "名称"` 查找目标聊天，获取 JID。直聊为 `<number>@s.whatsapp.net`，群组为 `<id>@g.us`
3. **确认发送内容**：向用户预览收件人与消息内容，确认后执行发送
4. **检索与回填**：按需使用 `messages search` 或 `history backfill`，回填需手机在线
5. **JSON 输出解析**：需要程序化处理结果时加 `--json` 参数

## 认证与同步

### 首次登录

运行 `wacli auth` 后终端显示二维码，打开手机 WhatsApp 的"设置 - 已关联设备"扫描二维码完成登录。登录后自动执行初始同步，拉取已有聊天与消息到本地存储。

### 持续同步

`wacli sync --follow` 启动持续同步进程，实时拉取新消息。该进程为长驻进程，需保持运行才能持续接收更新。同一时间仅允许一个 `sync --follow` 进程，多个实例会产生冲突。

### 环境诊断

`wacli doctor` 检查认证状态、存储目录完整性与连接质量，输出诊断报告。遇到同步异常或发送失败时优先运行 doctor 排查环境问题。

## 安全机制

- 发送消息必须明确指定收件人与消息文本，两者缺一不可
- 发送前向用户确认收件人号码/JID 与消息内容
- 信息有歧义时先提问澄清，不猜测收件人或内容
- 不用于用户与 Agent 的常规聊天，仅用于向第三方发送消息

## 注意事项

- 存储目录默认为 `~/.wacli`，可通过 `--store` 参数覆盖到自定义路径
- JID 格式严格区分：直聊为 `<number>@s.whatsapp.net`，群组为 `<id>@g.us`，不确定时用 `chats list` 查询
- 历史回填为尽力而为（best-effort），部分消息可能因服务器策略无法获取
- 回填与同步均要求手机端 WhatsApp 保持在线，手机离线时这些操作会失败
- 需要程序化解析结果时加 `--json` 参数，输出结构化 JSON 而非人类可读文本

## 案例展示

### 案例 1：向联系人发送会议提醒

**触发**：需要提醒联系人下午 3 点的会议

**执行**：

```bash
wacli send text --to "+14155551212" --message "下午 3 点的会议还准时吗？"
```

**结果**：联系人 `+14155551212` 收到一条 WhatsApp 文本消息。发送前已向用户确认收件人与消息内容。

### 案例 2：向群组发送议程文件

**触发**：需要向项目群组发送会议议程 PDF

**执行**：

```bash
wacli send file --to "1234567890-123456789@g.us" --file /path/agenda.pdf --caption "今日会议议程"
```

**结果**：群组收到一份 PDF 文件，附带 caption"今日会议议程"。群组 JID 通过 `wacli chats list --query "项目群"` 获取。

### 案例 3：按日期范围检索历史消息

**触发**：需要查找 2025 年 1-6 月间包含"发票"的消息

**执行**：

```bash
wacli messages search "发票" --after 2025-01-01 --before 2025-06-30 --limit 20
```

**结果**：返回最多 20 条匹配消息，含发送时间、发送者、聊天 JID 与消息内容。加 `--json` 可输出结构化数据供后续处理。

## 错误处理


| 错误 | 原因 | 处理方式 |
|------|------|---------|
| 认证失败 / 未登录 | 未完成 `wacli auth` 或登录已过期 | 重新运行 `wacli auth` 扫码登录，完成后 `wacli doctor` 验证 |
| 手机离线导致回填失败 | 历史回填要求手机在线 | 确保手机 WhatsApp 保持在线后执行ping命令测试网络连通性,检查防火墙和代理设置连接后重新执行命令 `history backfill` |
| JID 格式错误 | 直聊与群组 JID 混用或格式不完整 | 直聊用 `<number>@s.whatsapp.net`，群组用 `<id>@g.us`，通过 `chats list` 获取正确 JID |
| 收件人不在联系人 | 号码未注册 WhatsApp 或格式错误 | 确认号码含国际区号（如 `+1`），收件人需有有效 WhatsApp 账号 |
| 文件路径不存在 | `--file` 指向的路径无效 | 确认文件路径正确且可读，使用绝对路径避免相对路径歧义 |
| 速率限制 | 短时间发送大量消息触发限流 | 降低发送频率，消息间加入合理间隔 |
| sync 冲突 | 多个 `sync --follow` 进程同时运行 | 仅保留一个同步进程，终止多余实例后重新启动 |
| doctor 报告异常 | 存储目录损坏或连接状态异常 | 运行 `wacli doctor` 查看诊断详情，按提示修复存储目录 `~/.wacli` |

## 常见问题

### Q1：如何完成 wacli 首次登录？
A：运行 `wacli auth`，终端会显示二维码。打开手机 WhatsApp 设置中的"已关联设备"，扫描终端二维码完成登录。登录后自动执行初始同步。完成后运行 `wacli doctor` 确认连接状态正常。

### Q2：直聊和群组的 JID 格式有什么区别？
A：直聊 JID 格式为 `<number>@s.whatsapp.net`（如 `14155551212@s.whatsapp.net`），群组 JID 格式为 `<id>@g.us`（如 `1234567890-123456789@g.us`）。不确定 JID 时运行 `wacli chats list --query "名称或号码"` 查询，返回结果中包含正确的 JID。

### Q3：数据存储在哪里？
A：默认存储目录为 `~/.wacli`，包含认证凭证、同步的消息与聊天数据。可通过 `--store` 参数覆盖存储路径。不要手动修改该目录内容，如需重置认证删除该目录后重新运行 `wacli auth`。

### Q4：`--json` 参数有什么用？
A：`--json` 让命令输出 JSON 格式而非人类可读文本，便于程序化解析与后续处理。在需要将结果传递给其他工具或脚本时使用。例如 `wacli chats list --json --limit 20` 返回结构化的聊天列表数据。

### Q5：历史回填有什么前提条件？
A：回填要求手机 WhatsApp 保持在线状态，因为回填通过手机端向服务器请求历史数据。结果为尽力而为（best-effort），部分历史消息可能因服务器策略无法获取。通过 `--requests` 控制请求数、`--count` 控制每请求消息数，逐步回填。

### Q6：什么时候应该用 `wacli`，什么时候不用？
A：仅当用户明确要求向他人发送 WhatsApp 消息或检索/同步历史时使用。用户与 Agent 的常规聊天由平台自动路由，不应使用此工具。用户在 WhatsApp 上与 Agent 对话时，除非要求联系第三方，否则不需要调用 wacli。

## 已知限制

- 历史回填要求手机在线，结果为尽力而为，部分消息可能无法获取
- JID 格式严格区分直聊（`@s.whatsapp.net`）与群组（`@g.us`），混用会失败
- 发送消息必须明确收件人与内容，不支持模糊发送
- 不支持端到端加密内容读取，仅处理明文消息
- 速率限制存在，短时间大量发送会被限流
- 存储目录 `~/.wacli` 不可手动修改，损坏后需重新认证
- 不用于用户与 Agent 的常规聊天，仅用于向第三方发送消息
- 性能取决于手机端 WhatsApp 的在线状态与网络质量
