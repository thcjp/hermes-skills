## 核心能力

### 1. 引号规则
- 含逗号、引号或换行的字段**必须**用双引号包裹
- 引号字段内部的双引号转义为 `""`（两个引号），而非反斜杠
- 未引号字段的前导/尾随空格——部分解析器会裁剪，部分不会；要保留就引号
- 空字段 `，` 与空字符串 `,"",` 语义不同；要显式

**输入**: 用户提供引号规则所需的指令和必要参数.
**输出**: 返回引号规则的处理结果,包含执行状态码、结果数据和执行日志.
### 2. 分隔符识别
- CSV不总是逗号——检测 `;`（欧洲Excel）、`\t`（TSV）、`|`（遗留系统）
- Excel导出使用系统区域设置的分隔符；非美国地区常见分号
- 从首行嗅探分隔符但要验证——表头可能不含特殊字符

**输入**: 用户提供分隔符识别所需的指令和必要参数.
**处理**: 解析分隔符识别的输入参数,执行核心处理逻辑,返回结构化结果和执行状态.
### 3. 编码处理
- UTF-8 BOM（`0xEF 0xBB 0xBF`）会破坏简单解析器，但Excel需要它来识别UTF-8
- 为Windows上的Excel生成时：加BOM；为程序化使用时：省略BOM
- Latin-1与UTF-8歧义——解析前显式声明或检测编码

**输入**: 用户提供编码处理所需的指令和必要参数.
### 4. 常见解析失败
- 引号字段内的换行是合法的——不要在解析前按 `\n` 拆分
- 字段中间未转义的引号会破坏文件剩余部分——尽早校验
- EOF的尾随换行——部分解析器会创建空末行；剥离或处理
- 每行列数不一致——校验所有行匹配表头列数

**输入**: 用户提供常见解析失败所需的指令和必要参数.
### 5. 数字与日期
- `1,234.56` vs `1.234,56`——依赖区域设置；标准化或文档化格式
- 日期：ISO 8601（`2024-01-15`）是唯一无歧义格式；`01/02/24` 是混乱
- 数字字段中的前导零（`007`）——引号以保留或文档化为字符串

**输入**: 用户提供数字与日期所需的指令和必要参数.
**处理**: 解析数字与日期的输入参数,执行核心处理逻辑,返回结构化结果和执行状态.
**输出**: 返回数字与日期的处理结果,包含执行状态码、结果数据和执行日志.
### 6. Excel特性

- 公式注入：以 `=`、`+`、`-`、`@` 开头的字段会作为公式执行——前缀加 `'` 或制表符
- 长数字（>15位）丢失精度——引号并格式化为文本
- 数字中的 `E` 触发科学计数法——需字面文本时引号

## 使用流程

1. **识别目标工具**：确认CSV的消费方（Excel/Google Sheets/pandas/数据库导入）.
2. **选择分隔符**：默认逗号；欧洲Excel用分号；TSV用 `\t`；遗留系统用 `|`.
3. **决定编码与BOM**：Excel Windows加BOM；程序化使用省略BOM；显式声明UTF-8或Latin-1.
4. **应用引号规则**：含逗号/引号/换行的字段双引号包裹；内部引号转义为 `""`.
5. **处理数字与日期**：数字标准化或文档化；日期用ISO 8601；前导零引号保留.
6. **防御Excel特性**：`=+-@` 开头字段加 `'` 前缀；>15位数字引号；`E` 数字引号.
7. **校验列数一致**：所有行匹配表头列数；尽早校验未转义引号.
8. **处理尾随换行**：剥离EOF尾换行或处理空末行.
9. **跨工具验证**：在目标工具中打开验证渲染正确.
**结果验证**: 任务完成后,查看输出确认状态。成功时返回摘要和数据;失败时根据错误信息排查,参考恢复章节获取修复步骤.

## 错误处理

| 错误场景 | 原因 | 处理方式 |
|:---:|:---:|:---:|
| 字段含逗号未引号 | 违反RFC 4180引号规则 | 用双引号包裹含逗号/引号/换行的字段 |
| 内部引号未转义 | 单引号而非 `""` | 引号字段内部的双引号转义为两个双引号 |
| Excel中文乱码 | 缺UTF-8 BOM | 为Windows Excel加BOM（`0xEF 0xBB 0xBF`） |
| 程序化解析BOM污染 | BOM被当作字段内容 | 程序化使用时省略BOM，或解析时跳过前3字节 |
| 欧洲Excel分隔错误 | 用逗号而非分号 | 检测系统区域，欧洲地区生成分号分隔 |
| 引号内换行被拆行 | 解析前按 `\n` 拆分 | 使用支持RFC 4180的解析器，不预先按行拆分 |
| 未转义引号破坏文件 | 字段中间出现裸引号 | 尽早校验，引号字段内必须 `""` 转义 |
| EOF尾换行产生空行 | 解析器创建空末行 | 剥离EOF尾换行，或在解析时跳过空行 |
| 列数不一致 | 某些行多/少字段 | 校验所有行匹配表头列数，报错或补齐 |
| 数字精度丢失 | Excel 15位精度限制 | >15位数字引号包裹并格式化为文本 |
| 公式注入风险 | 字段以 `=+-@` 开头 | 前缀加 `'` 或制表符，防止Excel执行 |
| 科学计数法误触发 | 数字含 `E` | 需字面文本时引号包裹 |
| 日期区域歧义 | `01/02/24` 多义 | 统一用ISO 8601（`2024-01-15`） |
| 前导零丢失 | 数字字段被解析为数值 | 引号包裹或文档化为字符串 |
| Latin-1与UTF-8歧义 | 编码未声明 | 解析前显式声明或检测编码 |

## 常见问题

### Q1：CSV必须用逗号分隔吗？
不必须。RFC 4180默认逗号，但欧洲Excel默认分号，TSV用 `\t`，遗留系统可能用 `|`。需根据目标工具与区域设置选择.
### Q2：Excel打开CSV中文乱码怎么办？
Excel Windows需要UTF-8 BOM（`0xEF 0xBB 0xBF`）才能正确识别UTF-8。在文件开头加BOM即可。但程序化使用时应省略BOM避免污染.
### Q3：字段内有双引号怎么处理？
用双引号包裹整个字段，内部的双引号转义为两个双引号 `""`。例如 `他说"你好"` 应写作 `"他说""你好"""`.
### Q4：如何防止Excel公式注入？
以 `=`、`+`、`-`、`@` 开头的字段会被Excel当作公式执行。在前缀加 `'`（单引号）或制表符，使其作为文本显示.
### Q5：长数字为什么在Excel里变了？
Excel对数字仅保留15位精度，超过15位的数字会丢失精度（如16位信用卡号 `4532015112830366` 变成 `4532015112830360`）。解决方法是引号包裹并格式化为文本.
### Q6：日期应该用什么格式？
ISO 8601（`2024-01-15`）是唯一无歧义格式。`01/02/24` 在美国是1月2日，在欧洲是2月1日，应避免.
### Q7：前导零如何保留？
数字字段的前导零（如工号 `007`）会被解析为数值7。用双引号包裹 `"007"` 或文档化为字符串字段.
### Q8：如何选择编码？
UTF-8是默认首选。为Windows Excel生成时加BOM；为程序化使用（pandas、数据库导入）时省略BOM。若数据源是Latin-1，解析前显式声明避免歧义.
## 已知限制

- RFC 4180是事实标准，但部分工具（如老版Excel）有非标行为，需针对性适配.
- 编码检测非100%可靠，Latin-1与UTF-8在低字节范围可能歧义.
- 公式注入防御依赖目标工具的行为，非所有工具都尊重 `'` 前缀.
- Excel 15位精度限制是工具特性，非CSV格式限制，需在生成时规避.
- 不支持加密CSV文件的破解.
- 依赖目标工具的解析行为，跨工具兼容性需实际验证.
