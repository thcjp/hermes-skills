# 博客写作助手

基于作者已发表作品校准语气、用词、结构与长度,产出直白、有观点、对话式且扎根实战经验的博客文章。集成笔记平台自动发布,维护不断增长的定稿示例库。

## 依赖说明

### 运行环境
- **Agent平台**: 支持SKILL.md的任意AI Agent（Claude Code / Cursor / Codex / Gemini CLI等）
- **操作系统**: Windows / macOS / Linux

### 依赖项
| 依赖项 | 类型 | 是否必需 | 获取方式 |
|:-------|:-----|:---------|:---------|
| LLM API | API | 必需 | 由Agent内置LLM提供 |

### API Key 配置
需要配置对应API Key，详见上文环境配置章节

### 可用性分类
- **分类**: MD+EXEC（）


**API Key配置方式**:
```bash
export API_KEY="your_api_key_here"
```
配置后需重启会话或开启新终端生效。API Key应妥善保管,避免泄露到版本控制系统。
## 核心能力

### 1. 作者风格捕捉
- 参考 `references/blog-examples/` 中的示例文章校准语气、用词、结构与长度
- 直白且有观点:清晰陈述立场,包括反主流观点
- 对话式:像与同事交流,平易但不简化
- 第一人称分享经验时自然使用"我"
- 真实的怀疑态度:必要时批评趋势

**输入**: 用户提供作者风格捕捉所需的指令和必要参数。
**处理**: 按照skill规范执行作者风格捕捉操作,遵循单一意图原则。
**输出**: 返回作者风格捕捉的执行结果,包含操作状态和输出数据。

### 2. 研究材料整合
- 审阅并整合用户提供的任何信息、研究材料或链接
- 自然融入研究材料,不作块引用堆砌
- 保留作者视角,材料服务于论点而非主导

**处理**: 按照skill规范执行研究材料整合操作,遵循单一意图原则。
**输出**: 返回研究材料整合的执行结果,包含操作状态和输出数据。

### 3. 用户指令遵循
- 严格遵循用户对主题、角度、重点的具体要求
- 默认目标长度 800-1500 字,可按用户要求调整
- 在开始写作前充分审阅所有提供的材料

**处理**: 按照skill规范执行用户指令遵循操作,遵循单一意图原则。
**输出**: 返回用户指令遵循的执行结果,包含操作状态和输出数据。

### 4. 真实写作产出
- 强势开场:以清晰的、常为大胆的陈述开篇
- 子标题贯穿:每 2-3 段使用 `###` 格式拆分内容
- 短段落:极少超过 3-4 句
- 个人轶事穿插:用真实例子说明观点
- 实用要点:提供可操作洞察,不止于理论
- 反思性结尾:以行动号召或前瞻性希望收尾

**处理**: 按照skill规范执行真实写作产出操作,遵循单一意图原则。
**输出**: 返回真实写作产出的执行结果,包含操作状态和输出数据。

### 5. 笔记平台发布
草稿完成后(即使尚未定稿)发布到笔记平台的 TS Notes 数据库:
- 数据库数据源 ID: `04a872be-8bed-4f43-a448-3dfeebc0df21`
- Type 属性设为 `Writing`
- Project(s) 属性链接到 My Writing 项目,页面 URL: `https://notes.example.com/2a5b4629bb3780189199f3c496980c0c`
- Note 属性填博客标题
- Content 填完整文章的笔记平台 Markdown 格式

**输入**: 用户提供笔记平台发布所需的指令和必要参数。
**处理**: 按照skill规范执行笔记平台发布操作,遵循单一意图原则。
**输出**: 返回笔记平台发布的执行结果,包含操作状态和输出数据。

### 6. 示例库维护
用户确认定稿后:
- 保存到 `references/blog-examples/`,文件名格式 `YYYY-MM-DD-slug-title.md`
- 示例: `2025-11-25-why-ai-art-is-useless.md`
- 超过 20 篇时,征求用户许可删除最旧的 5 篇(按文件名日期前缀排序)

### 7. 词汇标记校准
- 使用 "leverage" 指代工具/技术
- 使用 "that said" 做过渡
- 接受 "this is useless" 或 "boy was I wrong" 等直白表达
- 自然使用缩约形式(I've, doesn't, won't)
- 避免企业黑话但保持专业度

**输入**: 用户提供词汇标记校准所需的指令和必要参数。
**处理**: 按照skill规范执行词汇标记校准操作,遵循单一意图原则。
**输出**: 返回词汇标记校准的执行结果,包含操作状态和输出数据。

### 8. 主题元素对齐
- AI 作为工具而非替代
- 实践优于理论
- 以人为中心的技术
- 对有效与无效的诚实评估

**输入**: 用户提供主题元素对齐所需的指令和必要参数。
**处理**: 按照skill规范执行主题元素对齐操作,遵循单一意图原则。

## 适用场景

| 场景 | 输入 | 输出 |
|------|------|------|
| 风格化博客 | 主题+研究材料 | 800-1500字符合作者风格的文章 |
| 思想领导力内容 | AI/生产力/营销观点 | 有立场的深度文章 |

| 笔记平台发布 | 完成草稿 | TS Notes 数据库新条目 |

不适用于:纯技术文档撰写、学术写作、新闻稿。

## 使用流程

1. 收集主题、角度、研究材料、目标长度
2. 参考 `references/style-guide.md` 与示例校准风格
3. 起草内容,强势开场、短段、子标题、反思结尾
4. 提交草稿并迭代至用户满意
5. 发布到笔记平台 TS Notes 数据库(Type=Writing, Project=My Writing)
6. 用户确认定稿后保存到 `references/blog-examples/`

## 错误处理

| 错误场景 | 原因 | 处理方式 |
|---------|------|---------|
| 笔记平台发布失败,数据源ID无效 | API token 过期或权限不足 | 重新认证,确认数据源 `04a872be-8bed-4f43-a448-3dfeebc0df21` 可访问 |
| 文章长度超出 800-1500 字范围 | 研究材料过多或论点发散 | 按 2-3 段一个子标题拆分,删除与主旨无关的轶事 |
| 风格偏离作者语气 | 未充分校准示例库 | 重新阅读 `references/blog-examples/` 前 3 篇,对比词汇标记与段落长度 |
| 研究材料整合生硬成块引用 | 直接粘贴而非改写 | 用作者第一人称视角改写材料,融入个人经验 |
| 示例库超过 20 篇未清理 | 缺少归档流程 | 按文件名日期前缀排序,征求用户许可删除最旧 5 篇 |
| 笔记平台 Project 链接错误 | 项目页面 ID 不匹配 | 确认 Project(s) 值为数组格式,包含完整页面 URL |
| 子标题密度不足,段落过长 | 未遵循每 2-3 段一个 `###` | 在长段落间插入动作导向的子标题,如 "Deploy to Production" |

## 常见问题

### Q1: 如何校准作者风格?
A: 阅读 `references/blog-examples/` 中的已发表作品,记录词汇标记(leverage、that said、缩约形式)、段落长度(2-4句)、子标题密度(每2-3段)、开场方式(大胆陈述)、结尾方式(行动号召)。

### Q2: 笔记平台发布是必须的吗?
A: 是。草稿完成(即使未定稿)必须发布到 TS Notes 数据库,Type=Writing,Project=My Writing。未发布视为失败。数据库数据源 ID 为 `04a872be-8bed-4f43-a448-3dfeebc0df21`。

### Q3: 示例库达到 20 篇怎么办?
A: 按文件名日期前缀排序识别最旧文件,征求用户许可删除最旧的 5 篇。不要自动删除,必须获得明确授权。

### Q4: 目标长度 800-1500 字如何把握?
A: 默认范围 800-1500 字。短文聚焦单一观点配 1-2 个轶事;长文覆盖多角度但每段保持 2-4 句。用户可显式指定长度,如 "写 2000 字深度文章"。

### Q5: 研究材料如何自然融入?
A: 不作块引用堆砌。用作者第一人称视角改写材料核心观点,融入个人经验作为佐证。材料服务于论点,而非主导结构。

### Q6: 定稿归档的文件名格式是什么?
A: `YYYY-MM-DD-slug-title.md`,日期为定稿日期,slug 为 URL 友好的标题缩写。示例: `2025-11-25-why-ai-art-is-useless.md`。
