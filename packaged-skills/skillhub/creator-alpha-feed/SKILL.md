# Creator Alpha Feed 创作者 AI 日报

## 概述

Creator Alpha Feed 是一个为 AI 内容创作者设计的内容采集与排名 Skill,核心目标是从 X(原 Twitter)平台每日采集 AI 领域的高质量内容,按创作者友好的三档结构(KOL / 教程 / 行业)进行排名,既推送精简版到群频道供快速浏览,又写入完整报告到 Obsidian Vault 供后续二次创作与归档。

该 Skill 通过 `${OBSIDIAN_CONFIG_PATH}` 读取采集配置,按"主页 feed → 白名单账号 → 关键词"的顺序执行采集,严格管控浏览器标签页(最多 7 个并发,结束时归零),并对 `@xiaohu @dotey @marclou` 三个必追账号做状态校验,确保关键 KOL 内容不遗漏。

## 依赖说明

### 运行环境
- **Agent平台**: 支持SKILL.md的任意AI Agent（Claude Code / Cursor / Codex / Gemini CLI等）
- **操作系统**: Windows / macOS / Linux

### 依赖项
| 依赖项 | 类型 | 是否必需 | 获取方式 |
|:-------|:-----|:---------|:---------|
| LLM API | API | 必需 | 由Agent内置LLM提供 |

### API Key 配置
需要配置对应API Key，详见上文环境配置章节

### 可用性分类
- **分类**: MD（纯Markdown指令，无需exec命令行能力）

## 核心能力

### 1. 三阶段顺序采集

按固定顺序执行采集,确保内容覆盖面与优先级:

- **阶段一 - X 主页 feed**:拉取登录用户的主页时间线,过滤出 AI 相关内容(通过关键词与账号白名单双匹配)。
- **阶段二 - 白名单账号**:逐个访问配置中的白名单账号主页,采集其最近 6 小时内的全部推文。
- **阶段三 - 关键词搜索**:对配置的关键词列表执行 X 搜索,补充前两阶段未覆盖的内容。

### 2. 浏览器标签页上限管控

严格执行浏览器并发管控,避免资源耗尽:

- 最多 7 个并发标签页
- 每个标签页完成采集后立即关闭,再开新标签页
- 任务结束时确保标签页数为 0(关闭全部临时标签页)
- 若遇登录拦截,暂停当前标签页并通知用户,等待最多 3 分钟,期间周期性检查登录状态

### 3. 三档结构化排名

采集到的内容按以下三档结构排名输出:

- **KOL TOP3(近 6 小时)**:从白名单 KOL 的推文中选出互动量最高的 3 条,标注作者、发布时间、点赞数、转发数。
- **实用/教程/观点 TOP10**:从全部采集中筛选实用教程类、操作指南类、深度观点类内容,按内容质量与互动综合排名取前 10。
- **行业 TOP3(近 6 小时)**:从行业媒体账号与关键词搜索结果中选出前 3 条行业动态。

### 4. Obsidian Vault 双通道输出
- **群频道精简版**:推送三档排名的标题与链接到群频道,支持 `@提问者` 提及(若频道支持 mention)。
- **Obsidian 完整报告**:以 `YYYY-MM-DD_HHMM.md` 命名写入 Vault 的 `Skill平台/项目/AI内容日报/01-日报/` 目录,包含每条内容的原文摘要、作者主页链接、采集时间戳、排名理由。

**输入**: 用户提供Obsidian Vault 双通道输出所需的指令和必要参数。
**处理**: 按照skill规范执行Obsidian Vault 双通道输出操作,遵循单一意图原则。
**输出**: 返回Obsidian Vault 双通道输出的执行结果,包含操作状态和输出数据。

### 5. 必追账号状态校验
强制校验 `@xiaohu @dotey @marclou` 三个账号的采集状态,若任一账号在本次采集中未抓取到内容(可能是账号停更、登录失效、或被 X 限流),在报告与群消息中明确标注"账号 X 本次未采集到内容,请检查",避免静默遗漏。

**输入**: 用户提供必追账号状态校验所需的指令和必要参数。
**输出**: 返回必追账号状态校验的执行结果,包含操作状态和输出数据。

### 6. Obsidian 目录结构化
使用结构化的 Obsidian 目录组织产出:

- `Skill平台/项目/AI内容日报/01-日报/`:最终日报报告
- `Skill平台/项目/AI内容日报/02-运行记录/`:验证与调试运行记录
- `Skill平台/项目/AI内容日报/03-文档/`:安装与操作文档

优先使用真实 Obsidian Vault 路径(非 workspace 镜像),确保报告可被 Obsidian 直接索引。

**输入**: 用户提供Obsidian 目录结构化所需的指令和必要参数。
**输出**: 返回Obsidian 目录结构化的执行结果,包含操作状态和输出数据。

### 7. 降级与回退机制

- API 可用时优先走 API 采集(速度快、结构化好)
- API 不可用时回退到浏览器采集(覆盖面广、抗限流)
- 浏览器采集失败时记录降级笔记,在报告中标注哪些源采用了回退策略
- 提供 `scripts/collect-v4.sh` 脚本作为确定性回退,在 Agent 不可用时仍可手动执行采集

## 适用场景

### 场景一：AI 自媒体每日选题

- **输入**: 创作者配置白名单 30 个 AI 领域 KOL,关键词 15 个(如 LLM、Agent、RAG、MCP)
- **输出**: 群频道推送三档排名精简版,Obsidian 日报写入 `01-日报/2026-07-20_0900.md`,含 KOL TOP3、教程 TOP10、行业 TOP3
- **价值**: 创作者每日早晨打开 Obsidian 即可获得当日选题素材,无需手动刷 X

### 场景二：内容策展周报归档

- **输入**: 每日采集的报告已写入 Obsidian 01-日报目录
- **输出**: 周末可基于 7 份日报汇总生成周报,引用每日报告的 frontmatter 与排名
- **价值**: 周报素材有据可查,引用链路完整

### 场景三：行业情报订阅

- **输入**: 配置行业类关键词(如模型发布、融资、政策)与行业媒体账号
- **输出**: 行业 TOP3 档位聚焦行业动态,群频道推送精简版供团队快速了解
- **价值**: 团队共享行业情报,无需每人单独订阅

### 场景四：多账号登录场景

- **输入**: 部分白名单账号需要登录才能查看完整推文
- **输出**: 采集时暂停并通知用户"账号 X 需登录,请在浏览器完成登录",等待最多 3 分钟,期间每 30 秒检查一次登录状态,登录后继续采集剩余源
- **价值**: 不会因单个账号登录问题中断整次采集

## 使用流程

### 步骤一：配置采集参数

编辑 Obsidian Vault 中的采集配置文件(路径通过 `${OBSIDIAN_CONFIG_PATH}` 环境变量指定,默认 `<vault>/Skill平台/项目/AI内容日报/采集配置.md`),配置以下字段:

- 白名单账号列表(必须包含 `@xiaohu @dotey @marclou`)
- 关键词列表
- 群频道 webhook 或 chat_id
- 是否启用 API 优先策略

### 步骤二：确认 Obsidian 目录结构

检查 Vault 中存在以下目录,若缺失则创建:

```
<vault>/Skill平台/项目/AI内容日报/01-日报/
<vault>/Skill平台/项目/AI内容日报/02-运行记录/
<vault>/Skill平台/项目/AI内容日报/03-文档/
```

### 步骤三：执行采集

向 Agent 发送触发指令(如"采集今日 AI 内容日报"),Skill 将按"主页 feed → 白名单账号 → 关键词"顺序执行采集,浏览器标签页严格管控在 7 个以内。

### 步骤四：处理登录拦截

若采集过程中遇到 X 登录拦截,Skill 会暂停并通知用户。用户在浏览器完成登录后,Skill 周期性检查(每 30 秒一次,最多 3 分钟)登录状态,恢复后继续采集剩余源;若 3 分钟超时仍未登录,跳过该源并记录降级笔记。

### 步骤五：接收群频道推送

采集完成后,Skill 将三档排名的精简版推送到配置的群频道,若频道支持 mention 会 `@提问者`。

### 步骤六：查阅 Obsidian 完整报告

完整报告以 `YYYY-MM-DD_HHMM.md` 命名写入 `01-日报/` 目录,在 Obsidian 中打开即可查阅每条内容的原文摘要、作者、链接、排名理由。

### 步骤七：校验必追账号状态

在报告末尾检查 `@xiaohu @dotey @marclou` 三个账号的采集状态,若有未采集到的账号,按报告提示排查(账号停更、登录失效、限流)。

**效果**: 用户第一时间发现 @marclou 采集异常,手动排查确认是限流,次日恢复后正常采集,避免静默遗漏关键 KOL 内容。

## 异常处理

### 1. OBSIDIAN_CONFIG_PATH 未配置且默认路径不存在

**原因**: 环境变量 `OBSIDIAN_CONFIG_PATH` 未设置,且默认 Vault 路径 `<vault>/Skill平台/项目/AI内容日报/采集配置.md` 不存在。

**处理**: Skill 启动时先尝试读取环境变量,失败后回退到默认路径,若默认路径也不存在则报错并提示用户:"请设置 OBSIDIAN_CONFIG_PATH 环境变量指向采集配置.md,或在默认 Vault 路径创建该文件"。建议在 `03-文档/` 目录提供配置模板。

### 2. 浏览器标签页超过 7 个上限

**原因**: 采集逻辑未及时关闭已完成任务的标签页,或多个采集源并行打开新标签。

**处理**: Skill 在每次打开新标签页前检查当前并发数,若已达 7 则等待最早标签页完成并关闭后再开新;若发现已超上限(异常情况),立即关闭最近打开的标签页并记录告警到 `02-运行记录/`。任务结束前强制关闭全部临时标签页,确保归零。

### 3. X 登录拦截且 3 分钟超时

**原因**: 用户未在 3 分钟内完成浏览器登录,或登录后 cookie 未生效。

**处理**: 超时后跳过该登录受限源,继续采集剩余源;在报告与群消息中标注"源 X 因登录超时未采集,已降级";记录到 `02-运行记录/` 供后续排查。不建议因单源失败中断整次采集。

### 4. 必追账号采集为空

**原因**: `@xiaohu @dotey @marclou` 任一账号本次采集返回 0 条,可能是账号停更、被限流、或登录态失效。

**处理**: 不静默跳过,在报告末尾的"必追账号校验"区块明确标注每个账号的采集条数与状态(正常/异常);群频道推送也包含该警示;建议用户手动访问异常账号主页确认是停更还是限流,必要时次日重试。

### 5. Obsidian Vault 路径不可写

**原因**: Vault 路径权限不足、磁盘已满、或路径被其他进程锁定。

**处理**: 写入报告前先尝试创建测试文件验证可写性;若不可写,回退将报告写入 workspace 镜像目录并在群频道推送报告全文;在 `02-运行记录/` 记录路径异常,提示用户检查 Vault 路径权限。

### 6. API 采集返回速率限制(429)

**原因**: X API 调用频率超出限额,或 token 过期。

**处理**: 自动回退到浏览器采集该源;记录降级笔记"源 X 因 API 429 回退浏览器采集";若浏览器也受限,等待 60 秒后重试一次,仍失败则跳过并标注。建议在配置中开启 API 与浏览器双通道,避免单点失败。

### 7. 群频道 webhook 失效

**原因**: webhook URL 过期、被管理员撤销、或群频道已解散。

**处理**: 推送失败时不中断采集流程,Obsidian 报告仍正常写入;在 `02-运行记录/` 记录推送失败详情;提示用户在采集配置中更新 webhook URL,或改用 chat_id + token 推送方式。

### 8. 采集内容去重后为空

**原因**: 全部采集源均无返回(可能是 X 大规模故障、网络中断、或全部源登录失效)。

**处理**: 不生成空报告,在群频道推送"今日采集失败:全部源无内容返回,请检查网络与登录态";在 `02-运行记录/` 记录每个源的失败原因;建议用户手动访问 X 确认是否为平台故障。

## 常见问题

### Q1: 如何新增白名单 KOL 账号?

编辑采集配置文件 `${OBSIDIAN_CONFIG_PATH}` 指向的 `采集配置.md`,在 `whitelist_accounts` 列表中追加账号 handle(如 `@new_kol`)。下次采集自动纳入。注意 `@xiaohu @dotey @marclou` 为系统强制必追,不可从配置中移除。

### Q2: 报告文件名为什么用 YYYY-MM-DD_HHMM 格式?

该格式兼顾可读性与 Obsidian 索引友好性:日期在前便于按日聚合,HHMM 精确到分钟避免同日多次采集文件名冲突,下划线分隔避免冒号在文件系统中的兼容问题。若同分钟内多次采集,文件名追加 `_2`、`_3` 后缀。

### Q3: 浏览器标签页上限可以调高到 10 个吗?

不建议。7 个并发是基于浏览器内存占用与 X 反爬策略的平衡值,超过可能导致浏览器卡顿或被 X 识别为异常行为限流。若确需提高,在 `scripts/collect-v4.sh` 中修改 `MAX_TABS` 变量,但需自行承担限流风险。

### Q4: 采集到的内容如何判断"实用/教程/观点"分类?

Skill 基于推文内容的关键词与结构启发式分类:含"教程/指南/step by step/如何"等词归教程类,含"实战/落地/案例/复盘"归实用类,含"观点/思考/趋势/预测"归观点类。分类不精确的内容默认归入实用类,在报告中标注"分类待复核"。

### Q5: 群频道推送可以自定义消息模板吗?

可以。在采集配置文件的 `group_message_template` 字段定义模板,支持变量 `kol_top3`、`tutorial_top10`、`industry_top3`、`asker_mention`(用双花括号包裹)。若不配置,使用默认精简模板(三档标题 + 链接 + 提问者 mention)。

### Q6: scripts/collect-v4.sh 在什么场景下使用?

当 Agent 不可用(如 skill-platform 服务停止)或需要确定性回退(如 Agent 采集结果异常需对照)时,手动执行 `scripts/collect-v4.sh` 完成基础采集。该脚本不依赖 Agent,但功能是 Skill 的子集(无智能分类与排名,仅采集与去重)。

## 错误处理


| 错误场景 | 原因 | 处理方式 |

## 已知限制

- 仅支持 X(原 Twitter)平台采集,不支持微信公众号、即刻、微博、Reddit 等其他内容源
- 三档排名基于启发式规则与互动量,非语义级内容质量评估,可能遗漏低互动高质量内容
- 浏览器采集依赖 X 的 DOM 结构,若 X 改版可能导致采集失败,需更新采集脚本
- 必追账号固定为 `@xiaohu @dotey @marclou` 三个,不可通过配置移除(可追加其他必追账号)
- 群频道推送的 mention 能力取决于目标频道协议,部分频道(如 Slack)可能不支持 `@提问者`
- 采集频率建议每日 1-2 次,高频采集可能触发 X 限流
- Obsidian 报告为 Markdown 格式,不直接生成 PDF 或图片摘要
