## 一、Critical 级别问题（4个）

### M-C1. 测试覆盖率为零——全项目无任何测试文件

**关联前两轮**：独立发现，但放大了第1轮所有问题的影响（无测试兜底，所有bug只能靠生产发现）

**问题证据**：
- 全项目16个Python文件、约5000+行代码，Glob搜索 `**/*test*` 仅返回1个JSON数据文件，**零测试代码**
- 5个被审文件无一有任何单元测试
- 无 pytest.ini / conftest.py / tests/ 目录
- 无 mock / fixture / 测试数据库准备脚本

**影响分析**：
- 第2轮发现的7个Critical（字段错位、双写无事务、硬编码路径、tools检查逻辑错误、SQL注入风险、无限循环风险、硬编码用户路径）**全部只能靠人工审查发现，无法被自动化测试拦截**
- scores表字段错位（第2轮C1）如果存在单元测试 `test_save_trace_score_writes_correct_fields`，在第一次运行时就会失败暴露
- 定价双写无事务（第2轮C2）如果存在 `test_apply_pricing_is_atomic` 就能拦截
- **这是整个方法论体系最大的漏洞**：没有测试，前两轮发现的所有问题修复后也无法验证是否真正修复、是否会回归

**TDD技能明确指出**：
> "NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST"
> "If you didn't watch the test fail, you don't know if it tests the right thing."

当前代码100%违反此铁律。

**改进建议**：
```python
# 优先为以下关键路径补测试（按ROI排序）
# 1. trace_llm_scorer.save_trace_score —— 字段映射正确性
# 2. pricing_engine.calculate_price —— 定价计算逻辑
# 3. upload_gate.run_gate_check —— 门控决策正确性
# 4. skill_batch_upgrader_v3.auto_fix —— 修复不破坏原内容
# 5. market_monitor.normalize_category —— 类别映射一致性

# 示例：最该先写的测试
def test_save_trace_score_maps_fields_correctly():
    """验证TRACE五维度写入正确的数据库字段"""
    save_trace_score(skill_id=1, checks={...}, static_scores={...}, llm_result={
        'trace_scores': {
            'trust': {'score': 8},
            'reliability': {'score': 7},
            'adaptability': {'score': 6},
            'convention': {'score': 9},
            'effectiveness': {'score': 8},
        },
        'total_score': 38,
        'quality_grade': 'B'
    })
    # 读取并验证每个字段语义正确
    row = db.execute("SELECT * FROM scores WHERE skill_id=1").fetchone()
    assert row['trust_score'] == 8        # 而非 debranding_score
    assert row['reliability_score'] == 7  # 而非 quality_score
```

**优先级**：P0（最高）
**实施路径**：
1. 立即创建 `tests/` 目录和 `conftest.py`（测试数据库fixture）
2. 为第2轮7个Critical问题各写1个回归测试（7个测试）
3. 为5个被审文件的核心函数写测试，目标覆盖率60%
4. 预计工时：3-5天

### M-C4. 批量操作无事务日志和错误恢复——中途失败留撕裂状态

**关联前两轮**：
- 第1轮 High#5"批量修复无回滚"
- 第2轮 C2"定价双写无事务实锤"
- 架构审查报告原文："upgrader_v3.auto_fix 在 1000 个 skill 上跑，第 500 个抛异常会留下 500 个已改 + 500 个未改的撕裂状态"

**问题证据**：

`skill_batch_upgrader_v3.py` `cmd_fix()`（第1046-1093行）：
```python
for slug, local_path in skills:
    skill_md = Path(local_path) / "SKILL.md"
    fixes = auto_fix(skill_md)  # 任一次抛异常，循环中断
    # 无 try/except，无状态记录，无续跑机制
```

`pricing_engine.py` `cmd_apply()` + `cmd_update_db()`：
```python
# 第一步：循环覆写1858个SKILL.md文件
for result in all_results:
    skill_md.write_text(new_content)  # 第N个失败则前N-1个已改
# 第二步：循环UPDATE数据库
for result in all_results:
    c.execute("UPDATE skills SET ...")  # 整批无事务包裹
conn.commit()  # 只在最后commit一次
```

`trace_llm_scorer.py` `cmd_static()`：
```python
for i, skill in enumerate(skills):
    result = save_trace_score(skill['id'], ...)  # 每次独立commit
    # 虽然单次操作原子，但批量无整体事务
```

**影响分析**：
- `batch_upgrade_report.json` 记录了1858个skill全部"upgraded"，但这是**事后记录**，不是**事前计划+事中检查点**
- 若 `cmd_apply` 在覆写到第900个SKILL.md时磁盘满/权限错误，前899个已改+后959个未改，且DB未更新=完全撕裂
- **赚钱威胁**：撕裂状态下，部分skill定价已更新部分未更新，部分合规已修复部分未修复，导致**商品列表不一致**，用户购买体验断裂

**改进建议**：
```python
# 1. 批量操作必须使用事务日志模式
def cmd_fix_with_transaction_log(skills):
    tx_log = TransactionLog(r"d:\skills\.tx\fix_%s.json" % datetime.now().strftime("%Y%m%d_%H%M%S"))
    
    for slug, local_path in skills:
        tx_log.begin(slug)
        try:
            fixes = auto_fix(skill_md)
            tx_log.commit(slug, fixes)
        except Exception as e:
            tx_log.fail(slug, str(e))
            # 继续下一个，不中断整个批次
    
    tx_log.summary()

# 2. 数据库批量操作使用显式事务
def cmd_update_db():
    conn = sqlite3.connect(DB_PATH)
    try:
        conn.execute("BEGIN TRANSACTION")
        for result in all_results:
            conn.execute("UPDATE skills SET ... WHERE slug=?", (...))
        conn.commit()
    except:
        conn.rollback()
        raise

# 3. 提供续跑命令
def cmd_resume(tx_log_path):
    """从中断的事务日志续跑"""
    tx_log = TransactionLog(tx_log_path)
    pending = tx_log.get_pending()
    cmd_fix_with_transaction_log(pending)
```

**优先级**：P0
**实施路径**：
1. 为所有批量操作加 try/except + 事务日志（2天）
2. 数据库操作包裹显式事务（半天）
3. 实现 resume 命令（1天）

### M-H1. 完全未实践TDD——代码先行，测试缺失

**关联前两轮**：独立发现，但这是第2轮所有Critical问题的根因

**问题证据**：
- 5个被审文件共计3534行代码，无一行测试代码
- 无任何 `test_` 前缀函数或类
- 无 `if __name__ == '__main__': unittest.main()` 痕迹
- 代码中的复杂业务逻辑（定价矩阵、TRACE评分映射、合规检查规则）从未经过测试验证

**TDD技能明确指出**：
> "Tests written after code pass immediately. Passing immediately proves nothing"
> "Tests-after answer 'What does this do?' Tests-first answer 'What should it do?'"

第2轮发现的"scores表字段错位"正是典型的 tests-after 盲区：写代码时按错误映射写入，如果有测试先行，写测试的瞬间就会发现"我该断言哪个字段存reliability？"

**改进建议**：
- 对所有新功能严格执行 RED-GREEN-REFACTOR
- 对现有代码补测试时，先写一个**期望正确行为**的测试（如字段映射），观看它失败（暴露bug），再修复代码使其通过
- 具体优先补测试的函数：
  1. `trace_llm_scorer.save_trace_score`（字段映射）
  2. `pricing_engine.calculate_price`（定价边界）
  3. `upload_gate.check_correctness`（tools检查逻辑——第2轮C4）
  4. `skill_batch_upgrader_v3.fix_reserved_words`（正则正确性）

**优先级**：P1
**实施路径**：与M-C1合并执行，先补关键路径测试

### M-H5. 无端到端验证机制——upload_gate是手动工具非自动门控

**关联前两轮**：第1轮"职责重叠"——upload_gate与upgrader_v3检查项重叠

**问题证据**：
- `upload_gate.py` 定义了 `cmd_check_all` 和 `gate` 命令
- 但无任何自动化流程调用它
- 无"修复后必须通过gate才能提交"的强制规则
- `skill_batch_upgrader_v3.py` 修复后不调用 `upload_gate` 验证修复结果
- `pricing_engine.py` 定价后不调用 `upload_gate` 验证定价合理性

**影响分析**：
- 修复可能引入新问题（如 `fix_tools_format` 把正确的改错——第2轮C4）
- 但修复后无自动验证，问题只能在下一次手动运行gate时发现
- 违反"验证先于完成"原则（superpowers:verification-before-completion）

**改进建议**：
```python
# skill_batch_upgrader_v3.py 的 cmd_fix 末尾自动验证
def cmd_fix(args):
    # ...修复逻辑...
    
    # 自动验证：修复后必须通过门控
    print("\n验证修复结果...")
    gate_results = []
    for slug, _ in skills:
        result = run_gate_check(skill_dir)  # 调用upload_gate
        gate_results.append(result)
    
    still_failing = [r for r in gate_results if not r['passed']]
    if still_failing:
        print(f"警告: {len(still_failing)} 个skill修复后仍未通过门控")
```

**优先级**：P1
**实施路径**：1天集成

## 八、结论

当前项目的开发方法论存在**系统性缺陷**：无测试、无CI、无回滚、无事务日志——这四个Critical缺陷构成了"缺陷直通生产"的高速通道。

对于"赚钱"目标而言，当前方法论的最大威胁不是"功能不够"，而是**已有的功能不可信**：
- 评分不可信（字段错位）
- 定价不可信（双写撕裂）
- 合规不可信（tools检查逻辑错误）
- 库存不安全（无回滚覆写）

**建议优先级**：先止血（P0的4项），再巩固（P1的7项），后优化（P2/P3）。两周内可将方法论成熟度从1.8/10提升至6/10，使赚钱目标具备可持续的工程基础。
