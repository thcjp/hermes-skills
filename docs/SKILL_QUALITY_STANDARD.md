## 一、TRACE评测体系对齐

### 1.1 五大维度定义

| 维度 | 含义 | 核心问题 | 评分标准 |
|------|------|---------|---------|
| **T - Trust** 可信任度 | 安全检测、国内可用、中文支持 | 能不能放心用 | 安全扫描通过+国内网络可跑+中文交互完整 |
| **R - Reliability** 可靠性 | 运行稳定、异常处理、边界输入 | 能不能稳定用 | 常见/边界/异常输入一致表现+失败有可理解反馈 |
| **A - Adaptability** 适用性 | 触发精准、边界清晰、不误触发 | 该不该在此场景用 | description精准+适用输入/输出/边界明确 |
| **C - Convention** 规范性 | 信息架构、文档、可维护性 | 能不能被理解和复用 | 清晰信息架构+充分不过载文档+已知限制说明 |
| **E - Effectiveness** 有效性 | 结果正确、输出完整、额外价值 | 问题是否真被解决 | 输出可直接使用+超出用户原本能力的价值 |

### 1.2 与本地评分的映射

| TRACE维度 | 本地scores表维度 | 评分方式 | 权重 |
|-----------|-----------------|---------|------|
| Trust | security + debranding | 静态检查+LLM审查 | 20% |
| Reliability | error_handling + robustness | LLM评估异常处理 | 20% |
| Adaptability | trigger_accuracy + boundary | LLM评估description+触发测试 | 20% |
| Convention | structure + readability | 静态检查+LLM评估文档质量 | 20% |
| Effectiveness | practicality + value_added | LLM评估输出价值 | 20% |

## 三、质量标准 (必须满足)

### 3.1 Frontmatter标准 (L1层 - 常驻上下文)

```yaml
---
name: kebab-case-slug          # 必需: 小写连字符
displayName: 中文名称           # 必需: ≤20字符，中文
version: "1.0.0"               # 必需: 语义版本号
summary: 精确描述触发场景        # 必需: ≤100字符，含"当...时使用"
description: |-                 # 必需: 50-300字符
  核心功能描述。
  适用场景说明。
  Use when 用户表达某需求时。
license: Proprietary            # 必需: 付费版用Proprietary
tools:                          # 必需: YAML数组格式
  - read
  - exec
---
```

**description写作公式**:
```
[核心功能一句话]。适用于[具体场景]，[输入类型]→[输出类型]。
Use when [触发条件1]、[触发条件2]。
不适用于[边界排除]。
```

**description质量检查**:
- [ ] 50-300字符（太短不够信息，太长浪费token）
- [ ] 包含"Use when"触发条件
- [ ] 包含"不适用于"边界排除
- [ ] 提到具体输入输出类型
- [ ] 不含"帮助用户处理各种任务"等宽泛表述

### 3.3 正文长度标准

| 类型 | 推荐行数 | 上限 | 说明 |
|------|---------|------|------|
| 简单工具类 | 80-150行 | 200行 | Tool Wrapper模式 |
| 创作生成类 | 150-250行 | 300行 | Generator模式 |
| 审查评估类 | 150-250行 | 300行 | Reviewer模式 |
| 多步流程类 | 200-350行 | 400行 | Pipeline模式 |

**超过上限的**: 将详细参考资料拆分到references/目录

### 3.4 五种设计模式选择

| 用户需求 | 选择模式 | 核心结构 |
|---------|---------|---------|
| 遵循某库/框架规范 | Tool Wrapper | references/放规范 |
| 输出固定格式 | Generator | assets/放模板+references/放风格 |
| 代码/内容审查 | Reviewer | references/放检查清单 |
| 先收集需求再执行 | Inversion | 多轮问答+assets/放输出模板 |
| 多步骤依赖流程 | Pipeline | 步骤间gate条件+三目录齐全 |

## 七、改进的升级方法论（v2.1）

### 7.1 升级流程改进

```
Phase 1: 机械重构 (Python脚本, 不需要LLM)
  → 章节重命名: "快速开始"→"使用流程", "故障排查"→"错误处理"等
  → description优化: 超过300字符自动精简到150-280字符
  → 内容提取: 从现有章节提取内容填充缺失章节
  → 表格格式化: 错误处理统一为表格格式

Phase 2: LLM增强 (子代理, 按批次并行)
  → 每批5个skill, 每个子代理负责一批
  → 读取机械重构后的SKILL.md
  → 重写description (遵循公式: [核心功能]+[适用场景]+[触发关键词])
  → 补充核心能力 (3-5条具体能力, 非通用描述)
  → 补充适用场景 (含输入输出+不适用场景)
  → 补充使用流程 (Step by step)
  → 补充示例 (至少2个输入→输出)
  → 补充错误处理 (场景|原因|处理方式表格)
  → 补充依赖说明 (LLM/API Key/运行环境表格)
  → 补充常见问题 (3-5个Q&A)
  → 补充已知限制 (3-5条具体限制)

Phase 3: LLM TRACE评估
  → 五维度评分 (T/R/A/C/E各0-10)
  → 识别残留问题
  → C级及以下skill返回Phase 2修复

Phase 4: 上传更新
  → 生成free+paid双版本
  → 上传到SkillHub
  → 记录到platform_uploads表
```

## 九、营销优化策略（v3.0新增）

> **来源**: Apple ASO方法论 + GPT Store SEO实践 + Coze商店发现机制 + Claude Skills触发优化

### 9.1 标题/名称SEO优化

| 规则 | 说明 | 示例 |
|------|------|------|
| 标题≤30字符 | 避免搜索结果截断 | "AI写作分身工厂"(8字符) |
| 品牌名+核心功能词 | 格式: 品牌名+功能 | "爆款预言机-内容预测" |
| 含核心关键词 | 用户搜索的高频词 | 含"文案""SEO""数据分析" |
| 禁占位词 | 不用"一款""工具" | 直接写功能 |
| 禁夸大词 | 不用"万能""最强" | 用具体功能名 |

### 9.2 description触发词优化

**公式**: `[做什么] + [何时使用] + [关键功能] + [具体触发短语]`

**触发词设计原则**:
- 包含用户可能说的**原话短语**（如"帮我写文案""分析竞品"）
- 包含**技术术语**（如"SEO""LTV""ROI"）便于精准触发
- 包含**负面触发条件**（如"不适用于实时协作"）避免误触发
- description前2行最重要（搜索权重最高+转化率最高）

**description质量检查清单**:
- [ ] 150-280字符（太短信息不足, 太长浪费token）
- [ ] 包含3+个触发短语
- [ ] 包含"不适用于"边界排除
- [ ] 包含具体输入输出类型
- [ ] 前2行含核心功能+用户价值
- [ ] 不含"帮助用户处理各种任务"等宽泛表述
- [ ] 不含"万能""超级""最强"等夸大词

### 9.3 定价策略最佳实践

#### 9.3.1 三级定价梯度

| 层级 | 价格区间 | 适用类型 | 策略 |
|------|---------|---------|------|
| 引流款 | 免费 | 高频工具类 | 免费引流,积累评分和下载量 |
| 按次付费 | 9.9-29.9/次 | 中频创作类 | 单次价值明确,用户决策成本低 |
| 月订阅 | 9.9-29.9/月 | 持续使用类 | 高频用户转化,稳定收入 |
| 买断 | 19.9-49.9 | 一次性高价值 | 低频高价值工具 |

#### 9.3.2 Freemium模式要点

- 免费版开放**基础功能**吸引流量
- 付费版解锁**高级功能/不限量/高清导出**
- 免费版作为**作品集和信用背书**
- 付费转化率参考: 5%左右
- 定价覆盖API调用成本(token费/GPU成本)

### 9.4 分类选择策略

| 策略 | 说明 | 示例 |
|------|------|------|
| 避开通用红海 | 通用写作/编程/问答已饱和 | 不选"通用文案"选"电商详情页文案" |

| 跨分类引流 | 一个skill覆盖多个分类 | "SEO文案"同时覆盖SEO和文案 |
