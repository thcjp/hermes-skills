## 核心策略

### 策略一：29个P0 + 185个P1 全量LLM再生成（214个skill，428个文件）

**方法**:
1. 从 `sf_batch_results_v3.json` 提取所有SF<70的skill列表
2. 按SF分数排序（最低的先处理）
3. 每批20个skill（40个文件），派发3组并行子agent
4. 使用Round 13验证有效的子Agent v4指令
5. 每批完成后运行L1+L2+SF三层验证
6. SF<70的skill必须重新生成

**预估**: 214个skill / 20个每批 = 11批 × 3组并行 = 约4轮并行
**每轮并行约30分钟** → 总计约120分钟

### 策略二：197个源能力点空skill改进

**方法**:
1. 分析源skill结构，找出能力点提取失败的原因
2. 改进 `extract_capability_points()` 的章节匹配逻辑
3. 增加更多源skill结构变体的支持
4. 对仍无法提取的，使用替代验证（L2-Capability≥85）

### 策略三：244个无源skill替代质量标准

**方法**:
1. 分类：原创skill vs 源skill丢失 vs slug不匹配
2. 对slug不匹配的：修正slug后重新搜索源skill
3. 对源skill丢失的：使用L2-Capability≥85作为标准
4. 对原创skill：使用L2-Capability≥85 + 核心能力≥3个 + 错误处理≥5个

### 策略四：L3功能验证机制建立与试点

**方法**:
1. 创建 `l3_function_checker.py`
2. 验证API/命令存在性（反引号包裹的命令在源skill中也存在）
3. 验证示例代码语法（Python/Bash/SQL语法检查）
4. 验证参数完整性（参数在源skill中存在）
5. 对100个skill进行L3试点验证

### 策略五：持续质量保障机制

**方法**:
1. 新skill入库质量门禁：L1+L2+SF三层验证必须全部通过
2. 定期SF批量检查（每周一次）
3. 质量趋势监控（SF分数变化趋势）
4. 用户反馈收集机制

## 注意事项

1. **P0优先**: 29个模板垃圾必须最先修复
2. **并行最大化**: 每轮3组并行子agent，加速再生成
3. **三层验证不可省略**: 每批必须运行L1+L2+SF
4. **SF<70不接受**: 再生成的skill必须SF≥70
5. **术语保留是关键**: 跨语言术语匹配需要持续改进
6. **L3是终极标准**: 关键词匹配≠功能正确，L3功能验证是最终合格标准
7. **无源skill不放过**: 244个无源skill必须有替代质量标准
8. **诚实数据**: 所有质量判断基于SF批量检查结果，不信文档描述
9. **-free版本name字段**: name必须与slug和folder_name一致
10. **tools格式**: YAML数组格式（`  - read`），不要 `  - - read`
