# 工具评估工具（免费版）

## 概述

工具评估工具免费版帮助个人开发者从问题出发而非从工具出发，找到真正适合自己的工具。提供工具发现方法、评估框架、试用周期设计与工作流连接指引，帮助你避免在工具切换中浪费时间.
## 核心能力

| 能力 | 说明 |
|---|---|
| 工具发现 | 从问题定义出发的候选工具检索 |
| 评估框架 | 5 维度评估（适配度、学习成本、迁移成本、维护性、社区） |
| 试用周期 | 结构化试用流程设计 |
| 工作流设计 | 连接现有工具的自动化指引 |
| 综合征识别 | 工具切换综合征的识别与应对 |
**技术实现要点**：核心能力基于`input_params`参数与`output_format`配置实现,支持创建/查询/修改/删除等操作模式,通过`config_options`进行运行时配置.
### 核心功能执行
用`input_params`参数进行配置.
**输入**: 用户提供核心功能执行所需的指令和必要参数.
**处理**: 解析核心功能执行的输入参数,完成核心逻辑,返回结构化响应.
**输出**: 返回核心功能执行的响应数据,包含状态码、结果和日志.
- 执行此能力时使用`input_params`参数,支持创建/查询/导出操作

### 参数配置与调用
用`config_options`参数进行配置.
**输入**: 用户提供参数配置与调用所需的指令和必要参数.
**处理**: 解析参数配置与调用的输入参数,完成核心逻辑,返回结构化响应.
**输出**: 返回参数配置与调用的响应数据,包含状态码、结果和日志.
- 执行此能力时使用`config_options`参数,支持修改/重置/导入操作

### 结果处理与输出
用`output_format`参数进行配置.
**输入**: 用户提供结果处理与输出所需的指令和必要参数.
**处理**: 解析结果处理与输出的输入参数,完成核心逻辑,返回结构化响应.
**输出**: 返回结果处理与输出的响应数据,包含状态码、结果和日志.
- 执行此能力时使用`output_format`参数,支持导出/保存/转换操作
**能力覆盖范围**：本skill的核心能力覆盖以下场景关键词：提供工具发现、评估框架与工作流、设计指引、帮助个人开发者选、对工具、避免工具切换综合、工具评估工具免费、面向个人开发者的、轻量级工具选择与、从问题出发的工具、发现方法、工具评估框架与试、用周期设计、工作流连接设计指、工具切换综合征识等。这些关键词对应description中声明的使用场景,均已在上述能力点中提供对应的操作支持.
## 使用场景

### 场景一：新需求时的工具选型

开发者面临新需求，需要快速评估候选工具.
```text
用户：我需要管理一个 20 人团队的任务，有哪些工具值得考虑？
助手：
让我从你的具体问题出发，先明确几个关键约束：
# ...
1. 团队规模与协作模式（同步/异步）
2. 是否需要与现有代码仓库集成
3. 预算范围
4. 团队对新工具的接受度
# ...
基于这些约束，候选工具的评估矩阵：
# ...
| 工具     | 适配度 | 学习成本 | 迁移成本 | 维护性 | 社区活跃 |
|:-----|:-----|:-----|:-----|:-----|:-----|
| 工具A    | 高     | 低       | 低       | 高     | 高       |
| 工具B    | 中     | 中       | 中       | 中     | 中       |
| 工具C    | 高     | 高       | 高       | 高     | 低       |
# ...
建议优先试用工具A，试用周期建议 2 周.
```

### 场景二：避免工具切换综合征

开发者频繁更换工具，希望判断是否真的需要切换.
```text
用户：我又想换笔记工具了，这已经是今年的第三次了.
助手：
让我们先判断这是「工具真的不合适」还是「还没学会有效使用」：
# ...
判断清单：
1. 你是否完整阅读过当前工具的使用文档？
2. 你是否遇到过当前工具无法解决的具体问题？
3. 新工具是否能解决这些问题，且不引入新问题？
4. 切换的学习与迁移成本是否值得？
# ...
如果以上多数为「否」，问题可能不在工具，而在使用方式.
建议：先深度使用当前工具 2 周，记录具体痛点，再决定是否切换.
```

### 场景三：设计连接工具的工作流

开发者希望将多个工具串联为自动化工作流.
```text
用户：我想把代码仓库、任务管理和文档工具连起来.
助手：
工作流设计三步法：
# ...
1. 识别重复手动操作（每天/每周重复的步骤）
2. 寻找集成点（API、Webhook、Zapier 等）
3. 设计触发-动作结构
# ...
示例工作流：
代码提交 → 自动更新任务状态 → 同步进度至文档
# ...
建议从最高频的手动操作开始自动化，避免一次性设计过于复杂的流程.
```

## 不适用场景

以下场景工具评估工具免费版不适合处理：

- 无明确技术栈的模糊需求
- 纯架构设计决策
- 运维部署管理

2. 检索候选工具
3. 填充评估矩阵
4. 设计试用周期
5. 决策与执行

**响应解析**: 完成完成后,查看输出响应确认任务状态。成功时输出包含解析摘要和响应数据;失败时根据错误信息排查问题,查阅错误解析章节获取恢复步骤.

## 最佳实践

* 始终从问题出发，而非从工具出发.
* 试用期间记录具体痛点，避免凭印象决策.
* 评估迁移成本时，考虑数据导出与团队适应.
* 工作流设计从最高频手动操作开始，逐步自动化.
* 30天内不重复评估同类工具，避免切换综合征.
* AI 时代的工具更迭很快，关注真正解决问题而非追新.
## 常见问题

**Q：免费版支持工具对比报告导出吗？**
A：免费版仅在对话中展示评估矩阵，不提供报告导出。如需 PDF/Markdown 报告，请考虑 PRO 版本.
**Q：免费版支持团队协作评估吗？**
A：免费版面向个人评估。如需多人协作评估与投票，请使用 PRO 版本.
**Q：评估矩阵的维度可以自定义吗？**
A：免费版使用固定 5 维度。如需自定义维度与权重，请使用 PRO 版本.
**Q：可以推荐具体的产品吗？**
A：免费版提供评估框架与方法，不内置产品数据库。具体推荐会根据你的约束在对话中给出.

## 错误处理

| 错误场景 | 原因 | 处理方式 |
|:---:|:---:|:---:|
