# 表情符号工具箱（免费版）

> **在表情符号中隐藏消息。Unicode变体选择器让隐秘通信成为可能。**

有些信息需要在公开渠道传递但不希望被一眼看到。表情符号工具箱利用Unicode变体选择器（Variation Selectors, VS1-VS256）技术，将文本编码到表情符号内部。聊天应用只显示表情符号本身，但隐藏的变体选择器字符被完整保留，接收方可以解码出原始文本。

## 核心理念

```text
┌──────────────────────────────────────────────────┐
│              隐藏消息编码原理                      │
├──────────────────────────────────────────────────┤
│                                                  │
│  表面层：接收方看到的表情符号                       │
│          🥜                                        │

│  隐藏层：实际携带的变体选择器序列                   │

│                                                  │

│    表情符号 → 提取变体选择器 → 字节还原 → 文本     │

└──────────────────────────────────────────────────┘
```

## 核心能力

### 1. 隐藏消息编码

将任意文本隐藏到表情符号内部，视觉上只显示表情符号。

**编码原理**：
- 输入：一个载体表情符号（推荐使用`🥜`）和一段文本
- 输出：一个视觉上看起来像原表情符号的字符串，但内部包含隐藏文本
- 技术：文本的每个UTF-8字节映射到对应的Unicode变体选择器（VS1-VS256）

**编码示例**：
```bash
# 将文本编码到表情符号中
node ./bin/emoji-toolkit.js encode "🥜" "你好"

# 将代币字符串编码到表情符号中
node ./bin/emoji-toolkit.js encode "🥜" "tokenA..."
```

**载体表情符号选择建议**：
| 表情符号 | 推荐度 | 理由 |
|----------|--------|------|
| 🥜 花生 | 高 | 不常见，不易引起注意 |
| 🌰 栗子 | 高 | 同上 |
| 🔭 药丸 | 中 | 可能有隐喻含义 |
| 🎱 台球 | 中 | 视觉简洁 |
| 自定义 | - | 避免使用过于常见的表情 |

> 技巧：在表情符号后跟随任意普通文本（哪怕一个字符），可以降低消息被截断或损坏的概率。

### 2. 隐藏消息解码

从包含隐藏消息的文本中提取原始文本。

**解码原理**：
- 输入：完整的消息文本（可能包含其他文本和表情符号）
- 输出：嵌入的UTF-8文本，通常是代币字符串或隐藏消息

**解码语义**（重要）：
解码器忽略普通字符，直到找到第一个变体选择器字节，然后收集字节直到遇到变体选择器之后的第一个普通字符。

```bash
# 解码隐藏消息
node ./bin/emoji-toolkit.js decode "<粘贴完整消息>"

# 解码并显示元数据
node ./bin/emoji-toolkit.js decode "<粘贴完整消息>" --metadata

# JSON格式输出
node ./bin/emoji-toolkit.js decode "<粘贴完整消息>" --metadata --json
```

**JSON输出示例（代币）**：
```json
{
  "text": "tokenB...",
  "isToken": true,
  "metadata": {
    "mint": "https://mint.example",
    "unit": "sat",
    "amount": 21
  },
  "metadataError": null
}
```

**JSON输出示例（普通隐藏消息）**：
```json
{
  "text": "你好，这是隐藏消息",
  "isToken": false
}
```

### 3. 代币元数据本地解析

无需联网即可解析代币的基本元数据信息。

| 元数据字段 | 说明 | 获取方式 |
|------------|------|----------|
| mint | 铸造源URL | 本地解析代币字符串 |
| unit | 货币单位 | 本地解析 |
| amount | 面额数量 | 本地解析 |

> 重要：元数据解析是**本地操作**，不访问铸造源服务器，无法证明代币未被花费或是否有效。仅用于预检查。

### 4. Unicode变体选择器原理

| 概念 | 说明 |
|------|------|
| 变体选择器 | Unicode码位U+FE00到U+FE0F（VS1-VS16）和U+E0100到U+E01EF（VS17-VS256） |
| 设计初衷 | 用于指定前一个字符的显示变体（如文本形式vs表情形式） |
| 隐藏编码原理 | 将文本的UTF-8字节值映射到变体选择器的序号，附加到表情符号后 |
| 可见性 | 大多数应用只显示基础表情符号，变体选择器不可见 |
| 保留性 | 部分聊天应用会截断或规范化Unicode，导致变体选择器丢失 |

### 5. 消息完整性检测

| 检测场景 | 说明 | 处理方式 |
|----------|------|----------|
| 正常解码 | 变体选择器完整保留 | 成功提取隐藏文本 |
| 部分截断 | 变体选择器被部分删除 | 解码返回不完整文本或乱码 |
| 完全丢失 | 变体选择器被全部删除 | 无法解码，提示重新发送 |
| 规范化 | 应用将变体选择器标准化 | 可能导致解码失败 |

> 如果解码返回部分代币或乱码，说明聊天应用可能破坏了变体选择器。建议重新发送，并在表情符号后添加一些普通文本。

### 6. 安全使用指南

| 安全原则 | 说明 |
|----------|------|
| 代币即现金 | 解码出的代币字符串是持有人资产，应像现金一样对待 |
| 本地解析局限性 | --metadata仅做本地解析，无法证明代币未被花费 |
| 消息截断风险 | 部分聊天应用会截断Unicode，导致隐藏信息丢失 |
| 传输可靠性技巧 | 在表情符号后添加普通文本可提高传输可靠性 |
| Telegram技巧 | 在代码块/等宽格式中发送可更好保留隐藏字符 |

## 使用场景

### 场景一：隐私消息传递（个人用户角色）

**痛点**：在公开群聊中需要传递只让特定人看到的消息，不想使用加密工具引起注意。

**解决方案**：
```bash
# 编码隐藏消息
node ./bin/emoji-toolkit.js encode "🥜" "晚上8点老地方见"

# 发送：群聊中只显示🥜，接收方解码获取消息
# 解码：
node ./bin/emoji-toolkit.js decode "<复制的消息>"
```

**效果**：群聊中只显示一个花生表情符号，只有知道方法的人才能解码出隐藏消息。

### 场景二：数字代币安全传输（加密货币用户角色）

**痛点**：需要在聊天应用中发送数字代币（如电子现金代币），但不想直接暴露代币字符串。

**解决方案**：
```bash
# 将代币编码到表情符号中
node ./bin/emoji-toolkit.js encode "🥜" "tokenB..."

# 接收方解码并检查元数据
node ./bin/emoji-toolkit.js decode "<复制的消息>" --metadata --json

# 输出：面额、单位、铸造源等元数据
```

**效果**：代币隐藏在表情符号中传输，聊天记录中不显示代币字符串，降低被窃取风险。

### 场景三：隐藏水印嵌入（内容创作者角色）

**痛点**：希望在分享的文本内容中嵌入隐藏水印，用于版权追溯，但不影响阅读体验。

**解决方案**：
```bash
# 在文章末尾的表情符号中嵌入水印
node ./bin/emoji-toolkit.js encode "🎯" "作者ID:12345 时间:2026-07-18"

# 文章末尾显示🎯，但隐藏了作者标识和时间戳
# 追溯时解码即可获取水印信息
```

**效果**：内容中嵌入不可见的版权水印，不影响阅读，但可追溯来源。

## 最佳实践

### 编码最佳实践

1. **选择合适载体**：使用不常见的表情符号（如🥜）作为载体
2. **添加尾部文本**：在表情符号后添加至少一个普通字符，提高传输可靠性
3. **短文本优先**：隐藏文本越短，变体选择器越少，传输越稳定
4. **测试传输**：编码后先测试发送，确认目标应用能保留变体选择器

### 解码最佳实践

1. **复制完整消息**：解码时必须复制完整的消息文本，不要只复制表情符号
2. **检查元数据**：使用--metadata参数检查代币基本信息
3. **JSON输出**：程序化处理时使用--json输出
4. **完整性检查**：如果解码结果不完整或乱码，提示重新发送

### 安全最佳实践

1. **代币即现金**：解码出的代币字符串等同于现金，妥善保管
2. **不信赖元数据**：--metadata仅做本地预检，不证明代币有效
3. **注意传输环境**：部分聊天应用会破坏变体选择器
4. **代码块发送**：在Telegram等应用中使用代码块格式发送，保留更好

## 常见问题

### Q1：为什么有些聊天应用无法正确传输隐藏消息？

许多聊天应用会截断或规范化Unicode字符。变体选择器是特殊的Unicode字符，如果应用在消息处理过程中将其删除或标准化，嵌入的隐藏信息就会丢失。建议在表情符号后添加普通文本、使用代码块格式发送，或更换支持完整Unicode保留的聊天应用。

### Q2：解码返回乱码或不完整文本怎么办？

这通常意味着聊天应用破坏了变体选择器字符。请要求发送方重新发送消息，建议在表情符号后添加一些普通文本（哪怕一个字符），这可以降低消息被截断的概率。在Telegram中，使用代码块/等宽格式发送也有帮助。

### Q3：--metadata能验证代币是否有效吗？

不能。--metadata是本地解析操作，仅从代币字符串中提取面额、单位和铸造源等基本信息，不访问铸造源服务器。它无法证明代币未被花费或是否仍然有效。要验证代币有效性，需要向铸造源发起兑换请求。

### Q4：隐藏消息会被检测到吗？

变体选择器在大多数应用中不可见，普通用户无法察觉。但技术上可以通过Unicode分析检测到变体选择器的存在。因此，这不适用于对抗专业检测的高安全场景，仅适用于日常隐私保护。
