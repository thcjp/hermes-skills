# 神经缓存增强（NeuroCache Pro）

**为什么不用向量检索？** 向量检索只能找与查询相似的文档。NeuroCache 通过图遍历找到*概念关联*的记忆——即使无关键词或 embedding 重叠。"我们当时对认证做了什么决定？"会同时激活时间+实体+概念神经元，找到交集。

## 核心能力

### 1. 扩散激活联想检索
通过图遍历找到概念关联记忆，即使无关键词/embedding 重叠。查询同时激活多类神经元（时间/实体/概念），通过交集定位关联记忆。支持 CAUSED_BY/LEADS_TO 突触链遍历追溯完整因果链，而非单独返回相似记忆。

**输入**: 用户提供扩散激活联想检索所需的指令和必要参数。
**处理**: 解析扩散激活联想检索的输入参数,完成核心逻辑,返回结构化响应。
**输出**: 返回扩散激活联想检索的响应数据,包含状态码、结果和日志。

- 执行此能力时使用`input_params`参数,支持创建/查询/导出操作
### 2. 赫布学习与 20 种突触类型
频繁共访问的记忆自动强化连接（赫布学习），记忆形成神经图谱。支持 20 种类型化突触：时间（BEFORE/AFTER）、因果（CAUSED_BY/LEADS_TO）、语义（IS_A/HAS_PROPERTY/PART_OF）、情感（FELT/EVOKES）、冲突（CONTRADICTS）、关联（SIMILAR_TO/RELATED_TO）、层级（PARENT_OF/CHILD_OF）、引用（REFERENCES/CITED_BY）、位置（LOCATED_AT/CO_OCCURRED）、动作（PERFORMED_BY/RESULTED_IN）。

**输入**: 用户提供赫布学习与 20 种突触类型所需的指令和必要参数。
**处理**: 解析赫布学习与 20 种突触类型的输入参数,完成核心逻辑,返回结构化响应。
**输出**: 返回赫布学习与 20 种突触类型的响应数据,包含状态码、结果和日志。

- 执行此能力时使用`input_params`参数,支持创建/查询/导出操作
### 3. 矛盾自动检测与降权
当新记忆与已有记忆通过 CONTRADICTS 突触连接时，系统自动比较时间戳与优先级，降权过时/低优先级记忆，标记冲突供用户确认。新旧事实矛盾时降权旧记忆保留新记忆为主；同优先级矛盾标记待确认暂不降权。

**输入**: 用户提供矛盾自动检测与降权所需的指令和必要参数。
**处理**: 解析矛盾自动检测与降权的输入参数,完成核心逻辑,返回结构化响应。
**输出**: 返回矛盾自动检测与降权的响应数据,包含状态码、结果和日志。

- 执行此能力时使用`input_params`参数,支持创建/查询/导出操作
### 4. 4 级深度分级检索
depth=0 即时（<10ms，快速事实）、depth=1 上下文（~50ms，标准召回，默认）、depth=2 习惯（~200ms，模式匹配/工作流建议）、depth=3 深度（~500ms，跨领域关联/因果链），让用户控制速度/深度权衡，避免所有查询都走全图遍历浪费 token。

**输入**: 用户提供4 级深度分级检索所需的指令和必要参数。
**处理**: 解析4 级深度分级检索的输入参数,完成核心逻辑,返回结构化响应。
**输出**: 返回4 级深度分级检索的响应数据,包含状态码、结果和日志。

- 执行此能力时使用`input_params`参数,支持创建/查询/导出操作
### 5. 艾宾浩斯衰减与版本快照
陈旧记忆自然衰减（可调参数：initial_decay/decay_interval_days/min_activation/prune_threshold/reinforce_factor），高优先级记忆衰减更慢，访问时自动强化。支持大脑版本快照与一键回滚（`nmem_version snapshot/rollback`），以及大脑移植跨项目知识迁移（`nmem_transplant`，保留原突触结构）。

**输入**: 用户提供艾宾浩斯衰减与版本快照所需的指令和必要参数。
**处理**: 解析艾宾浩斯衰减与版本快照的输入参数,完成核心逻辑,返回结构化响应。
**输出**: 返回艾宾浩斯衰减与版本快照的响应数据,包含状态码、结果和日志。

## 适用场景

**何时使用：**
- 需要因果链推理的复杂查询："上周部署为什么失败？"追溯 CAUSED_BY 突触链
- 跨领域概念关联发现："认证和缓存有什么关系？"通过扩散激活找交集
- 长期项目知识沉淀：决策、错误、洞察随时间积累并自动关联
- 冲突信息自动识别：新旧事实矛盾时自动降权过时记忆
- 需要版本化管理的记忆库：风险操作前快照，出问题回滚
- 跨项目知识复用：将高价值决策和教训从项目 A 迁移到项目 B

**输入输出：**
- 输入：决策/错误/事实/洞察/偏好/待办等记忆条目（含 type/priority/tags）；检索查询（含 depth 参数）
- 输出：按关联度激活的记忆条目（含因果链/跨领域关联）；矛盾检测结果（含降权标记与附注）

**不适用场景：**
- 简单的关键词精确匹配检索（用传统搜索或 TF-IDF 即可）
- 需要向量语义相似检索的场景（向量检索找"相似"文档更合适）
- 记忆量极小（<50 条）的场景（图遍历优势不明显，开销不划算）
- 需要 LLM 实时推理的记忆系统（本系统核心检索纯算法零 LLM 依赖）

```bash
nmem stats
```

应显示大脑统计（神经元数、突触数、纤维数）。

### 第 4 步：会话开始时注入上下文

调用 `nmem_context` 注入近期记忆到感知。若用户提及特定主题，调用 `nmem_recall` 查询相关记忆。

### 第 5 步：对话进行中存储与检索

| 情境 | 动作 |
|:---|:---|
| 做出决策 | `nmem_remember` type="decision" |
| 发生错误 | `nmem_remember` type="error" |
| 用户表达偏好 | `nmem_remember` type="preference" |
| 询问过去事件 | `nmem_recall` 选择合适深度（0-3） |
| 识别工作流模式 | `nmem_habits` |

### 第 6 步：会话结束时自动提取

调用 `nmem_auto` action="process" 处理重要对话片段，自动提取事实、决策、错误、TODO。

### 第 7 步：定期维护

定期运行 `nmem_health` 大脑健康诊断；超过 5000 神经元启用自动剪枝（weekly）；风险操作前用 `nmem_version snapshot` 创建快照。

**响应解析**: 完成完成后,查看输出响应确认任务状态。成功时输出包含解析摘要和响应数据;失败时根据错误信息排查问题,查阅错误解析章节获取恢复步骤。


## 示例

### 示例

**输入：** 用户问"上周部署为什么失败？"

**输出：** 调用 `nmem_recall(query="部署失败原因", depth=2)`，图遍历 CAUSED_BY 突触链：
- 部署失败 → CAUSED_BY → 数据库连接超时
- 数据库连接超时 → CAUSED_BY → 连接池耗尽
- 连接池耗尽 → CAUSED_BY → 未设置 max_connections
- 未设置 max_connections → LEADS_TO → 需更新配置规范

返回完整因果链，而非单独的相似记忆。

### 示例 2：跨领域关联发现

**输入：** 用户问"认证和缓存有什么关系？"

**输出：** 调用 `nmem_recall(query="认证 缓存 关联", depth=3)`，扩散激活同时激活认证相关神经元（JWT/session/token）和缓存相关神经元（Redis/TTL/invalidation），交集发现：token 缓存策略。返回："认证 token 用 Redis 缓存，TTL=3600s，上次因缓存未失效导致权限问题（3 周前）"

### 示例 3：矛盾自动处理

**输入：**
- 会话 A（1 月）：`nmem_remember("API 限流 100 req/min", type="fact", priority=5)`
- 会话 B（2 月）：`nmem_remember("API 限流调整为 500 req/min", type="fact", priority=8)`
- 会话 C（3 月）：`nmem_recall("API 限流多少")`

**输出：**
- 会话 B：检测到与 1 月记忆矛盾 → 创建 CONTRADICTS 突触 → 降权旧记忆 → 标记变更
- 会话 C：返回 500 req/min（新，主）+ 附注"曾为 100 req/min（1 月，已过时）"

### 示例 4：版本快照与回滚

**输入：** 风险操作前需保护记忆库

**输出：**
1. `nmem_version snapshot --label "重构前基线"` 创建快照
2. 操作后发现问题：`nmem_version diff --from "重构前基线" --to current` 显示新增/修改/删除的神经元
3. 回滚：`nmem_version rollback --to "重构前基线"` 恢复到快照状态

## 错误处理

| 错误场景 | 原因 | 处理方式 |
|:---|:---|:---|
| 召回为空 | 大脑为空或路径错误 | `nmem stats` 确认非空，检查大脑路径，确认已存入记忆 |
| 召回太慢 | 神经元规模过大 | 降低 depth 到 1-2；启用自动剪枝；超 10000 考虑分区 |
| 矛盾未检测 | 记忆 type/tags 不正确 | 检查 CONTRADICTS 突触是否创建，确认记忆 type/tags 正确 |
| 记忆过早衰减 | priority 设置过低 | 提高 priority 到 7+；调低 initialDecay；调高 reinforceFactor |
| 大脑损坏 | 数据文件损坏 | `nmem_health` 诊断，从版本快照回滚 |
| 移植失败 | 过滤条件过严或源大脑不存在 | 放宽 filter 条件，确认 from 大脑存在 |
| 工具协议连接失败 | python3 路径或包未安装 | 确认 `pip install neural-memory` 成功，检查 python3 路径 |
| 赫布学习过拟合 | 噪声连接被强化 | 调低 reinforceFactor，调高 min_activation 阈值 |

## 常见问题

**Q1：扩散激活比向量检索好在哪？**
A：向量检索找"相似"文档，扩散激活找"关联"记忆。例如"认证决策"能通过图遍历找到"部署配置"（因部署依赖认证），即使两者无关键词/embedding 重叠。

**Q2：衰减会不会丢重要记忆？**
A：不会。高优先级记忆衰减慢，访问时自动强化（赫布学习）。仅长期未访问且低优先级的记忆会被降权/剪枝。建议重要记忆设 priority>=7。

**Q3：图谱太大影响性能怎么办？**
A：启用自动剪枝（weekly），调整 minActivation 阈值。超过 10000 神经元考虑大脑分区（按项目/领域拆分）或迁移到 SQLite 后端。

**Q4：矛盾检测会误报吗？**
A：可能。系统标记"潜在矛盾"供用户确认，不自动删除。同优先级矛盾暂不降权，等用户确认。

**Q5：零 LLM 依赖是真的吗？**
A：核心检索纯算法（regex + 图遍历 + 赫布学习）。仅 `nmem_auto` 自动提取可选调用 LLM，也可用 regex 模式匹配替代。

## 已知限制

1. **图谱规模影响性能**：1000 神经元以下无影响，5000+ 时 depth=2/3 明显变慢，10000+ 全深度变慢，需启用分区或迁移 SQLite 后端，无自动水平扩展机制。
2. **矛盾检测基于语义重叠**：依赖 type/tags 匹配发现冲突，语义不同但实质矛盾的记忆（如用不同术语描述同一决策）可能漏检，无法像 LLM 那样理解深层语义矛盾。
3. **突触关系需正确标注**：CAUSED_BY/LEADS_TO 等因果链追溯依赖存储时正确指定突触类型，若 Agent 未标注突触类型则无法进行关联检索。
4. **衰减参数需手动调优**：默认艾宾浩斯衰减参数（initial_decay=0.1/interval=7 天）不适合所有场景，长期项目和快速迭代项目需分别调优，无自适应参数调整。
5. **大脑移植保留原结构但可能引入噪声**：跨项目迁移知识时保留原突触结构，但不同项目的上下文差异可能导致迁移的记忆在目标大脑中产生错误关联，需人工审查。
