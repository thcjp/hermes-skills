## 使用场景

### 场景一：多项目并行记忆管理（独立开发者角色）

```bash
# 为每个项目创建独立脑
nmem create-brain --name project-a
nmem create-brain --name project-b
nmem create-brain --name project-c
# ...
# 切换到项目A的脑
nmem switch-brain --name project-a

  content="项目A使用React + TypeScript",
  type="decision",
  tags=["frontend", "tech-stack"],
  priority=8
)
```

**效果**：3个项目记忆完全隔离，切换成本从5分钟降至0。跨项目不会混淆上下文.

## 性能优化策略

### 多脑管理优化

1. **脑大小控制**：单个脑建议不超过10万条记忆，过大时拆分
2. **自动切换优化**：配置上下文关键词，提升自动切换准确率
3. **脑索引优化**：定期重建脑索引，提升检索速度
4. **冷脑归档**：不活跃的脑归档压缩，节省存储

### 深度检索优化

1. **深度按需选择**：日常用depth=1，复杂问题用depth=3
2. **超时控制**：depth=3设置500ms超时，避免长时间阻塞
3. **结果缓存**：相同查询缓存结果，避免重复检索
4. **并行检索**：多个查询并行执行，提升吞吐

### 快照管理优化

1. **增量快照**：仅快照变更部分，节省存储与时间
2. **快照压缩**：旧快照自动压缩归档
3. **快照保留策略**：保留最近30个快照，自动清理旧快照
4. **快照验证**：定期验证快照完整性

### 自动提取优化

1. **置信度调优**：调整提取置信度阈值，平衡召回与精确
2. **类别过滤**：仅提取特定类别，过滤噪音
3. **批量处理**：大量对话批量提取，减少API调用
4. **去重策略**：启用自动去重，避免重复记忆

### 成本控制

- 深度检索按需使用：日常depth=1，复杂问题depth=3
- 自动提取阈值调优：避免提取低价值信息
- 快照定期清理：避免存储膨胀
- 冷脑归档：不活跃脑压缩存储

## FAQ

### Q1：免费版与专业版有什么区别？

免费版提供基础联想记忆、单脑存储、手动记忆管理、深度0-1的检索。专业版解锁多脑管理、脑版本快照、脑移植、深度2-3的因果链追踪、习惯模式识别、矛盾自动检测、自动事实提取七大高级功能。此外提供多角色场景指南、性能优化策略和多平台集成示例.
### Q2：多脑管理如何实现项目隔离？

每个脑是独立的SQLite数据库，存储在`~/.neuralmemory/brains/<brain>.db`。脑间严格隔离，无数据泄露。通过`nmem switch-brain`切换当前活跃脑，所有后续操作仅作用于该脑。自动切换模式可根据上下文关键词自动切换到对应脑.
### Q3：脑快照会占用很多存储吗？

快照采用增量存储，仅记录变更部分。单个快照通常为脑大小的5-15%。建议配置保留最近30个快照，自动清理旧快照。旧快照自动压缩归档，进一步节省存储。可配置每日自动快照，确保操作可回滚.
### Q4：深度3检索会很慢吗？

depth=3检索约500ms，适用于复杂问题。对于日常查询，建议使用depth=1（~50ms）。depth=3沿因果突触（CAUSED_BY/LEADS_TO）跨域追踪，适合"为什么会这样"类问题。可设置超时（默认500ms），避免长时间阻塞.
### Q5：习惯模式识别如何工作？

习惯识别分析记忆访问序列，识别重复的行为模式。例如，如果多次出现"测试→代码审查→构建"的序列，系统会识别为工作流模式，并建议自动化。识别阈值可配置（默认min_pattern_length=3，建议阈值0.7）.
### Q6：矛盾检测会自动删除旧记忆吗？

不会。矛盾检测不会删除任何记忆，而是：(1) 在冲突记忆间建立CONTRADICTS突触；(2) 降低过时记忆的检索权重；(3) 提示用户确认冲突。用户可选择保留两条、删除旧记忆或合并。这是安全的冲突处理机制.
### Q7：自动事实提取的准确率如何？

自动提取基于模式匹配与上下文理解，对中英文混合对话的提取准确率约85%。可通过`confidence_threshold`参数调整提取置信度阈值（默认0.7）。建议重要对话人工审核提取结果。提取类别可配置（决策/错误/待办/事实）.
### Q8：脑移植会复制所有记忆吗？

不会。脑移植支持按标签过滤，仅迁移特定领域的知识。例如`--filter-tags database,infrastructure`仅迁移数据库与基础设施相关记忆。移植时自动去重，避免重复记忆。移植是复制操作，源脑记忆不受影响.
### Q9：可以同时使用多个脑吗？

可以。虽然同一时间只有一个活跃脑，但可通过`nmem_recall --brain <name>`跨脑检索。专业版支持跨脑检索，将多个脑的检索结果合并。这适合需要跨项目知识关联的场景.
### Q10：记忆数据存储在哪里？安全吗？

所有记忆存储在本地`~/.neuralmemory/brains/`目录下的SQLite数据库中。数据不发送到外部服务（除非配置可选的嵌入提供方）。每个脑独立隔离，无跨脑污染。建议将`~/.neuralmemory/`纳入Git版本控制，实现额外备份.
### Q11：如何备份与恢复脑数据？

专业版提供三种备份方式：(1) 脑快照（`nmem snapshot`），用于操作前备份；(2) 脑导出（`nmem export --format json`），用于完整备份；(3) Git版本控制，将`~/.neuralmemory/`纳入Git。恢复方式：快照回滚（`nmem rollback`）或脑导入（`nmem import`）.

## 故障排查表

| 问题 | 可能原因 | 解决方案 | 优先级 |
|:------:|--------|:-------|:------:|
| 多脑切换失效 | 脑名称错误或脑不存在 | 验证脑名称；检查脑是否存在 | 高 |
| 快照回滚失败 | 快照损坏或版本不兼容 | 验证快照完整性；创建新快照 | 高 |
| 深度检索超时 | 神经图谱过大或深度过高 | 降低depth；增加超时时间；优化图谱 | 中 |
| 习惯识别无结果 | 行为序列不足或阈值过高 | 降低min_pattern_length；降低建议阈值 | 低 |
| 矛盾检测误报 | 语义相似但非真实矛盾 | 调整矛盾检测灵敏度；手动标记非矛盾 | 中 |
| 自动提取质量差 | 对话上下文不足或阈值过低 | 调整提取阈值；人工审核提取结果 | 中 |
| 脑移植后重复 | 去重未启用 | 启用`--deduplicate`参数 | 低 |
| 跨脑检索缓慢 | 脑数量过多或图谱过大 | 限制跨脑数量；优化各脑图谱 | 低 |
| 记忆衰减过快 | priority设置过低 | 提高关键记忆的priority | 中 |
| 突触连接异常 | 神经图谱损坏 | 执行`nmem health --repair`修复 | 高 |
