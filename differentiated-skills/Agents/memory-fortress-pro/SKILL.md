## 六层记忆架构

### 第一层：热内存（SESSION-STATE.md）

活跃工作记忆，在上下文压缩、重启或中断后依然存活。采用写前日志（WAL）协议。

```markdown
## 当前任务
[我们正在做什么]

## 关键上下文
- 用户偏好：...
- 已做决策：...
- 当前阻塞：...

## 待办事项
- [ ] ...
```

**核心规则**：在响应前先写入。由用户输入触发，而非依赖Agent记忆。

### 第二层：温存储（LanceDB向量数据库）

跨所有记忆的语义搜索。自动召回注入相关上下文。专业版完整启用向量语义检索。

```bash
# 存储记忆
memory_store text="用户偏好深色模式" category="preference" importance=0.9

# 语义召回记忆（非关键词匹配，而是语义理解）
memory_recall query="项目状态" limit=5

# 模糊语义搜索
memory_recall query="之前讨论的前端框架选择" limit=10
```

**专业版优势**：
- 基于向量嵌入的语义搜索，理解意图而非匹配关键词
- 自动召回相关上下文，无需用户明确指定搜索词
- 支持模糊查询（"之前讨论的那个数据库"也能找到）
- 相似度阈值可配置（minScore），平衡召回率与精确度

### 第三层：冷存储（Git-Notes知识图谱）

结构化的决策、经验与上下文存储。支持分支感知。

```bash
# 记录决策
python3 memory.py -p $DIR remember '{"type":"decision","content":"前端使用React"}' -t tech -i h

# 检索记忆
python3 memory.py -p $DIR get "前端"

# 按分支检索
python3 memory.py -p $DIR get "前端" --branch feature/auth

# 导出全部记忆
python3 memory.py -p . export --format json > memories.json
```

### 第四层：策展归档（MEMORY.md + 日志）

人类可读的长期记忆。日志目录 + 提炼后的智慧沉淀。

```text
workspace/
├── MEMORY.md              # 策展长期记忆（精华内容）
└── memory/
    ├── 2026-01-30.md      # 每日日志
    ├── 2026-01-29.md
    ├── topics/            # 主题专属文件
    ├── projects/          # 项目专属记忆
    ├── decisions/         # 决策记录
    └── lessons/           # 经验教训
```

### 第五层：云备份（跨设备同步）— 专业版启用

跨设备同步，可与知识库对话。专业版完整支持云备份。

```bash
export CLOUD_MEMORY_API_KEY="[REDACTED]"

# 添加记忆至云端
cloud_memory add "重要上下文：项目架构决策"

# 语义搜索云端知识库
cloud_memory search "我们决定了什么关于..."

# 跨设备同步
cloud_memory sync --direction both
```

**专业版优势**：
- 跨设备同步：在电脑、笔记本、服务器间共享记忆
- 知识库对话：用自然语言查询全部历史记忆
- 增量同步：仅传输变更部分，节省带宽
- 冲突解决：自动合并或提示手动解决

### 第六层：自动提取（Mem0）— 专业版启用

自动从对话中提取事实，减少约80%的token消耗。专业版完整支持自动事实提取。

```bash
npm install mem0ai
export MEM0_API_KEY="[REDACTED]"
```

```javascript
const { MemoryClient } = require('mem0ai');
const client = new MemoryClient({ apiKey: process.env.MEM0_API_KEY });

// 对话自动提取事实
await client.add(messages, { user_id: "user123" });

// 检索相关记忆
const memories = await client.search(query, { user_id: "user123" });
```

**专业版优势**：
- 自动提取偏好、决策、事实，无需手动记录
- 自动去重与更新已有记忆
- 减少80%的token消耗（相比原始对话历史）
- 跨会话自动工作，无需用户干预

## WAL协议（写前日志）— 关键机制

**写前日志**：在响应前先写入状态，而非响应后。

| 触发条件 | 执行动作 | 存储层级 |
|----------|----------|----------|
| 用户表达偏好 | 写入SESSION-STATE.md → 然后响应 | 热内存 + 温存储 |
| 用户做出决策 | 写入SESSION-STATE.md → 然后响应 | 热内存 + 冷存储 |
| 用户给出截止日期 | 写入SESSION-STATE.md → 然后响应 | 热内存 |
| 用户纠正你 | 写入SESSION-STATE.md → 然后响应 | 热内存 + 经验教训 |
| 对话中产生事实 | Mem0自动提取 → 存入温存储 | 自动提取层 |

**为什么？** 如果你先响应，在保存前发生崩溃或压缩，上下文就会丢失。WAL协议确保记忆持久性。这是数据库领域经典的持久性保障机制，同样适用于Agent记忆管理。

## 核心能力
### 会话开始时
1. 读取SESSION-STATE.md——这是你的热上下文
2. 语义检索相关历史上下文（`memory_recall`，专业版使用向量搜索）
3. 查看 memory/YYYY-MM-DD.md 了解近期活动
4. 检查云端是否有新同步的记忆（专业版）

输出结果包含操作状态和返回数据。
### 对话过程中
1. **用户给出具体细节？** → 先写入SESSION-STATE.md，再响应
自动执行记忆检索和存储,处理用户输入并返回相关上下文。

2. **重要决策做出？** → 静默存入Git-Notes + 向量数据库
3. **偏好表达？** → `memory_store` 设置importance=0.9
4. **对话产生事实？** → Mem0自动提取（专业版，无需手动操作）

输出结果包含操作状态和返回数据。
### 会话结束时

执行记忆归档处理,将会话摘要保存到长期记忆库。

1. 用最终状态更新SESSION-STATE.md
2. 将重要条目移至MEMORY.md（如值得长期保留）
3. 创建/更新 memory/YYYY-MM-DD.md 日志
4. 触发云端增量同步（专业版）

### 记忆卫生（每周）
1. 审查SESSION-STATE.md——归档已完成任务
2. 检查向量数据库中的垃圾数据：`memory_recall query="*" limit=50`
3. 清除无关向量：`memory_forget id=<id>`
4. 将日志合并至MEMORY.md
5. 检查Mem0提取的记忆质量，修正错误提取
6. 验证云端同步状态：`cloud_memory status`

## 性能优化策略

### 向量数据库优化

1. **索引调优**：根据记忆数量调整向量索引参数（小规模用Flat，大规模用IVF）
2. **批量插入**：大量记忆写入时使用批量模式，减少IO开销
3. **分区存储**：按项目或时间分区，加速检索
4. **定期压缩**：清理低重要性向量，减少索引体积

### Mem0提取优化

1. **阈值调优**：调整自动提取的置信度阈值，平衡召回与精确
2. **分类过滤**：仅提取特定类别（决策/偏好/事实），过滤噪音
3. **去重策略**：启用自动去重，避免重复记忆占用空间
4. **更新策略**：新记忆与旧记忆冲突时，自动更新而非追加

### 云同步优化

1. **增量同步**：仅传输变更部分，非全量同步
2. **压缩传输**：同步数据启用压缩，节省带宽
3. **冲突解决**：配置自动合并策略（最新优先/手动解决）
4. **离线缓存**：网络不可用时缓存变更，恢复后自动同步

### 成本控制

- 低重要性记忆使用本地存储，不同步至云端
- Mem0提取设置最小重要性阈值，避免提取无价值信息
- 向量搜索限制maxResults，避免过度检索
- 定期执行记忆卫生，清理过期与无用记忆

## FAQ

### Q1：免费版与专业版有什么区别？

免费版提供核心三层记忆（热内存、冷存储、策展归档）。专业版解锁全部六层，新增向量语义搜索（LanceDB）、Mem0自动事实提取（减少80%token）、云备份（跨设备同步）。此外提供多角色场景指南、性能优化策略和多平台集成示例。

### Q2：WAL协议是什么？为什么重要？

WAL（Write-Ahead Log）即写前日志，要求Agent在响应前先将状态写入持久存储。这确保即使发生崩溃、压缩或中断，关键上下文也不会丢失。这是数据库领域经典的持久性保障机制。在Agent记忆场景中，WAL协议保证用户偏好、决策和关键信息在被处理前已持久化。

### Q3：向量语义搜索与关键词搜索有什么区别？

关键词搜索只匹配字面相同的词，而向量语义搜索理解查询的语义意图。例如，搜索"前端框架选择"可以找到包含"React vs Vue讨论"的记忆，即使没有"前端框架"这个确切词组。专业版使用LanceDB向量数据库，通过嵌入模型将文本转为向量进行相似度检索。

### Q4：Mem0自动提取如何工作？

Mem0在对话过程中自动识别并提取偏好、决策、事实等结构化信息，无需用户手动记录。它还会自动去重和更新已有记忆。例如，当用户说"我们改用PostgreSQL吧"，Mem0会自动提取"决策：数据库从X改为PostgreSQL"并更新之前的相关记忆。这减少了约80%的token消耗，因为不需要在上下文中保留完整对话历史。

### Q5：SESSION-STATE.md应该多大？

建议保持在2KB以内。它只存储当前会话的活跃上下文，完成的任务应归档至日志或MEMORY.md。过大会影响Agent读取速度。专业版的Mem0自动提取可以进一步减轻SESSION-STATE.md的负担，因为事实已自动存储在向量数据库中。

### Q6：记忆太多会影响Agent性能吗？

会。记忆过多会导致上下文过载和检索变慢。建议每周执行记忆卫生流程：清理无关向量、归档旧日志、将精华合并至MEMORY.md。MEMORY.md建议控制在5KB以内。专业版支持按重要性阈值过滤，仅注入高相关性记忆。定期运行 `memory_recall query="*" limit=50` 检查记忆质量。

### Q7：云备份如何处理冲突？

专业版云备份支持三种冲突解决策略：
- **自动合并**（默认）：将冲突的记忆合并，保留两边的信息
- **最新优先**：以时间戳最新的版本为准
- **手动解决**：检测到冲突时提示用户选择

可通过 `agent-config.json` 中的 `conflictResolution` 字段配置。

### Q8：如何确保子Agent也能获取记忆？

在派生子Agent的任务提示中显式包含关键上下文。子Agent默认不继承父Agent的完整上下文，需要手动传递。也可以让子Agent直接读取SESSION-STATE.md。专业版支持通过向量搜索让子Agent按需检索相关记忆，而非全量继承。

### Q9：Git-Notes冷存储与向量数据库有什么区别？

Git-Notes存储结构化的决策与经验，支持分支感知和版本追踪，适合机器检索和版本管理。向量数据库存储非结构化的记忆片段，支持语义搜索，适合模糊查询和自动召回。两者互补使用：Git-Notes用于正式决策记录，向量数据库用于对话中的上下文召回。

### Q10：可以只使用部分层级吗？

可以。六层架构是模块化设计，可按需启用。最简配置仅需热内存（第一层）。推荐配置为热内存+冷存储+策展归档（免费版）。完整配置启用全部六层（专业版）。可通过 `agent-config.json` 中的 `plugins.entries` 配置各层启用状态。

### Q11：记忆数据存储在哪里？安全吗？

所有记忆数据存储在本地 `~/workspace/` 目录下。云备份功能（专业版）会将记忆同步至云端，数据传输使用加密通道。API Key存储在环境变量中，不硬编码在配置文件里。建议将 `~/workspace/memory/` 加入git版本控制，实现额外的版本备份。

| 问题 | 可能原因 | 解决方案 | 优先级 |

| 向量数据库占用过大 | 低重要性记忆未清理 | 设置minImportance过滤；定期执行 `memory_prune` | 低 |
