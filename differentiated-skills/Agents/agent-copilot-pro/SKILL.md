# 代理副驾驶（Agent Copilot Pro）

面向 AI Agent 开发者的**工程化副驾驶**，把零散的 Prompt 工程经验沉淀为模板库、评估器与循环工程工作流，让 Agent 从"能跑"升级到"稳定可控"。

## 核心能力

### 1. System Prompt 五段式结构化生成
强制按"角色定位→能力边界→行为规范→输出格式→异常处理"五段生成，配套质量检查清单（5 项必检）

**输入**: 用户提供System Prompt 五段式结构化生成所需的指令和必要参数。
**处理**: 按照skill规范执行System Prompt 五段式结构化生成操作,遵循单一意图原则。
**输出**: 返回System Prompt 五段式结构化生成的执行结果,包含操作状态和输出数据。

- 执行此能力时使用`input_params`参数,支持创建/查询/导出操作
### 2. 上下文腐烂诊断器
量化信息密度衰减、噪声累积、注意力漂移三因素，给出治理建议（摘要压缩、约束重申、噪声清理、分段会话、工具结果归档）

**输入**: 用户提供上下文腐烂诊断器所需的指令和必要参数。
**处理**: 按照skill规范执行上下文腐烂诊断器操作,遵循单一意图原则。
**输出**: 返回上下文腐烂诊断器的执行结果,包含操作状态和输出数据。

- 执行此能力时使用`input_params`参数,支持创建/查询/导出操作
### 3. 工具选择决策矩阵
基于任务类型（检索/计算/创作/执行）、参数复杂度（低/中/高）、失败成本（可撤销/可恢复/不可逆）三维度推荐工具调用策略

**输入**: 用户提供工具选择决策矩阵所需的指令和必要参数。
**处理**: 按照skill规范执行工具选择决策矩阵操作,遵循单一意图原则。
**输出**: 返回工具选择决策矩阵的执行结果,包含操作状态和输出数据。

- 执行此能力时使用`input_params`参数,支持创建/查询/导出操作
### 4. 任务拆解 DAG
将复杂任务拆为有向无环图，每节点附 `estimated_tokens`、`required_tools`、`retry_policy`、`granularity_score`（建议 0.6-0.8）

**输入**: 用户提供任务拆解 DAG所需的指令和必要参数。
**处理**: 按照skill规范执行任务拆解 DAG操作,遵循单一意图原则。
**输出**: 返回任务拆解 DAG的执行结果,包含操作状态和输出数据。

### 5. Prompt 质量评估器（含幻觉检测）
五维评估（明确性/完整性/可执行性/鲁棒性/幻觉风险）；自动生成 5-10 个回归测试用例（含正常/边界/恶意输入）

**输入**: 用户提供Prompt 质量评估器（含幻觉检测）所需的指令和必要参数。
**处理**: 按照skill规范执行Prompt 质量评估器（含幻觉检测）操作,遵循单一意图原则。
**输出**: 返回Prompt 质量评估器（含幻觉检测）的执行结果,包含操作状态和输出数据。

- 执行此能力时使用`input_params`参数,支持创建/查询/导出操作

## 适用场景

**何时使用**：
- 构建客服/助理类 Agent（退款咨询、工单分发、智能问答）
- 自动化工作流编排（多步骤任务的 DAG 拆解与节点编排）
- RAG 应用增强（Prompt 质量提升与幻觉风险降低）
- 多 Agent 协作系统（节点间数据传递 schema 与全局质量门禁）
- 长会话上下文治理（>10 轮会话的腐烂诊断与治理）
- Prompt 评审与回归测试（批量测试用例生成与通过率统计）

**输入**：自然语言描述的 Agent 需求 / 待诊断的会话历史 / 待评估的 Prompt 文本
**输出**：结构化 Prompt 文本 / 诊断报告（含指标与建议）/ DAG 拆解树 / 评估评分表

**不适用场景**：
- 单次简单问答（无需结构化 Prompt）
- 纯代码实现任务（不涉及 Prompt 工程）
- 会话轮次 <10 的短对话（上下文腐烂诊断无意义）

## 使用流程

### Step 1：明确需求类型
判断用户需求属于以下哪类：生成 Prompt / 诊断会话 / 拆解任务 / 评估 Prompt / 选择 Loop 模式

### Step 2：按需求类型执行
- **生成 Prompt** → 走五段式模板（见 Step 3）
- **诊断会话** → 走上下文腐烂诊断器（见 Step 4）
- **拆解任务** → 走 DAG 拆解（见 Step 5）
- **评估 Prompt** → 走质量评估器（见 Step 6）

### Step 3：五段式 Prompt 生成
```text
[1. 角色定位] 你是{角色}，服务{目标用户}，核心目标是{目标}。
[2. 能力边界] 能做：{列表} / 不能做：{列表} / 不确定时：{兜底}
[3. 行为规范] 语气：{描述} / 流程：{步骤} / 禁忌：{禁止行为}
[4. 输出格式] 响应 schema（JSON / Markdown 模板）
[5. 异常处理] 输入异常：{处理} / 工具失败：{重试降级} / 转人工条件：{触发}
```
生成后过质量检查清单：角色含目标用户、能力区分能做/不能做、行为规范可执行、输出有 schema、异常含转人工条件。

### Step 4：上下文腐烂诊断
统计三因素指标：

| 因素 | 检测方法 | 触发阈值 |
|------|----------|----------|
| 信息密度衰减 | 每轮非重复信息 token 占比 | < 30% 触发治理 |
| 噪声累积 | 无关工具结果/客套话 token 占比 | > 40% 触发清理 |
| 注意力漂移 | 核心约束关键词出现频率 | 连续 5 轮未引用触发提醒 |

输出治理建议：摘要压缩早期轮次、每 5 轮重申核心约束、移除已完成工具原始结果、超 30 轮开新会话。

### Step 5：任务 DAG 拆解
```text
任务："调研竞品并生成报告"
[收集竞品列表] ──┬──> [抓取功能] ──┐
                 ├──> [抓取定价] ──┤──> [聚合分析] ──> [生成报告] ──> [质量校验]
                 └──> [抓取评价] ──┘
每节点属性：node_id / description / depends_on / estimated_tokens / required_tools / retry_policy / granularity_score
```
颗粒度评分：`< 0.4` 拆太细（编排开销大）；`> 0.9` 拆太粗（单节点失败影响大）；建议 0.6-0.8。

### Step 6：Prompt 质量评估与幻觉检测
五维评分（明确性/完整性/可执行性/鲁棒性/幻觉风险），每维有及格线。幻觉检测 5 条规则：是否含"不确定时承认不知道"、是否禁止编造引用、是否要求标注来源、是否对事实性输出加校验、是否限制输出范围。

### Step 7：Agent Loop 模式选择
```text
是否需要调用工具？
├─ 否 → CoT
└─ 是 → 任务步数是否 > 5？
        ├─ 否 → ReAct
        └─ 是 → 是否需要全局规划？
                ├─ 是 → Plan-Execute
                └─ 否 → ReAct + Reflection（每 3 步反思）
```

## 示例

### 示例

**输入**："帮我设计一个客服 Agent 的 system prompt，处理退款咨询"

**输出**（五段式 Prompt）：
```text
[角色定位] 你是退款客服专员，服务申请退款的用户，核心目标是高效处理退款咨询。
[能力边界] 能做：查询订单状态、发起退款、跟踪退款进度 / 不能做：修改订单金额、绕过审批 / 不确定时：转人工
[行为规范] 语气：专业温和 / 流程：核实订单→判断退款条件→执行或转人工 / 禁忌：不承诺具体到账时间
[输出格式] {"decision":"approve|reject|escalate","reason":"...","confidence":0.0-1.0}
[异常处理] 输入异常：提示用户提供订单号 / 工具失败：重试2次后转人工 / 转人工条件：confidence<0.6
```
配套：工具选择矩阵（操作执行×中参数×高失败成本→人工确认）；Loop 选择（步数<5→ReAct）；回归测试用例（正常退款/超期/欺诈/转人工）。

### 示例二：诊断长会话上下文腐烂

**输入**："这个 Agent 聊到第 20 轮就开始跑偏，帮我诊断"

**输出**（诊断报告）：
```text
信息密度衰减率：第 1-5 轮 85% → 第 16-20 轮 42%（触发，<30% 阈值未到但趋势恶化）
噪声累积：无关工具调用结果占上下文 38%（接近 40% 阈值）
注意力漂移：核心约束"必须核实订单号"在第 12 轮后未被引用（触发）
治理建议：
1. 第 1-10 轮压缩为摘要，保留关键决策
2. 移除已完成工具的原始结果，仅保留结论
3. 每 5 轮重申核心约束
4. 建议第 25 轮开新会话，传递上下文摘要
```

### 示例三：多 Agent 协作任务编排

**输入**："把'调研竞品并生成报告'这个任务拆解为多 Agent 协作流程"

**输出**（DAG 拆解 + 编排方案）：
```text
DAG：
[收集竞品列表] ──┬──> [抓取功能] ──┐
                 ├──> [抓取定价] ──┤──> [聚合分析] ──> [生成报告] ──> [质量校验]
                 └──> [抓取评价] ──┘

节点分配：
- [收集竞品列表]  Agent: 检索Agent  tokens: 2000  tools: [搜索引擎]  granularity: 0.8
- [抓取功能/定价/评价]  Agent: 爬虫Agent  tokens: 3000  tools: [浏览器]  granularity: 0.7（并行3节点）
- [聚合分析]  Agent: 分析Agent  tokens: 4000  tools: [计算器]  granularity: 0.75
- [生成报告]  Agent: 写作Agent  tokens: 3000  granularity: 0.8
- [质量校验]  Agent: 评审Agent  tokens: 1500  granularity: 0.9

全局质量门禁：每节点输出经 schema 校验，失败则重试→换Agent→转人工
数据传递 schema：{"competitors":[...],"features":{...},"analysis":{...}}
```

## 错误处理


| 场景 | 原因 | 处理方式 |
|------|------|----------|
| Agent 反复调用同一工具 | 工具结果未被正确解析 | 检查输出 schema，增加结果校验步骤 |
| 长会话输出质量骤降 | 上下文腐烂 | 运行三因素诊断器，应用治理建议 |
| 任务拆解后无法执行 | DAG 依赖关系有环 | 检查 DAG 是否有环，重新排序节点 |
| Prompt 评估分数低 | 缺少异常处理段 | 补充第 5 段异常处理，重评 |
| 工具调用参数错误 | 参数复杂度高未确认 | 启用调用前确认机制（中/高复杂度） |
| 输出 schema 校验失败 | LLM 输出不符合 schema | 执行ping命令测试网络连通性,检查防火墙和代理设置连接后重新执行命令 2 次（附错误反馈），仍失败转人工 |
| DAG 颗粒度评分异常 | 拆太细（<0.4）或太粗（>0.9） | 调整拆解粒度后重新评分至 0.6-0.8 |
| 幻觉检测未通过 | Prompt 缺事实性约束 | 补充"不确定时承认不知道"等 5 条约束 |

## 常见问题

**Q1：五段式 Prompt 会不会太长，浪费 token？**
A：五段式结构化反而更省 token。模糊 Prompt 会导致多轮澄清，结构化 Prompt 一次到位。实测平均节省 30% 总 token。

**Q2：上下文腐烂诊断需要多少轮会话才有意义？**
A：建议至少 10 轮。少于 10 轮时密度衰减不显著，诊断结果参考价值有限。

**Q3：任务拆解颗粒度评分怎么用？**
A：评分 0.6-0.8 为最佳区间。`< 0.4` 说明拆太细，编排开销大于执行开销；`> 0.9` 说明拆太粗，单节点失败影响大。可调整拆解粒度后重新评分。

**Q4：循环工程和传统 Prompt 工程能共存吗？**
A：可以。循环工程是 Prompt 工程的系统化升级，单节点内部仍用 Prompt 工程方法，循环工程负责节点间编排与质量保障（任务发现→分配→执行→质检→迭代）。

**Q5：幻觉检测能 100% 防止幻觉吗？**
A：不能。幻觉检测只能降低风险，通过约束+校验+来源标注把幻觉概率从约 15% 降到 3% 以下。完全消除需要结合 RAG 与事实校验工具。

## 已知限制

1. **幻觉检测非万能**：只能降低幻觉概率至 3% 以下，无法 100% 消除，需结合 RAG 与事实校验工具
2. **上下文腐烂诊断需≥10轮**：短会话诊断结果参考价值有限，密度衰减在少轮次时不显著
3. **DAG 颗粒度评分为启发式**：基于预估耗时/token/依赖复杂度/失败概率的经验公式，非精确算法，需人工复核
4. **不提供具体 LLM 调用代码**：本技能输出 Prompt 文本与设计方案，实际 LLM 调用由 Agent 平台执行
5. **工具选择矩阵覆盖常见场景**：极端边缘场景（如混合型任务）需人工判断，矩阵仅供参考
