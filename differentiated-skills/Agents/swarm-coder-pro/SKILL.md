### 2. 并行任务调度（专业版独有）

独立任务并行执行，提升开发速度：

```bash
# 分析任务依赖关系
swarm schedule analyze --plan docs/plans/feature-plan.md
# ...
# 输出依赖图：
# 任务A ──┬──> 任务C（依赖A）
#         └──> 任务D（依赖A）
# 任务B ──┼──> 任务E（依赖B）
#         └──> 任务F（依赖B）
#
# 可并行组：
# 组1：[任务A, 任务B]（无依赖，可并行）
# 组2：[任务C, 任务D, 任务E, 任务F]（A/B完成后可并行）
# ...
# 执行并行调度
swarm schedule execute --max-parallel 3
```

**调度特性**：
- 依赖图分析：自动识别任务间依赖关系
- 并行分组：无依赖的任务自动并行执行
- 负载均衡：round-robin / weighted两种策略
- 冲突检测：检测文件级冲突，避免并行修改

**输入**: 用户提供并行任务调度（专业版独有）所需的指令和必要参数。

### 4. 多层review（专业版独有）

五层review流水线，全面保证代码质量：

| 层级 | 审查内容 | 严重度阈值 | 修复循环 |
|---:|---:|---:|---:|
| 1. 规范合规 | 是否符合规范要求 | 任何不合规 | 实现→修复→重审 |
| 2. 代码质量 | 可读性/测试/错误处理 | Important+ | 实现→修复→重审 |
| 3. 架构一致性 | 模块边界/依赖/模式 | 任何架构异味 | 实现→修复→重审 |
| 4. 安全性 | 注入/认证/敏感信息 | 任何高危漏洞 | 实现→修复→重审 |
| 5. 性能 | 复杂度/查询/内存 | 🔴需优化 | 实现→修复→重审 |

**执行顺序**：1→2→3→4→5，每层通过后才进入下一层。Critical问题立即停止。

### 5. 自动修复循环（专业版独有）

失败自动重试，避免手动干预：

```text
自动修复流程：
1. review发现问题
2. 自动分派修复子代理（同一实现者）
3. 修复策略选择：
   - refactor：重构问题代码
   - simplify：简化复杂逻辑

5. 循环直到通过或达到最大次数（3次）

熔断机制：
- 同一问题修复3次未通过 → 熔断
- 熔断后冷却5分钟
- 冷却后降级为手动修复
```

### 7. 失败回滚（专业版独有）

任务失败自动回滚，保护已完成工作：

```bash
# 创建检查点
swarm checkpoint create --label "任务3开始前"
# ...
# 任务失败时自动回滚
swarm rollback auto --on-failure
# ...
# 手动回滚
swarm rollback to --checkpoint "任务3开始前" --confirm
# ...
# 幂等检查
swarm rollback idempotency-check --task task-3
```

**回滚特性**：
- 自动回滚：任务失败时自动回滚至上一检查点
- 检查点：每任务开始前自动创建
- 幂等保证：回滚后可安全重试
- 策略选择：git-revert / git-reset / 自定义

## 性能优化策略

### 并行调度优化

1. **依赖图缓存**：依赖分析结果缓存，避免重复计算
2. **智能分组**：基于任务相似性分组，提升缓存命中率
3. **负载预测**：基于历史数据预测任务耗时，优化调度
4. **冲突避免**：文件级冲突检测，避免并行修改

### 成本控制优化

1. **上下文压缩**：智能压缩任务上下文，减少Token消耗
2. **review分层**：低风险任务跳过架构/安全review
3. **缓存复用**：相似任务的review结果复用
4. **模型路由**：简单任务用GPT-4o-mini，复杂任务用GPT-4o

### 多层review优化

1. **并行review**：架构/安全/性能review可并行（无依赖时）
2. **增量review**：仅review变更部分，非全量
3. **提前终止**：Critical问题立即终止，不继续后续review
4. **结果缓存**：相同代码的review结果缓存

### 失败回滚优化

1. **增量检查点**：仅保存变更部分，非全量快照
2. **幂等设计**：任务设计为幂等，可安全重试
3. **快速回滚**：git-revert策略，秒级回滚
4. **状态清理**：回滚后自动清理部分完成的状态

### 成本控制

- 并行任务数限制（默认max=5），避免资源过度消耗
- review层按需启用（低风险任务跳过架构/安全review）
- 自动修复循环限制（默认max=3），避免无限循环
- 检查点按需创建（每任务开始前），避免存储浪费

## FAQ

### Q1：专业版与免费版的核心区别是什么？

专业版在免费版的串行执行+两阶段review基础上，新增六大高级功能：并行任务调度（依赖图+负载均衡）、成本预估与控制（Token预算+实时监控）、多层review（规范+质量+架构+安全+性能五层）、自动修复循环（失败重试+熔断）、多角色协作（架构师/安全/性能/测试专家）、失败回滚（检查点+幂等）。专业版使用GPT-4o模型路由，免费版使用GPT-4o-mini。

### Q2：并行调度如何避免任务冲突？

专业版采用文件级冲突检测：分析每个任务涉及的文件，如果两个任务修改同一文件，则串行执行；如果无文件重叠，则并行执行。可通过`.scheduler-config.json`的`conflictDetection`启用。

### Q3：成本预估准确吗？

成本预估基于历史数据与任务复杂度分析，准确率约80%。实际消耗可能因review循环次数、修复复杂度等因素偏离预估。建议设置80%预警阈值和90%硬停止，留有缓冲。

### Q4：五层review会拖慢开发吗？

会有些许影响，但专业版提供优化策略：低风险任务可跳过架构/安全review；架构/安全/性能review可并行执行（无依赖时）；review结果缓存复用。综合来看，五层review增加约30%时间，但减少约80%后期修复成本。

### Q5：自动修复循环会无限循环吗？

不会。专业版设置最大循环次数（默认3次），超过后触发熔断，冷却5分钟后降级为手动修复。熔断机制避免无限循环消耗资源。

### Q6：失败回滚会丢失已完成的工作吗？

不会。回滚仅回滚失败任务本身，不影响已完成任务。检查点创建在每任务开始前，回滚至该检查点即恢复到失败前的状态。已完成任务的工作保留。

### Q7：多角色协作如何避免职责冲突？

专业版明确角色边界：实现者负责编码，审查者负责review，不交叉。架构/安全/性能review可并行（无依赖时），但都依赖规范+质量review通过。测试专家可与架构review并行。

### Q8：并行调度支持多少个任务并行？

默认max=5，可配置至10。建议根据Agent平台资源限制调整。并行数过高可能导致资源竞争与性能下降。建议通过`swarm schedule optimize`自动推荐最优并行数。

### Q9：成本报告可以导出吗？

可以。使用`swarm cost export --format csv/json/markdown`导出成本数据。支持按任务/阶段/角色统计。导出数据可用于项目经理汇报或财务对账。

### Q10：检查点会占用多少存储？

检查点采用增量存储，仅保存变更部分。单个检查点通常<1MB。建议每任务开始前创建检查点，完成后可清理中间检查点，仅保留关键节点。

### Q11：专业版与免费版的数据可以互通吗？

可以。专业版完全兼容免费版的.swarm/目录结构，可无缝切换。从专业版降级到免费版时，并行调度、成本控制、多层review、失败回滚功能将停止，但基础的两阶段review继续可用。建议降级前导出成本报告和检查点作为备份。

### Q12：如何确保集群开发的安全性？

专业版提供多层安全保障：
- 分支隔离：在独立分支开发，不直接修改main/master
- 安全review：强制安全审查层，检测注入/认证/敏感信息
- 失败回滚：失败自动回滚，保护代码完整性
- 成本控制：硬停止机制，避免资源过度消耗
- 幂等设计：任务幂等，可安全重试
