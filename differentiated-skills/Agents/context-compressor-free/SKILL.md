## 核心能力
### 1. 智能混合提取

大多数记忆工具在面对非结构化日志时会失效，本工具采用三步混合策略：

**第一步 - 关键词匹配**：扫描标题，识别40+种中英文模式：

| 类别 | 中文模式 | 英文模式 |
|------|----------|----------|
| 事件 | 重大进展、突破、里程碑、决策 | breakthrough, milestone, decision |
| 教训 | 教训、反思、洞察、收获 | insight, takeaway, lesson |
| 成长 | 进化、改进、优化 | evolution, improvement |
| 待办 | 待办、紧急、重要 | todo, next step, urgent |

**第二步 - 兜底提取**：当无关键词匹配时（如基于时间的标题`## 08:44 站会`），自动提取所有章节的顶部条目。确保零数据丢失。

**第三步 - 混合模式**：对于多日合并文件，匹配章节用关键词提取，未匹配章节用兜底提取。两者共存，不丢弃任何内容。

### 2. 四大压缩原则

| 原则 | 含义 | 示例 |
|------|------|------|
| 去重复 | 提及一次即足够 | 不在3个章节重复"WebSocket重连" |
| 留转折 | 只保留变化点 | "从nginx切换到直连Node.js WSS" > 5段调试过程 |
| 去过程 | 结果重于过程 | "失败3次后用X修复" > 3段失败描述 |
| 留白 | 让读者自行推断 | 层级列表暗示关系，无需赘述 |

**输入**: 用户提供四大压缩原则所需的指令和必要参数。

## 使用场景

### 场景一：独立开发者的日志归档（开发者角色）

**痛点**：独立开发者使用AI Agent辅助开发，每日产生约2500字的对话日志，一个月后MEMORY.md膨胀至75000字，Agent读取缓慢且关键决策被淹没。

**解决方案**：
```bash
# 每周批量压缩本周日志
for file in memory/2026-03-{08..14}.md; do
    [ -f "$file" ] && node （请参考skill目录中的脚本文件） "$file" "/tmp/$(basename $file)"
done

# 合并压缩结果到长期记忆
cat /tmp/2026-03-*.md >> MEMORY.md
```

**效果**：7天日志从17500字压缩至约2200字，压缩比8:1。Agent读取MEMORY.md的速度提升，关键决策不再被过程描述淹没。

### 场景二：个人AI助手的记忆瘦身（个人用户角色）

**痛点**：个人使用AI助手管理日常事务，长期对话导致上下文窗口被旧日志占满，新对话响应变慢，Token成本逐月上升。

**解决方案**：
```bash
# 每月定期压缩
node （请参考skill目录中的脚本文件） memory/2026-02.md /tmp/feb-summary.md

# 审核后替换原始日志
mv /tmp/feb-summary.md memory/2026-02-compressed.md
```

**效果**：月度日志从75000字压缩至约10000字，Token消耗降低约60%，Agent响应速度显著提升。

### 场景三：学生项目协作的记忆整理（学生角色）

**痛点**：学生团队使用AI Agent协助完成学期项目，3个月下来日志累积过多，期末回顾时无法快速定位关键决策和技术选型原因。

**解决方案**：
```bash
# 压缩整个项目周期的日志
for file in memory/2026-{02,03,04}-*.md; do
    [ -f "$file" ] && node （请参考skill目录中的脚本文件） "$file" "/tmp/$(basename $file)"
done

# 生成项目决策摘要
cat /tmp/2026-*.md > PROJECT-DECISIONS.md
```

**效果**：3个月日志压缩为结构化决策文档，期末回顾时快速定位每个技术选型的原因和教训。

## FAQ

### Q1：压缩后会丢失重要信息吗？

不会。本工具采用"零关键信息丢失"原则，通过三步混合提取策略确保所有关键事件、决策、教训都被保留。压缩去除的是冗余重复、过程描述和无关细节，而非关键信息。建议首次使用时人工审核压缩结果，确认无误后即可放心使用。

### Q2：支持哪些格式的日志文件？

支持标准Markdown格式的日志文件，包括使用`#`、`##`、`###`标题层级的文件。对于无Markdown结构的纯文本文件，会给出友好提示。支持UTF-8编码，自动处理BOM头。非UTF-8编码文件会警告但继续处理。

### Q3：压缩比能达到多少？

平均压缩比为4-8倍，具体取决于原始日志的冗余程度。冗余越多，压缩比越高。对于高度结构化且无重复的日志，压缩比可能为2-3倍。对于充满重复过程描述的日志，压缩比可达10倍以上。

### Q4：如何集成到Agent的维护流程？

将压缩命令加入Agent的心跳维护周期（每2-3天执行一次）。本工具提供了可复制的维护模板，只需替换日期参数即可。建议在Agent规则中明确要求定期执行记忆维护。

### 已知限制

免费版支持单文件压缩、固定输出模板、基础关键词匹配。批量压缩需手动循环执行。不支持自定义模板、压缩质量评分、增量压缩等高级功能。解锁全部功能请使用专业版：context-compressor-pro。
