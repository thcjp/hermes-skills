# 密钥保险箱(专业版)

本工具为团队与企业提供全功能密钥安全管理能力，涵盖多保险箱管理、审计日志、密钥轮换与团队共享等高级特性，满足企业级安全合规要求。

## 概述

在企业环境中，密钥管理不仅是技术问题，更是合规与治理问题。专业版在免费版的本地密钥守护能力之上，构建了完整的密钥生命周期管理体系：多保险箱架构实现按项目隔离密钥；全量审计日志记录每一次密钥操作，满足SOC2、ISO27001等合规审计要求；自动化密钥轮换降低泄露风险；团队共享模块采用端到端加密，确保密钥在分发过程中安全可控。

专业版面向安全团队、DevOps工程师、合规审计人员等角色，提供从密钥创建到撤销的全流程治理能力。

## 使用场景

### 场景一：企业多项目密钥隔离
大型企业同时运行多个项目，每个项目使用不同的API密钥集合。通过多保险箱管理，为每个项目创建独立保险箱，密钥按项目隔离存储与访问。开发者切换项目时自动切换保险箱上下文，避免密钥混淆与误用。

### 场景二：安全合规审计
安全团队需要向审计方证明密钥操作的合规性。专业版的全量审计日志记录每次密钥验证、API代理、文件读取等操作的详细信息(时间、操作者、操作类型、目标密钥)，可一键生成符合SOC2要求的合规报告。

### 场景三：自动化密钥轮换
安全策略要求每90天轮换一次API密钥。专业版的密钥轮换功能支持定时任务，到期前自动提醒并辅助生成新密钥，更新后自动同步到所有使用该密钥的配置文件，降低人工操作遗漏风险。

### 场景四：团队密钥安全共享
新成员加入团队需要获取项目API密钥。通过团队共享模块，管理员将密钥加密分发给指定成员，成员的本地保险箱自动接收并存储。分发过程端到端加密，即使中间人截获也无法解密。成员离职时一键撤销访问权限。

### 场景五：CI/CD流水线密钥注入
DevOps团队需要在CI/CD流水线中安全注入API密钥。专业版提供与主流CI/CD系统的集成接口，流水线运行时从指定保险箱动态获取密钥，运行结束后自动清除，避免密钥残留在构建产物中。

## 快速开始

本工具属于复杂工具，预计180秒内可完成多保险箱配置与首次调用。

### 步骤1：安装与注册
```bash
# 克隆工具到本地
git clone https://example.com/key-vault-manager.git

# 复制环境变量示例文件
cp .env.example .env

# 在Agent中注册MCP工具
/mcp add key-vault-manager node /path/to/key-vault-manager/key-vault-manager.js
```

### 步骤2：创建多保险箱
```text
调用: create_vault({ name: "project-alpha", description: "Alpha项目密钥集合" })
返回: { vault_id: "vlt_001", name: "project-alpha", status: "active" }
```

### 步骤3：导入密钥到保险箱
```text
调用: import_keys({
  vault_id: "vlt_001",
  source: ".env.alpha",
  encrypt: true
})
返回: { imported: 5, encrypted: true, vault_id: "vlt_001" }
```

### 步骤4：配置密钥轮换策略
```text
调用: set_rotation_policy({
  vault_id: "vlt_001",
  key_name: "OPENAI_API_KEY",
  interval_days: 90,
  notify_before: 7
})
返回: { policy_id: "rot_001", next_rotation: "2026-10-16" }
```

### 步骤5：启用审计日志
```text
调用: enable_audit({ vault_id: "vlt_001", log_dir: "./audit-logs" })
返回: { enabled: true, log_format: "json" }
```

## 配置示例

### 多保险箱配置文件
```yaml
# vaults-config.yaml
vaults:
  - name: project-alpha
    description: Alpha项目密钥集合
    keys:
      - OPENAI_API_KEY
      - DEEPL_API_KEY
    rotation:
      interval_days: 90
      notify_before: 7
  - name: project-beta
    description: Beta项目密钥集合
    keys:
      - STRIPE_API_KEY
      - SENDGRID_API_KEY
    rotation:
      interval_days: 30
      notify_before: 3
audit:
  enabled: true
  log_dir: ./audit-logs
  format: json
  retention_days: 365
sharing:
  encryption: e2e
  default_expiry: 24h
```

## 最佳实践

### 实践一：按项目隔离保险箱
为每个项目创建独立保险箱，避免跨项目密钥混淆。开发者通过`switch_vault`命令切换上下文，确保操作针对正确的密钥集合。这一实践能显著降低误用风险，并简化权限管理。

### 实践二：分级轮换策略
根据密钥敏感度设置差异化轮换周期。支付类密钥(如Stripe)建议30天轮换；一般API密钥(如OpenAI)90天即可；低风险密钥可适当延长。配合到期提醒，确保轮换不遗漏。

### 实践三：审计日志的主动分析
不要仅将审计日志用于事后追溯，应定期主动分析。关注异常模式，如非工作时间的密钥访问、高频API代理调用、未知来源的验证请求等，这些可能是安全事件的早期信号。

### 实践四：团队共享的最小权限
分发密钥时遵循最小权限原则，仅共享成员实际需要的密钥。设置合理的过期时间(默认24小时)，过期后自动撤销访问。成员离职或角色变更时，立即执行权限回收。

### 实践五：CI/CD的动态注入
在CI/CD流水线中避免将密钥硬编码到构建配置。使用专业版的动态注入接口，运行时从保险箱获取密钥，构建完成后清除环境变量。结合审计日志可追溯每次构建使用的密钥版本。

## 常见问题

### Q1：多保险箱之间如何切换？
A：使用`switch_vault({ vault_id: "vlt_xxx" })`命令切换当前上下文。切换后所有密钥操作针对选中的保险箱，避免跨项目误操作。

### Q2：审计日志占用空间过大？
A：专业版支持日志压缩与归档。建议配置`retention_days`参数(默认365天)，过期日志自动归档压缩。也可定期导出日志到对象存储释放本地空间。

### Q3：密钥轮换失败怎么办？
A：轮换失败通常因为目标API拒绝新密钥或网络问题。专业版会保留旧密钥备份并发出告警。检查新密钥格式与API权限后，手动触发`retry_rotation`重试。

### Q4：团队成员如何接收共享密钥？
A：管理员通过`share_key`发起共享，成员的本地保险箱会收到加密包。成员需使用其私钥解密并确认接收。整个过程端到端加密，即使管理员也无法查看成员的私钥。

### Q5：共享密钥的过期时间如何管理？
A：发起共享时可设置`expiry`参数(默认24小时)。过期后访问权限自动撤销。如需延长，管理员可执行`extend_share`续期，但需重新确认成员身份。

### Q6：合规报告包含哪些内容？
A：合规报告包含审计日志摘要、密钥轮换记录、团队共享记录、异常操作告警等。支持JSON、CSV、PDF格式导出，适配不同审计方的要求。

### Q7：CI/CD集成支持哪些平台？
A：专业版提供通用API接口，支持GitHub Actions、GitLab CI、Jenkins等主流CI/CD平台。通过环境变量注入密钥，构建完成后自动清除。

### Q8：如何检测密钥泄露？
A：专业版的审计分析模块会检测异常模式，如高频调用、非工作时间访问等。结合密钥轮换策略，即使密钥泄露也能在下次轮换时自动失效。

### Q9：多保险箱的权限如何管理？
A：每个保险箱可设置独立的访问权限列表。管理员通过`grant_access`与`revoke_access`管理成员权限，支持读写分离与只读模式。

### Q10：专业版是否支持硬件密钥？
A：支持。专业版兼容YubiKey等硬件安全密钥，可作为保险箱主密钥的二级保护。硬件密钥丢失时可通过恢复流程重置。

| 现象 | 可能原因 | 解决方案 | 优先级 |

| 权限回收延迟 | 缓存未刷新 | 执行refresh_access强制刷新 | 中 |
| 硬件密钥认证失败 | 驱动或连接问题 | 检查硬件连接与驱动状态 | 高 |
