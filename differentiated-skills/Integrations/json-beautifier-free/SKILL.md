# JSON美化工具（免费版）

本工具提供JSON数据的格式化、压缩、验证与路径提取能力，帮助开发者快速整理JSON，提升可读性，并为下游处理提供标准化输出。

## 概述

JSON是现代API与配置文件的主流数据格式，但原始JSON往往以压缩形式传输，可读性差；或因手写编辑导致缩进混乱、语法错误。本工具围绕JSON的"整理"场景，提供四项核心能力：格式化（美化）、压缩（最小化）、验证（语法检查）、路径提取（JSONPath）。

## 核心能力

### 格式化（美化）
- 支持2空格、4空格、Tab三种缩进风格
- 自动处理嵌套结构的缩进对齐
- 保留或规范化字符串转义
- 支持Unicode字符的转义与原样输出两种模式

### 压缩（最小化）
- 移除所有非必要空白字符
- 保留字符串内部的合法空格
- 输出单行紧凑JSON，适合网络传输
- 压缩率指标反馈

### 验证（语法检查）
- 检测JSON语法错误并定位行列号
- 识别常见错误：尾随逗号、单引号、未转义字符
- 输出结构化错误报告
- 不修复错误，仅报告（修复能力在专业版）

### 路径提取
- 生成所有叶子节点的JSONPath路径
- 路径格式：`$.key.subkey[0].name`
- 支持按路径前缀过滤
- 输出路径与值的映射表

### 场景3：验证并提取路径

输入：

```json
{
  "formatted": "{\n  \"user\": {\n    \"name\": \"张三\",\n    \"age\": 30\n  },\n  \"tags\": [\n    \"dev\",\n    \"vip\"\n  ]\n}",
  "valid": true,
  "size": 78,
  "paths": [
    "$.user.name",
    "$.user.age",
    "$.tags[0]",
    "$.tags[1]"
  ]
}
```

## 配置示例

### 格式化参数表

| 参数 | 类型 | 默认值 | 可选值 | 说明 |
|------|------|--------|--------|------|
| `indent` | integer | 2 | 2/4/0 | 缩进空格数，0表示Tab |
| `sort_keys` | boolean | false | true/false | 是否按键名排序 |
| `ensure_ascii` | boolean | false | true/false | 是否转义非ASCII字符 |
| `compact` | boolean | false | true/false | 是否压缩为单行 |

### 输出结构字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `formatted` | string | 格式化后的JSON字符串 |
| `valid` | boolean | 是否通过语法验证 |
| `size` | integer | 格式化后字符数 |
| `paths` | array | 所有叶子节点的JSONPath列表 |

### JSONPath语法速查

| 表达式 | 含义 |
|--------|------|
| `$.key` | 根对象的key字段 |
| `$.key.subkey` | 嵌套字段 |
| `$[0]` | 数组首元素 |
| `$.key[*]` | 数组所有元素 |
| `$.key[0:2]` | 数组切片 |

## 最佳实践

### 缩进风格选择
- 团队协作推荐2空格（节省垂直空间）
- 教学演示推荐4空格（层级更清晰）
- 生产传输推荐压缩（节省带宽）
- 配置文件推荐保持原缩进（避免diff噪音）

### 键名排序策略
- 配置文件建议按字母排序，便于查找
- API响应不建议排序，保持业务语义顺序
- 缓存键建议排序，确保序列化稳定性

### Unicode处理
- 含中文的JSON建议`ensure_ascii=false`，提升可读性
- 跨系统传输建议`ensure_ascii=true`，避免编码问题
- 日志场景建议原样输出，便于人工阅读

### 验证与修复分离
- 本免费版仅验证不修复，确保操作可预测
- 修复能力由专业版提供，包含尾随逗号、单引号等自动修复
- 生产环境建议先验证再处理，避免静默错误

## 常见问题

### Q1：格式化后中文变成\uXXXX怎么办？
A：这是Unicode转义模式。将`ensure_ascii`设为`false`即可保留中文原样。默认值即为false，若出现转义说明参数被误改。

### Q2：验证报错但看不出哪里错了？
A：本工具的错误报告包含行列号。优先检查报错位置的前一个字符，常见原因是尾随逗号或单引号。专业版提供更详细的错误上下文与修复建议。

### Q3：路径提取漏了某些字段？
A：路径提取仅生成叶子节点（非对象、非数组的值）的路径。若某字段是对象或数组，其路径不会单独列出，但其子节点的路径会包含父级前缀。

### Q4：压缩后体积没明显减小？
A：若原JSON已是压缩格式或键值都很短，压缩空间有限。压缩率取决于原JSON的空白量与缩进深度。可查看size字段对比前后大小。

### Q5：键名排序后业务语义混乱？
A：排序是为缓存键稳定性设计，不适用于所有场景。API响应与业务文档建议保持原序，仅在需要哈希一致性时启用排序。
