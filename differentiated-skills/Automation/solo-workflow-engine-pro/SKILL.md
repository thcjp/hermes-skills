## 核心能力
### Step 1：识别自动化机会

并非所有任务都适合自动化。先找到最高价值的机会.
**自动化审计（花1小时）**：

1. 追踪你一周内做的每件事（用记事本或简单表格）
2. 对每件事记录：
   - 耗时多少
   - 频率如何（每日/每周/每月）
   - 是重复的还是需要判断
3. 计算每个任务的时间成本：

```text
时间成本 = (单次耗时分钟 × 月频率) / 60
# ...
示例：15分钟的任务 × 每月20次 = 每月5小时
```

4. 按时间成本从高到低排序

**适合自动化的任务特征**：

| 特征 | 适合自动化 | 不适合自动化 |
|:-----|:-----|:-----|
| 重复性 | 每次步骤相同 | 每次都不同 |
| 规则性 | 规则驱动，无需判断 | 需要复杂判断 |
| 频率 | 每日或每周 | 每月1次以下 |
| 耗时 | 10分钟以上 | 2分钟以下 |

**自动化机会示例**：

| 适合自动化 | 不适合自动化 |
|---:|---:|
| 给客户发周报（固定格式、固定时间） | 做客户访谈（需要细腻判断） |
| 付款后自动创建发票 | 写定制方案（需要创造力） |
| 表单提交自动加入CRM | 做产品决策（需要综合考虑） |
| 定时发布社交媒体内容 | 回答客户咨询（需要理解上下文） |

**低垂果实清单（从这里开始）**：

- 表单提交自动邮件通知
- 表单回复自动保存到表格
- 提前排期社交媒体发布
- 付款确认后自动创建发票
- 工具间数据同步（CRM ↔ 邮件工具 ↔ 表格）

**输入**: 用户提供Step 1：识别自动化机会所需的指令和必要参数.
**处理**: 解析Step 1：识别自动化机会的输入参数,完成核心逻辑,返回结构化响应.
**输出**: 返回Step 1：识别自动化机会的响应数据,包含状态码、结果和日志.

### Step 3：设计工作流

在构建之前，先在纸上或白板上画出工作流.
**工作流设计模板**：

```text
触发器：什么事件启动工作流？
  示例："Google表格新增一行"
# ...
条件（可选）：每次都执行，还是满足条件才执行？
  示例："仅当状态列 = '已批准'"
# ...
操作：执行什么动作？
  步骤1：[动作]
  步骤2：[动作]
  步骤3：[动作]
# ...
错误处理：某步失败时怎么办？
  示例："发Slack消息通知我"
```

**示例工作流（线索收集 → CRM → 邮件）**：

```text
触发器：网站新表单提交
# ...
条件：邮箱字段不为空
# ...
操作：
  步骤1：添加线索到CRM（如Airtable或HubSpot）
  步骤2：通过邮件工具发送欢迎邮件（如ConvertKit）
  步骤3：在项目管理工具创建跟进任务（如Notion），3天后跟进
  步骤4：发Slack通知："新线索：[姓名]"
# ...
错误处理：如果步骤1失败，发邮件警报给我
```

**设计原则**：

- 保持简单——从2-3步开始，以后再加复杂度
- 每步单独测试后再串联
- 需要时在操作间加延迟（有些API较慢）
- 必须包含错误通知，这样出问题时你会知道

**处理**: 解析Step 3：设计工作流的输入参数,完成核心逻辑,返回结构化响应.
**输出**: 返回Step 3：设计工作流的响应数据,包含状态码、结果和日志.
### Step 4：构建与测试

在你选的工具中实现工作流.
**构建流程（Zapier示例）**：

1. 选择触发器应用（如Google Forms、Typeform、网站表单）
2. 连接你的账户（OAuth认证）
3. 测试触发器（提交一个测试表单，确认数据能过来）
4. 添加操作（如"添加行到Google表格"）
5. 映射字段（匹配表单字段到表格列）
6. 测试操作（运行测试，确认行被正确添加）
7. 重复添加更多操作
8. 开启工作流（Zapier叫"turn on Zap"）

**测试清单**：

- [ ] 通过触发器提交测试数据
- [ ] 验证每个操作正确执行
- [ ] 检查数据映射到正确字段
- [ ] 用边界情况测试（空字段、特殊字符、长文本）
- [ ] 测试错误处理（故意制造失败，看告警是否工作）

**常见问题与修复**：

| 问题 | 原因 | 修复 |
|:------|------:|:------|
| 工作流不触发 | 触发条件太窄 | 检查过滤设置，放宽条件 |
| 操作失败 | API限流或权限不足 | 在操作间加延迟，重新认证 |
| 数据缺失或错误 | 字段映射错误 | 仔细检查哪些字段被映射 |
| 工作流执行多次 | 重复触发 | 基于唯一ID去重 |

**铁律**：用真实数据测试后再依赖自动化。不要等真实客户遇到了才发现bug.

**输出**: 返回Step 4：构建与测试的响应数据,包含状态码、结果和日志.
### Step 5：维护与监控 — 专业版独有

自动化不是"设置完就忘了"。它们会坏。工具会变。API会更新。你需要一个维护计划.
**每周检查（5分钟）**：

- 扫描工作流日志，检查是否有错误（大多数工具会显示运行记录+失败记录）
- 立即处理任何失败

**月度审计（15分钟）**：

- 审查所有活跃的工作流
- 检查：这个还在用吗？还在节省时间吗？
- 禁用或删除不用的工作流（它们会占用仪表板，造成混乱）
- 更新任何依赖你已切换工具的工作流

**工作流文档存储**：

- 为每个工作流创建一个简单文档（Notion、Google Doc）
- 包含：做什么、何时运行、连接哪些应用、如何排障
- 如果你有10+个工作流，这份文档在出问题时会帮你节省数小时

**错误处理配置**：

- 将所有错误通知路由到一个地方（Slack频道、邮件收件箱或任务管理器）
- 设置："如果任何工作流失败，发消息到[你的错误频道]"
- 每周审查错误并修复根本原因

**维护日志模板**：

```text
工作流名称：表单线索自动入库
创建日期：2026-01-15
最后检查：2026-02-01
状态：运行正常
# ...
本月运行次数：47次
本月失败次数：2次（均为API限流，已自动重试成功）
# ...
连接的应用：
  - Typeform（触发器）
  - Airtable（CRM）
  - ConvertKit（邮件）
  - Slack（通知）
# ...
排障指南：
  1. 如果不触发：检查Typeform表单是否活跃
  2. 如果Airtable失败：检查API令牌是否过期
  3. 如果邮件未发送：检查ConvertKit订阅状态
```

**处理**: 解析Step 5：维护与监控 — 专业版独有的输入参数,完成核心逻辑,返回结构化响应.
**输出**: 返回Step 5：维护与监控 — 专业版独有的响应数据,包含状态码、结果和日志.

### Step 7：计算自动化ROI — 专业版独有

并非每个自动化都值得投入时间。计算ROI来决定优先级.
**ROI公式**：

```text
月节省时间(小时) = (单次耗时分钟 / 60) × 月频率
月节省价值(元) = 月节省时间 × 你的时薪
建设成本(元) = 建设耗时(小时) × 时薪 + 工具月费
回收周期(月) = 建设成本 / 月节省价值
# ...
判断标准：
  回收周期 < 3个月  → 值得做
  回收周期 > 6个月  → 可能不值得（除非有其他价值）
```

**示例计算**：

```text
任务：手动复制表单提交到CRM
  单次耗时：15分钟
  月频率：20次
  你的时薪：200元/小时
  建设耗时：1小时
  工具月费：$20/月（约140元）
# ...
计算：
  月节省时间 = (15/60) × 20 = 5小时
  月节省价值 = 5 × 200 = 1000元
  建设成本 = 1 × 200 + 140 = 340元
  回收周期 = 340 / 1000 = 0.34个月
# ...
结论：回收周期<1个月 → 绝对值得 ✅
```

**ROI优先级矩阵**：

| 回收周期 | 建议 | 优先级 |
|---:|:---|---:|
| <1个月 | 立即做 | 最高 |
| 1-3个月 | 值得做 | 高 |
| 3-6个月 | 可以做 | 中 |
| >6个月 | 暂不做 | 低 |

**铁律**：聚焦回收周期<3个月的自动化，那是你最高杠杆的投资.

## 自动化常见错误

### 错误1：先自动化再优化流程

**问题**：自动化一个糟糕的流程，只是让糟糕的流程跑得更快.
**正确做法**：先优化流程，再自动化。如果手动做都很痛苦，自动化后只会更痛苦.
### 错误2：过度自动化

**问题**：试图自动化一切，包括罕见或需要判断的任务.
**正确做法**：不是所有事情都需要自动化。如果任务罕见或需要判断，手动做。聚焦高频、规则驱动的任务.
### 错误3：没有错误处理

**问题**：自动化坏了你却不知道，造成静默失败.
**正确做法**：始终设置错误告警。将所有错误通知路由到一个地方，每周审查.
### 错误4：测试不充分

**问题**：一个坏掉的自动化比没有自动化更糟——它会产生错误数据或遗漏任务.
**正确做法**：用真实数据彻底测试，包括边界情况。不要等真实客户遇到了才发现问题.
### 错误5：太快变得复杂

**问题**：一开始就建10步工作流，出问题时无法定位.
**正确做法**：从简单的2-3步开始，简单版本完美运行后再加复杂度.
### 错误6：不写文档

**问题**：未来的你会忘记这个工作流怎么工作的.
**正确做法**：为每个工作流写文档，包括做什么、何时运行、连接哪些应用、如何排障.

## FAQ

### Q1：专业版包含哪些核心能力？

专业版解锁全部七步完整框架（识别、选择、设计、构建、维护、高级场景、ROI计算），新增维护监控（Step 5）、四种高级场景（Step 6：客户入驻/内容分发/客户健康/发票追踪）和ROI计算（Step 7）。专业版还提供常见错误清单（6项）、完整故障排查表（8项）、多角色场景指南（7种角色）和维护计划模板.
### Q2：自动化后如何维护？

专业版建议建立维护计划：每周花5分钟扫描工作流日志，处理失败；每月花15分钟审计所有活跃工作流，检查是否还在用、是否还在节省时间。为每个工作流创建文档，记录做什么、何时运行、连接哪些应用、如何排障。将所有错误通知路由到一个地方，每周审查.
### Q3：如何计算自动化是否值得？

专业版提供ROI计算公式：月节省时间=（单次耗时/60）×月频率；月节省价值=月节省时间×时薪；回收周期=建设成本/月节省价值。回收周期<3个月值得做，>6个月暂不做。用数据驱动决策，而非凭感觉.
### Q4：高级场景和基础场景有什么区别？

基础场景是2-3步的简单工作流（如表单→CRM、付款→发票）。高级场景是6步以上的复杂工作流（如客户入驻全流程、内容多平台分发、客户健康监控）。高级场景通常涉及4个以上应用的编排，需要更复杂的错误处理和条件判断.
### Q5：什么任务不该自动化？

不该自动化的任务特征：罕见（每月不到1次）、需要创造性判断（如写定制方案）、需要人际互动（如客户访谈）、需要综合考虑（如产品决策）。这些任务手动做更好，自动化反而会增加复杂度.
### Q6：工具之间API变更导致工作流失效怎么办？

这是自动化维护的核心挑战。专业版建议：每周检查工作流日志，及时发现API变更导致的问题。关注你使用的工具的变更通知。为关键工作流准备手动备份方案。使用支持自动重试的工具（如Make）。在维护文档中记录每个工作流依赖的API版本.
### Q7：n8n自托管和Zapier哪个更划算？

n8n自托管无月费，但需要技术能力来部署和维护。Zapier每月$20-50，但零维护。如果你有技术背景且工作流数量多（10+），n8n自托管更划算。如果你不想维护基础设施，Zapier更省心。独立开发者建议从Zapier开始，当月费超过$50时考虑迁移到n8n.
### Q8：如何开始使用专业版？

无需迁移数据。专业版包含完整的四步基础框架和场景示例，并在此之上扩展了维护计划（Step 5）、ROI计算（Step 7）和高级场景（Step 6）。开始使用后建议对已有的自动化工作流补充维护计划（Step 5），并计算其ROI（Step 7）验证是否值得继续维护。可以逐步尝试高级场景，扩展自动化覆盖范围.
### Q9：可以同时管理多个工作流吗？

可以，但需要系统化管理。专业版建议：为每个工作流编号并维护文档，每月审计时检查所有工作流的运行状态。使用统一的错误通知渠道。超过10个工作流时，建议用项目管理工具（如Notion）建立工作流清单，记录每个工作流的状态、ROI和最后检查日期.
### Q10：如何避免工作流之间相互干扰？

确保工作流之间没有隐式依赖。如果工作流A的输出是工作流B的输入，应在设计时明确这种依赖关系，并在B中添加等待逻辑或重试机制。避免多个工作流同时写同一个目标（如同一个Google表格），可能造成数据冲突。使用唯一ID确保数据不重复.
## 故障排查表

| 问题 | 可能原因 | 解决方案 | 优先级 |
|----|----|----|----|
| 工作流不触发 | 触发器配置错误或源应用无新数据 | 检查触发器设置；确认源应用有新事件产生 | 高 |
| 操作执行失败 | API限流、权限不足或认证过期 | 添加延迟；重新认证；检查API配额 | 高 |
| 数据映射错误 | 字段名变更或格式不匹配 | 检查字段映射；确认源字段格式 | 高 |
| 工作流执行多次 | 重复触发或缺少去重 | 基于唯一ID去重；检查触发器配置 | 中 |
| 错误通知未收到 | 通知渠道配置错误 | 检查通知设置；测试通知是否可达 | 中 |
| 工作流变慢 | 数据量增大或API响应慢 | 分批处理；添加缓存；检查API性能 | 中 |
| 工具变更导致失效 | 第三方工具API更新 | 关注工具变更通知；及时更新工作流 | 中 |
| 工作流过于复杂 | 步骤过多难以维护 | 拆分为多个简单工作流；减少步骤 | 低 |
