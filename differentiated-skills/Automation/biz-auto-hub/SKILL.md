# 商业自动中枢

把AI Agent变成商业自动化架构师。识别吞噬时间的手动流程、设计自动化、落地实现、量化收益、持续监控。核心信条：**先证明该自动化，再动手；先证明自动化还活着，再放手。**

## 三大痛点与对策

| 痛点 | 典型表现 | 本skill对策 |
|:-----|:---------|:------------|
| ROI难量化 | "感觉省时间但说不清省多少" | 准入评分卡 + ROI量化器（带真实数字） |
| 静默故障 | 自动化挂了三周才被发现 | 强制告警路由 + 健康度周报 |
| 过度自动化 | 把一年一次的任务也自动化了 | 5x准入规则 + 评分卡过滤 |

## 第一步：自动化准入（先决策再做）

### 5x准入规则

只自动化满足以下任一条件的流程：
- 每周至少发生5次
- 单次耗时超过30分钟

否则自动化成本 > 手工成本，得不偿失。

### 准入评分卡

对每个候选流程按五维打分（每维0-3分）：

| 维度 | 0分 | 1分 | 2分 | 3分 |
|:-----|:----|:----|:----|:----|
| 频次 | 月级 | 周级 | 日级 | 每日多次 |
| 单次耗时 | <5分钟 | 5-15分钟 | 15-60分钟 | >1小时 |
| 错误影响 | 仅外观 | 需返工 | 客户可见 | 影响营收 |
| 流程复杂度 | 5+决策点 | 3-4决策点 | 1-2决策点 | 纯规则 |
| 集成系统数 | 4+系统 | 3系统 | 2系统 | 1系统 |

**决策阈值：**
- 12-15分：立即自动化，最高ROI
- 8-11分：下个迭代规划
- 4-7分：考虑部分自动化
- 0-3分：跳过，手工即可

## 第二步：工作流设计

### 工作流架构模板（YAML）

```yaml
workflow:
  name: "发票自动处理"
  id: "invoice-auto"
  version: "1.0"
  trigger:
    type: email
    config:
      inbox: "finance@company.com"
      filter: "主题包含'发票' AND 有附件"
  inputs:
    - name: attachment
      type: file
      required: true
  steps:
    - id: extract
      action: transform
      config: { parser: "ocr_invoice" }
      on_success: validate
      on_failure: notify_human
      timeout: 30s
      retry: { max_attempts: 3, backoff: exponential }
    - id: validate
      type: condition
      rules:
        - condition: "amount <= 5000"
          goto: auto_approve
        - condition: "default"
          goto: manager_review
  error_handling:
    - id: notify_human
      action: notify
      config:
        channel: slack
        message: "工作流[invoice-auto]在{failed_step}失败：{error}"
      then: human_review
  outputs:
    - name: payment_record
      destination: 财务系统
      format: json
  monitoring:
    success_metric: "成功率>99%"
    alert_threshold: "连续3次失败"
```

### 六类高频工作流模板

#### 1. 线索入站处理
```
触发：表单提交/邮件/聊天
  → 去重校验
  → 富化（公司规模、行业）
  → 评分（0-100按ICP匹配度）
  → 路由：
    - 80+分：即时Slack告警+日历链接
    - 40-79分：进入培育序列
    - <40分：自动回复资源
  → 写入CRM
  → 更新仪表盘
```

#### 2. 发票与付款处理
```
触发：收到发票（邮件附件/上传）
  → 提取数据（供应商、金额、明细、到期日）
  → 匹配PO/预算科目
  → 校验：
    - 金额在审批范围 → 自动批准
    - 超阈值 → 路由给经理
    - 无匹配PO → 标记复核
  → 按账期安排付款
  → 更新财务系统
  → 发送付款确认
```

#### 3. 员工入职
```
触发：offer签署完成
  → 创建账号（邮箱、聊天工具、代码仓库）
  → 加入团队与频道
  → 生成欢迎包
  → 排定首日会议（经理1:1、IT、HR、团队午餐）
  → 分配入职清单
  → 设置30/60/90天提醒
  → 通知用人经理"X日一切就绪"
```

#### 4. 报告生成与分发
```
触发：每周一8:00
  → 从多源拉取数据（数据库、API、表格）
  → 计算KPI vs 目标
  → 检测异常（偏离均值>2标准差）
  → 生成格式化报告
  → 添加变化点评
  → 分发：
    - 高管摘要 → 领导层聊天群
    - 完整报告 → 邮件给干系人
    - 异常告警 → 运维群
  → 归档
```

#### 5. 客服升级
```
触发：新工单
  → 分类（账单/技术/功能/缺陷）
  → 查客户层级（企业/专业/免费）
  → 检索知识库
  → 可自动解决：
    - 发送方案+"是否解决？"
    - 24小时无回复 → 关闭
  → 不可自动解决：
    - 按分类路由专家
    - 按层级设SLA计时
    - SLA达80% → 升级组长
    - SLA违约 → 告警经理+更新客户
```

#### 6. 内容发布流水线
```
触发：内容标记"待审核"
  → 质量检查（语法、SEO、链接）
  → 路由审核者
  → 通过：
    - 按平台格式化（博客、社交、邮件）
    - 按内容日历排期
    - 配置UTM追踪
    - 准备社交放大队列
  → 驳回：通知作者+48小时提醒
  → 发布后24小时：采集互动数据→更新表现追踪表
```

## 第四步：静默故障防护网

> 自动化最大的风险不是失败，是失败了没人知道。

### 强制告警路由矩阵

| 故障类型 | 告警通道 | 响应时限 |
|:---------|:---------|:---------|
| 临时错误（超时、限流） | 日志记录 | 自动重试 |
| 持续错误（连续3次失败） | Slack告警 | 4小时内人工介入 |
| 数据缺失（应到未到） | 邮件+Slack | 当日处理 |
| 凭证失效（401/403） | 紧急Slack | 1小时内 |
| 业务影响（客户可见） | 电话/短信 | 立即 |

## 第五步：ROI量化器

### 公式

```
月度ROI = (节省小时数 × 时薪) - 自动化成本

节省小时数 = 频次 × 单次耗时 × 成功率
时薪 = 员工成本 / 工作小时数
自动化成本 = 工具费用 + 维护小时数 × 时薪
```

### 真实示例

```
流程：发票处理
改造前：50张/周 × 12分钟 = 10小时/周 = 40小时/月
改造后：50张/周 × 1分钟复核 = 0.83小时/周 = 3.3小时/月
节省：36.7小时/月
按$50/小时：月省$1835
自动化成本：2小时/月维护 × $50 = $100/月
净ROI：$1735/月 = $20820/年
回本周期：建设投入10小时 × $50 / $1735 ≈ 0.29个月
```

### 准入门槛

- 回本周期 < 3个月 → 值得做
- 3-6个月 → 视战略价值
- > 6个月 → 不值得（除非解锁其他价值）

## 第六步：高级模式

### 凭证保险柜模式

```yaml
credential_vault:
  location: "${HOME}/.automation-vault/"  # 永不硬编码在脚本里
  structure:
    - name: slack_webhook
      type: webhook
      rotated: "2026-06-01"
      next_rotation: "2026-09-01"
    - name: finance_api
      type: oauth
      refresh_token_encrypted: true
  rules:
    - 所有脚本通过环境变量引用，不写入代码
    - 每90天轮换一次
    - 失效告警路由到紧急通道
```

### 人在回路审批门

```yaml
approval_gate:
  name: "经理审批"
  trigger: "amount > 5000 OR new_vendor = true"
  action:
    - 通过Slack/邮件发送审批请求
    - 包含：摘要、金额、上下文、批准/拒绝按钮
    - 截止时间：24小时
  on_approve: continue_workflow
  on_reject: notify_requestor_with_reason
  on_timeout:
    - 升级到上级
    - 或：金额<10000时自动通过
```

### 优雅降级五级

```
L1 重试（瞬时错误：超时、限流）
L2 回退（用缓存数据、备用API、简化逻辑）
L3 排队（服务恢复后再处理）
L4 告警（通知人工，提供上下文与建议修复）
L5 安全停止（保留状态、不丢数据、停止工作流）
```

### 多系统同步策略

- 每类数据指定唯一"真相源"系统
- 同步方向：源 → 副本（避免双向）
- 必须双向时：用冲突解决规则（最后写入胜出、人工合并）
- 每周对账：对比系统、标记不一致

## 边界情况与陷阱

- **时区混乱**：内部统一存UTC，仅展示时转换。测试夏令时切换前后。
- **API限流**：跟踪调用计数，实现退避，尽量批量请求，缓存响应。
- **部分失败**：5步中第3步失败，能从第3步恢复吗？设计幂等性。
- **数据增长**：100条能跑，10000条就崩。提前规划分页、分块、归档。
- **凭证轮换**：API换Key。为鉴权失败建告警，避免一切崩了才知道。
- **Schema变更**：外部API加减字段。防御性校验输入，遇意外数据不崩溃。
- **重复处理**：用幂等键。处理前先查"是否已处理"。付款、邮件尤其重要。
- **测试自动化**：始终用真实但安全的数据测试。发邮件、扣款、改生产数据的操作必须有dry-run模式。

## FAQ

**Q：自动化做完后多久能回本？**
A：用ROI量化器算。回本<3个月值得做，>6个月通常不值得。示例中发票处理回本约0.29个月。

**Q：自动化经常静默失败怎么办？**
A：上"强制告警路由矩阵"——临时错误重试、持续错误Slack告警、凭证失效紧急告警。每周跑健康度周报。

**Q：哪些任务不该自动化？**
A：用5x规则过滤——每周<5次且单次<30分钟的不值得。还需人工判断的、一年才一次的、需要创造力的都不该自动化。

**Q：凭证散落各脚本里怎么管？**
A：上凭证保险柜模式——统一目录、环境变量引用、90天轮换、失效告警。

## 故障排查

| 症状 | 可能原因 | 解决 |
|:-----|:---------|:-----|
| 工作流不触发 | 触发条件太窄 | 放宽过滤条件，检查webhook签名 |
| 步骤间歇失败 | API限流或超时 | 加延迟、加退避重试 |
| 数据字段错位 | 字段映射变了 | 双向校验映射，加schema版本号 |
| 重复执行多次 | 重复触发 | 用幂等键去重 |
| 突然全部失败 | 凭证过期 | 检查Token过期时间，启用轮换告警 |
| 健康度周报缺失 | cron未启用 | 检查cron.enabled配置与Gateway运行状态 |
