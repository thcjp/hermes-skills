# 自动驾驶流程（Autopilot Flow）

把重复任务变成自动驾驶。核心信条：**不是"回答问题"，而是"看到重复→立刻构建自动化"。**

> 触发思维：遇到重复操作时，问自己——能否写成脚本？下次能否自动跑？能就直接做，不等用户要求。

观察（重复日志法）→抽象（模式提取）→设计（流程图+触发器）→实现（脚本/无代码配置）→测试（dry-run验证）→优化（指标迭代），每步有明确动作与产物

## 适用场景

**何时使用**：
- 每周手动做周报（花30分钟整理数据发邮件）
- 每天清洗相似结构的数据（去重、补缺失、标准化）
- 下载目录天天堆满待整理文件
- 任何"这事我做过三次了"的重复任务
- Agent 观察到用户重复操作时主动提议自动化

**输入**：重复任务描述 / 重复日志（含频次、耗时、步骤）/ 待自动化的原始数据或文件
**输出**：可执行脚本（Python/Shell）+ cron配置 / 无代码平台配置方案 / 健康度看板（YAML）

**不适用场景**：
- 罕见或需人工判断的一次性任务（不值得自动化）
- 流程本身有问题未优化的任务（先修流程再自动化）
- 涉及高度创意性/情感判断的任务（无法模式化）

## 使用流程

### Step 1：观察（用户在重复做什么？）
维护"重复日志"一周，每发生一次重复任务记录：
```yaml
repeat_log:
  task: "把销售群消息整理成日报"
  time: "2026-07-18 18:30"
  duration_min: 15
  steps: [翻群消息, 复制关键信息, 贴到表格, 发邮件]
```
一周后按 `频次 × 单次耗时` 降序排序。**产物**：重复任务清单。

### Step 2：抽象（这个任务的模式是什么？）
把多次实例的步骤对齐，找共性，标记固定步骤（可自动化）与人工步骤（保留），提取输入/输出契约：
```yaml
pattern:
  name: "销售日报生成"
  input: { source: "销售群消息", period: "当日" }
  fixed_steps: [翻消息, 提取关键信息, 贴表格]
  human_steps: [审核异常项]
  output: { format: "邮件+表格", recipient: "管理层" }
```
**产物**：任务模式定义。

### Step 3：设计（怎么自动完成？）
画流程图（文字版），标注触发器、步骤、错误处理：
```text
触发：每日18:00
  → 调聊天API拉当日销售群消息
  → 正则提取关键信息（客户、金额、状态）
  → 写入表格 → 生成邮件正文 → 发送邮件
  → 异常项标记"需人工审核"另发通知
```
确定触发方式：定时（cron）/事件（webhook）/手动。**产物**：流程图+步骤定义。

### Step 4：实现（写脚本/配置）
按复杂度选实现方式：简单2-3步→无代码平台（Zapier/Make）；复杂逻辑→Python/Node脚本；需定时→cron+脚本。写脚本骨架并配置 cron：
```bash
# 每日18:00跑
0 18 * * * /usr/bin/python3 /path/to/daily_report.py >> logs/daily.log 2>&1
```
**产物**：可执行脚本 + cron配置。

### Step 5：测试（小范围验证）
启用 dry-run 模式（不发真实邮件、不写生产表格），用过去一周真实数据跑一遍，人工核对输出，测试错误处理（故意制造失败看告警），测试幂等性（同一天跑两次结果一致）。**产物**：测试报告+修复后脚本。

### Step 6：优化（迭代改进）
上线后跟踪指标（成功率/耗时/人工干预），每周复盘，发现模式优化（某客户消息提取错→加白名单；邮件判垃圾→换服务商；耗时增长→加分页）。**产物**：优化后脚本+迭代记录。

## 示例

### 示例

**输入**："每周五花30分钟手工整理本周数据发周报，帮我自动化"

**输出**（端到端方案）：
```text
触发：每周五17:00（cron: 0 17 * * 5）
设计：
  → 从项目管理工具拉本周完成任务
  → 从CRM拉本周新增线索
  → 计算KPI（完成率、转化率）
  → 套用周报模板 → 发邮件给干系人
实现：Python脚本 + cron
测试：用上周数据dry-run，核对数字
优化：某KPI总算错→修公式；邮件判垃圾→加SPF记录
```
**效果**：每周省30分钟，成功率目标>99%。

### 示例二：数据清洗自动化

**输入**："每天手工清洗上传的CSV，去重、补缺失、标准化格式"

**输出**（Python脚本核心逻辑）：
```python
import pandas as pd
def clean_csv(input_path, output_path):
    df = pd.read_csv(input_path)
    df = df.drop_duplicates(subset=['id'])          # 按主键去重
    df['amount'] = df['amount'].fillna(df['amount'].mean())  # 数值列补均值
    df['category'] = df['category'].fillna('未知')   # 文本列补默认
    df['date'] = pd.to_datetime(df['date'], format='ISO8601')  # 日期标准化
    df.to_csv(output_path, index=False)
# cron: 0 9 * * * python3 clean.py >> logs/clean.log 2>&1
```
**效果**：每日省20分钟，准确率98%+。

### 示例三：文件整理自动化

**输入**："下载目录天天堆满文件，每周手工整理一次，能自动化吗？"

**输出**（watch 监控方案）：
```python
from watchdog.observers import Observer
from watchdog.events import FileSystemEventHandler
import os, shutil, time

class FileOrganizer(FileSystemEventHandler):
    def on_created(self, event):
        if event.is_directory: return
        ext = os.path.splitext(event.src_path)[1].lower()
        # 按扩展名分类
        target = {'jpg/png/gif': 'Pictures', 'pdf/docx': 'Documents',
                  'zip/rar': 'Archives'}.get(ext, 'Others')
        dest = f"~/Downloads/{time.strftime('%Y-%m')}/{target}/"
        os.makedirs(dest, exist_ok=True)
        shutil.move(event.src_path, dest + os.path.basename(event.src_path))

observer = Observer()
observer.schedule(FileOrganizer(), '~/Downloads/', recursive=False)
observer.run()  # 常驻进程
```
**触发方式**：watch 监控（事件驱动，非 cron）；**优化**：重复文件名加序号，归类错加规则。

## 错误处理


| 场景 | 原因 | 处理方式 |
|------|------|----------|
| 自动化不触发 | cron未配置/进程未启动 | 检查cron配置与常驻进程状态 |
| 间歇性失败 | API限流/网络超时 | L1执行ping命令测试网络连通性,检查防火墙和代理设置连接后重新执行命令（指数退避3次）+ 加延迟 |
| 输出错乱 | 数据源schema变更 | 加schema校验，dry-run对比基线 |
| 耗时变长 | 数据量增长 | 改流式/分块处理，加分页 |
| 静默失败 | 无错误告警 | 必加L4告警到Slack/邮件，禁止静默 |
| 重复执行 | 无幂等键 | 按唯一ID去重，保证幂等性 |
| 服务不可用 | 外部依赖宕机 | L2回退用缓存数据/备用API/简化逻辑 |
| 无法自动恢复 | 严重故障 | L5安全停止，保留状态，等待人工介入 |

## 常见问题

**Q1：怎么判断一个任务该不该自动化？**
A：用"三次规则"——做过三次以上的重复任务就值得。再算ROI：每周频次 × 单次耗时 > 1小时就值得自动化。

**Q2：自动化上线后坏了怎么办？**
A：必须有错误告警（L4）+ 状态保存（L5）。坏了告警通知人工，状态不丢，修复后能从断点继续。

**Q3：不会写脚本能自动化吗？**
A：简单2-3步用无代码平台（Zapier/Make）。复杂逻辑才需要脚本。先从无代码起步，验证模式后再升级为脚本。

**Q4：自动化越攒越多管不过来？**
A：每周巡检5问（都跑了吗？有失败吗？人工干预几次？耗时变长了吗？有新重复任务吗？）。月度审计禁用未使用的自动化，每个配健康度看板。

**Q5：Agent 怎么主动发现重复？**
A：用触发思维——用户第三次问类似问题、第三次粘贴相似数据、说"又来一遍"时，主动提议自动化。

## 已知限制

1. **不适用于一次性任务**：罕见或需高度人工判断的任务不应自动化，自动化坏流程比不自动化更危险
2. **需先优化流程再自动化**：流程本身有问题时，自动化会放大问题，必须先修流程
3. **脚本实现需编程基础**：复杂逻辑需 Python/Node 脚本，无代码平台仅适合简单2-3步场景
4. **必须有错误告警**：自动化静默失败最危险，L4告警是硬性要求，不配告警的自动化不应上线
5. **健康度看板需手动维护**：看板数据需定期录入与巡检，不自动采集（除非集成监控系统）
