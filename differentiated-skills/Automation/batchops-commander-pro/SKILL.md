# 批处理指挥官（专业版）

> **企业级批处理编排系统。并行决策+高级检查点+回滚模式+子Agent委派，大规模处理高效安全可恢复。**

批量处理是企业高频且高风险的操作。专业版不仅覆盖批处理全生命周期，更提供并行vs串行自动决策、高级检查点恢复、回滚模式、子Agent委派与幂等性设计，确保万级以上项目的大规模处理高效、安全、可恢复。

## 核心理念

**批处理五原则**：
1. **先试后做**：dry-run预览，确认无误再执行
2. **进度可见**：每10项报告进度，始终知道处理到哪了
3. **错误不蔓延**：单条失败不中断整批，记录后继续
4. **可恢复**：检查点保存，中断后从断点续传
5. **可回滚**：失败后自动回滚已处理项，恢复原始状态

### 功能二：并行vs串行决策矩阵 — 专业版启用

自动分析任务特征，选择最优执行策略：

```yaml
parallel_decision:
  auto: true
  rules:
    - condition: "items > 100 AND no_dependency_between_items"
      strategy: "parallel"
      workers: "min(items/100, cpu_cores)"
    - condition: "items > 100 AND has_dependency"
      strategy: "pipeline"           # 流水线并行
      stages: "auto_detect"
    - condition: "items <= 100 OR api_rate_limit"
      strategy: "sequential"
    - condition: "items > 10000"
      strategy: "delegate"            # 委派给子Agent
      split_size: 1000
```

**决策矩阵**：

| 条件 | 推荐策略 | 理由 |
|:-----|:-----|:-----|
| 项目间无依赖 + >100项 | 并行 | 无依赖可安全并行 |
| 项目间有依赖 | 流水线 | 按依赖关系分阶段 |
| API有速率限制 | 串行+延迟 | 避免触发限流 |
| >10000项 | 子Agent委派 | 单Agent上下文过载 |
| <100项 | 串行 | 并行开销不值得 |
| 需严格顺序 | 串行 | 保证处理顺序 |

## 性能优化策略

### 并行化优化

1. **Worker数量调优**：根据CPU核心数与IO等待比设置，CPU密集型=核心数，IO密集型=2x核心数
2. **负载均衡**：动态分配任务给空闲Worker，避免忙闲不均
3. **批大小调优**：太小则开销大，太大则延迟高，建议100-1000项每批
4. **背压控制**：消费者过载时降速，避免内存溢出

### 检查点优化

1. **增量保存**：仅保存变更部分，非全量
2. **压缩存储**：检查点文件压缩，节省磁盘
3. **异步保存**：不阻塞主流程，异步写入
4. **频率调优**：间隔太小影响性能，太大则恢复损失大，建议50-100项

### 内存优化

1. **流式处理**：不全部加载到内存，逐项或分块处理
2. **缓冲区调优**：根据可用内存设置缓冲区大小
3. **定期GC**：大批次定期触发垃圾回收
4. **对象复用**：复用处理对象，减少分配开销

### 成本控制

- 小规模数据（<1000项）不启用委派，避免子Agent开销
- 非关键任务降低检查点频率
- 串行任务合并小批次，减少启动开销
- 按业务时段调度，利用空闲资源

## FAQ

### Q1：免费版与专业版有什么区别？

免费版提供批处理生命周期管理、进度追踪、基础检查点、三级错误处理与dry-run预览。专业版新增八大高级功能：并行vs串行决策矩阵、高级检查点恢复（增量恢复、版本对比）、回滚模式、多策略错误处理（五级策略）、性能优化（并行化、批处理调优）、子Agent委派、幂等性设计、大规模处理策略（分块、流式、分页）。此外提供7种角色场景指南、性能优化策略与完整故障排查表。

### Q2：并行vs串行如何自动决策？

Agent分析任务特征：项目间无依赖且>100项→并行；有依赖→流水线；API有速率限制→串行+延迟；>10000项→子Agent委派；<100项→串行。也可手动指定策略。

### Q3：高级检查点恢复如何工作？

增量保存：仅保存变更部分，非全量。恢复时：检测检查点→验证数据完整性→版本对比→跳过已处理项→从断点继续。相比免费版的基础检查点，支持增量恢复与数据完整性验证。

### Q4：回滚模式如何触发？

默认失败率超过10%触发自动回滚。也可手动触发。回滚时：停止所有处理→从备份恢复已处理项→保留失败项供分析→记录回滚日志→通知管理员。

### Q5：子Agent委派如何工作？

当项目数超过阈值（默认1000）时，自动拆分为多个块，分配给子Agent并行处理。每个子Agent独立维护检查点，结果自动合并。子Agent失败不影响其他子Agent（故障隔离）。

### Q6：幂等性如何保证？

使用项目哈希或业务ID作为幂等键，处理前检查是否已处理过。如果已处理则跳过，确保重复执行不产生副作用。这对网络不稳定导致的重试场景尤为重要。

### Q7：大规模处理（10万+项）如何优化？

采用分块（每块1000项）+并行（4-8块同时处理）+流式（不全部加载到内存）+定期GC（每1000项触发垃圾回收）的组合策略。100万项约2小时完成。

### Q8：如何处理API速率限制？

自动检测429状态码，启用指数退避重试（1s→2s→4s）。也可预设速率限制（如10请求/秒），Agent自动控制请求频率。严重限流时自动切换为串行模式。

### Q9：批处理失败后如何重试？

失败项保存在failed.json中，可单独重试：`batch retry --from failed.json`。重试时启用幂等性检查，确保不会重复处理已成功的项。

### Q10：如何监控批处理性能？

通过Prometheus指标监控：总项目数、已处理数、失败数、吞吐量（项/秒）、P95延迟。设置告警：失败率>5%、吞吐量异常下降。完成后输出完整报告含平均吞吐与耗时。

### Q11：如何与CI/CD流水线集成？

提供标准CI/CD集成配置，将批处理编排为流水线步骤。支持作为构建步骤（批量验证）、部署步骤（批量配置）或测试步骤（批量测试）。通过API触发与监控。
