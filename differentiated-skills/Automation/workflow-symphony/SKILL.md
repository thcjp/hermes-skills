# 工作流交响曲

让一人公司用无代码工具scale业务而不用招人。核心信条：**先找高价值机会，再选合适工具；先测透再上线，上线后持续维护。**

> 自动化准则：每周做超过两次、且不需要创造性思考的事，就该自动化。

## 第一步：自动化机会雷达图

> 不要盲目自动化。先花1小时审计，找最高价值机会。

### 三维定位法

对每个候选任务按三维度打分（1-5分）：

| 维度 | 1分 | 3分 | 5分 |
|:-----|:----|:----|:----|
| 频次 | 月级 | 周级 | 日级 |
| 单次耗时 | <5分钟 | 15-30分钟 | >30分钟 |
| 重复度（步骤一致性） | 每次都不同 | 部分相同 | 完全相同 |

**高价值区**：三维度都≥3分 → 立即自动化
**中价值区**：两维≥3分 → 下个迭代
**低价值区**：仅一维≥3分 → 跳过

### 自动化审计（1小时）

1. 用一周记录每个任务（笔记本或表格）
2. 每个任务标注：耗时、频次、是否重复
3. 计算时间成本：`时间成本 = (单次分钟 × 月频次) / 60`
4. 按时间成本降序排列

```
示例：15分钟任务 × 每月20次 = 5小时/月
```

### 适合自动化的特征

- 重复（每次步骤相同）
- 规则化（无需复杂判断）
- 高频（每日或每周）
- 耗时（10分钟以上）

### 适合 vs 不适合

| 适合自动化 | 不适合自动化 |
|:-----------|:------------|
| 每周给客户发同格式报告 | 客户访谈（需要察言观色） |
| 付款后自动开票 | 定制化提案（需要创造力） |
| 表单提交自动入CRM | 战略决策（需要判断力） |
| 按计划发社交内容 | 一年一次的任务 |

### 低垂果实清单（从这里开始）

- 表单提交邮件通知
- 表单响应自动存表格
- 提前排期社交内容
- 付款确认自动开票
- 工具间数据同步（CRM ↔ 邮件工具 ↔ 表格）

## 第二步：工具决策树

> 选错工具的代价是后续痛苦迁移。按场景选。

### 决策树

```
你的工作流有几步？
├─ 2-3步
│  └─ 预算<$20/月？
│     ├─ 是 → Zapier免费档
│     └─ 否 → Make入门档
├─ 4-10步 + 有分支

│     ├─ 是 → Make
│     └─ 否 → n8n

   └─ 愿意自部署？
      ├─ 是 → n8n自部署（免费）
      └─ 否 → Make Pro档
```

### 工具对比

| 工具 | 最适合 | 价格 | 学习曲线 | 能力 |
|:-----|:-------|:-----|:---------|:-----|
| Zapier | 简单2-3步工作流 | $20-50/月 | 易 | 低-中 |
| Make | 可视化多步工作流 | $9-30/月 | 中 | 中-高 |
| n8n | 复杂、开发者友好、自部署 | 免费（自部署）或$20/月 | 中-难 | 高 |

### 选择指引

- 预算<$20/月 → Zapier免费档或n8n自部署
- 需要可视化构建 → Make
- 简单2步工作流 → Zapier
- 复杂分支逻辑 → Make或n8n
- 想完全掌控+定制 → n8n
- 高频大量任务（Zapier限流） → n8n自部署

**一人公司推荐路径**：从Zapier起步（最易学），遇到限流或复杂需求再迁Make/n8n。

## 第三步：工作流设计

### 设计模板

```text
触发：什么事件启动工作流？
  示例：Google表格新增一行

条件（可选）：每次都跑还是满足条件才跑？
  示例：仅当"状态"列 = "已批准"

动作：结果应该发生什么？
  步骤1：[动作]
  步骤2：[动作]
  步骤3：[动作]

错误处理：失败时怎么办？
  示例：失败时给我发Slack
```

### 示例

```text
触发：网站表单提交

条件：邮箱字段非空

动作：
  步骤1：加入CRM（Airtable/HubSpot）
  步骤2：发欢迎邮件（ConvertKit）
  步骤3：在项目管理工具（Notion）建任务，3天后跟进
  步骤4：给我发Slack通知"新线索：[姓名]"

错误处理：步骤1失败则邮件告警
```

### 设计原则

- 保持简单——从2-3步开始，复杂度后加
- 单步测试——先单独测每个步骤，再串起来
- 加延迟——部分API慢，动作间加缓冲
- 必有错误通知——坏了要知道

## 第四步：构建与测试

### 构建（Zapier示例）

1. 选触发器应用（Google Forms、Typeform等）
2. 连接账号（OAuth）
3. 测触发器（提交测试表单，看数据是否进来）
4. 加动作（如"Google表格新增行"）
5. 字段映射（表单字段对表格列）
6. 测动作（跑测试验证行被正确添加）
7. 重复添加后续动作
8. 开启工作流

### 12项上线前checklist

> 强制全部通过才能上线。否则就是带病上线。

- [ ] 提交测试数据走完全流程
- [ ] 每个动作单独执行正确
- [ ] 字段映射核对（源→目标对得上）
- [ ] 边界测试：空字段、特殊字符、超长文本
- [ ] 错误处理测试：故意制造失败看告警是否到
- [ ] 重复触发测试：同一条数据跑两次，结果一致（幂等）
- [ ] 时区核对：cron按哪个时区跑
- [ ] 限流测试：高频跑会不会被API拒
- [ ] 凭证有效期：Token何时过期
- [ ] 通知通道测试：告警真的能收到
- [ ] 回滚预案：上线后坏了怎么停
- [ ] 文档已写：未来自己也能看懂

### 常见问题

| 问题 | 原因 | 修复 |
|:-----|:-----|:-----|
| 工作流不触发 | 触发条件太窄 | 放宽过滤，检查webhook签名 |
| 动作失败 | API限流或权限 | 动作间加延迟，重新授权 |
| 数据缺失/错位 | 字段映射错 | 核对映射，加schema版本号 |
| 工作流跑多次 | 重复触发 | 按唯一ID去重 |

**铁律**：用真实数据测试后再依赖。别等真实客户时才发现bug。

## 第五步：维护日历

> 自动化不是"配完就忘"。它会坏、工具会改、API会变。

### 周度检查（5分钟）

- 扫工作流日志找错误
- 失败立即处理

### 月度审计（15分钟）

- 审查所有活跃工作流
- 检查：还在用吗？还在省时间吗？
- 禁用或删除未使用工作流（避免仪表盘混乱）
- 更新依赖已切换工具的工作流

### 季度演练（1小时）

- 模拟一个工作流挂掉，验证告警是否到
- 检查凭证是否快过期
- 评估是否需要迁移工具（如Zapier限流了）
- 更新工作流文档

### 工作流文档模板

每个工作流一份文档（Notion/Google Doc）：

```markdown
# 工作流：[名称]

## 它做什么
[一句话描述]

## 何时运行
[触发条件与频率]

## 连接的应用
- 触发器：[应用A]
- 动作：[应用B]、[应用C]

## 字段映射
| 源字段 | 目标字段 | 转换 |
|:-------|:---------|:-----|
| name | contact.name | 无 |
| email | contact.email | 小写 |

## 故障排查
- 不触发：检查[条件]
- 字段错位：检查[映射]

## 上次更新
2026-07-18
```

> 有10+工作流时，这份文档能在出问题时省你几小时。

## 错误处理

- 所有错误告警统一路由到一个地方（Slack频道、邮箱、任务管理器） - 处理方式: 按上述步骤操作并确认结果
- 设置："任何工作流失败，发消息到[你的错误频道]" - 处理方式: 按上述步骤操作并确认结果
- 每周复查错误，修根因 - 处理方式: 按上述步骤操作并确认结果

## 第七步：ROI计算

### 公式

```
月节省小时 = (单次分钟 / 60) × 月频次
成本 = (搭建小时 × $50/小时) + 工具月费
回本月数 = 搭建成本 / 月节省价值

回本 < 3个月 → 值得
回本 > 6个月 → 不值得（除非解锁其他价值）
```

### 实战示例

```
任务：手工把表单提交复制到CRM（15分钟 × 每月20次 = 月省5小时）
搭建：1小时
工具：Zapier $20/月
回本：($50搭建) / ($250/月节省价值) = 0.2个月 → 绝对值得
```

**铁律**：聚焦回本<3个月的自动化。那是最高杠杆投资。

## 自动化要避免的坑

- **优化前先自动化**：别自动化坏流程。先修流程再自动化。
- **过度自动化**：不是所有事都该自动化。罕见或需判断的任务手工做。
- **无错误处理**：自动化坏了不知道，造成静默失败。必有错误告警。
- **测试不充分**：坏自动化比没自动化更糟——产生错误数据或漏任务。
- **一上来就搞复杂**：从简单2-3步开始，简单版完美后再加复杂度。
- **不写文档**：未来的你会忘记这是怎么配的。写下来。

## 静默失败防护

> 一人公司最大的风险：自动化挂了一周你都不知道。

### 统一告警路由

| 故障类型 | 告警通道 | 响应时限 |
|:---------|:---------|:---------|
| 临时失败（限流） | 日志 | 自动重试 |
| 持续失败（3次） | Slack | 4小时内 |
| 数据缺失 | 邮件+Slack | 当日 |
| 凭证失效 | 紧急Slack | 1小时 |

### 每周巡检5问

1. 所有自动化这周都跑了吗？
2. 有没有该到没到的数据？
3. 有没有客户反馈"没收到邮件"？
4. 凭证还有多久过期？
5. 哪个自动化这周最慢？

## FAQ

**Q：我该从哪个自动化开始？**
A：从"低垂果实清单"挑一个——表单邮件通知、自动存表格、付款自动开票。这些回本快、风险低。

**Q：Zapier限流了怎么办？**
A：先看能不能优化（合并动作、减少频次）。不行就迁Make或n8n自部署。迁移前先跑并行验证。

**Q：自动化经常静默失败怎么办？**
A：统一告警路由到Slack一个频道。每周5分钟巡检。每个自动化配"失败3次即告警"。

**Q：自动化越攒越多管不过来？**
A：上月度审计——禁用未使用的、合并相似的、文档化存活的。季度演练一次故障告警。

**Q：怎么知道自动化还值不值？**
A：算ROI。月节省小时 × 时薪 vs 工具费+维护时间。回本<3个月值得，>6个月考虑停。

## 故障排查

| 症状 | 可能原因 | 解决 |
|:-----|:---------|:-----|
| 工作流不触发 | 触发条件太窄 | 检查过滤、webhook签名、OAuth状态 |
| 间歇失败 | API限流 | 动作间加延迟，迁高频到n8n |
| 字段错位 | 源系统改了字段 | 重新映射，加schema版本号 |
| 重复执行 | 重复触发 | 按唯一ID去重 |

| 客户没收到邮件 | 被判垃圾邮件 | 检查SPF/DKIM，换邮件服务商 |

### 核心功能执行
用`input_params`参数进行配置。

**输入**: 用户提供核心功能执行所需的指令和必要参数。
**处理**: 解析核心功能执行的输入参数,完成核心逻辑,返回结构化响应。
**输出**: 返回核心功能执行的响应数据,包含状态码、结果和日志。
- 执行此能力时使用`input_params`参数,支持创建/查询/导出操作

### 参数配置与调用
用`config_options`参数进行配置。

**输入**: 用户提供参数配置与调用所需的指令和必要参数。
**处理**: 解析参数配置与调用的输入参数,完成核心逻辑,返回结构化响应。
**输出**: 返回参数配置与调用的响应数据,包含状态码、结果和日志。
- 执行此能力时使用`config_options`参数,支持修改/重置/导入操作

### 结果处理与输出
用`output_format`参数进行配置。

**输入**: 用户提供结果处理与输出所需的指令和必要参数。
**处理**: 解析结果处理与输出的输入参数,完成核心逻辑,返回结构化响应。
**输出**: 返回结果处理与输出的响应数据,包含状态码、结果和日志。
- 执行此能力时使用`output_format`参数,支持导出/保存/转换操作
