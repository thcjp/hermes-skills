# 批处理指挥官（免费版）

> **批量处理多项目的安全管家。进度追踪+检查点保存+错误恢复，批量任务可监控、可恢复、可追溯。**

批量处理是高频但高风险的操作——一次错误可能影响数百条数据。本技能提供完整的批处理生命周期管理，确保每次批量操作安全可控。

## 核心理念

**批处理三原则**：
1. **先试后做**：dry-run预览，确认无误再执行
2. **进度可见**：每10项报告进度，始终知道处理到哪了
3. **错误不蔓延**：单条失败不中断整批，记录后继续

## 核心功能

### 功能一：批处理生命周期

#### 开始前

```text
1. Dry-run：先用2-3项测试，验证处理逻辑正确
2. 计数：明确告知总数与预估时间
   "处理47项，预计2分钟"
3. 确认破坏性操作：
   "这将删除200个文件。确认继续？"
   "这将向500用户发送邮件。确认继续？"
```

#### 处理中

```text
- 每10项报告进度："23/47完成（49%）"
- 每10-50项保存检查点：中断后可恢复
- 出错时：记录错误，继续处理其余项（不中断整批）
```

#### 完成后

始终报告完整结果：

```text
批处理完成
成功：44项
失败：3项（已保存至failed.json供重试）
跳过：0项
总耗时：2分15秒
```

### 功能二：进度追踪

```yaml

  estimated_time: true          # 显示预估剩余时间
  example: "23/47完成（49%），预计还需1分钟"
```

**进度报告示例**：
```text

[20/47] 完成（43%）
[30/47] 完成（64%）
[40/47] 完成（85%）
[47/47] 完成（100%）
```

### 功能三：检查点保存

```yaml
checkpoint:
  save_interval: 10             # 每10项保存
  save_location: ".batch-checkpoint.json"
  save_contents:
    - total_items
    - processed_items
    - failed_items
    - last_position
  resume_support: true          # 支持从检查点恢复
```

**检查点文件示例**：
```json
{
  "batch_id": "file-rename-20260115",
  "total": 47,
  "processed": 30,
  "failed": [12, 18],
  "last_position": 30,
  "timestamp": "2026-01-15T10:30:00Z"
}
```

**从检查点恢复**：
```text
检测到检查点：已处理30/47项，3项失败
从第31项继续处理
```

### 功能四：错误处理矩阵

| 错误类型 | 处理策略 | 示例 |
|---------|---------|------|
| 超时、速率限制 | 重试3次，指数退避（1s, 2s, 4s） | API调用超时 |
| 格式错误、数据缺失 | 跳过，记录，继续 | 文件格式不正确 |
| 认证失败、磁盘满 | 中止整批，保护数据 | 凭据过期 |

**错误记录格式**：
```json

  "failed_items": [
    {"index": 12, "item": "file_12.txt", "error": "格式不支持", "action": "skipped"},
    {"index": 18, "item": "file_18.txt", "error": "文件不存在", "action": "skipped"},
    {"index": 25, "item": "file_25.txt", "error": "API超时", "action": "retried_3_times_failed"}
  ]
}
```

3. 执行批量重命名，每10个报告进度
4. 完成报告：44成功，3失败（文件名含特殊字符）

**效果**：47个文件1分钟完成，失败项有记录可单独处理。

**痛点**：有500条客户数据需要清洗（去空格、统一格式、验证邮箱），手动不可行。

**使用流程**：
1. Dry-run预览5条数据，验证清洗规则
2. 确认总数："处理500条，预计5分钟"
3. 每50条保存检查点
4. 执行批量清洗，记录格式错误项
5. 完成报告：487成功，13失败（邮箱格式异常）

**效果**：100台服务器3分钟完成，离线服务器有记录可排查。

## 批处理模板

### 文件批量处理模板

```bash
#!/bin/bash
set -euo pipefail

ITEMS=(*.txt)
TOTAL=${#ITEMS[@]}
PROCESSED=0
FAILED=0
FAILED_ITEMS="failed.json"

echo "[]" > "$FAILED_ITEMS"

for item in "${ITEMS[@]}"; do
  if process_item "$item"; then
    ((PROCESSED++))
  else
    ((FAILED++))
    # 记录失败项
    jq --arg item "$item" --arg err "$?" '. += [{"item": $item, "error": $err}]' "$FAILED_ITEMS" > tmp && mv tmp "$FAILED_ITEMS"
  fi

  # 每10项报告进度
  if (( PROCESSED % 10 == 0 )); then
    echo "$PROCESSED/$TOTAL完成（$(( PROCESSED * 100 / TOTAL ))%）"
  fi

  # 每50项保存检查点
  if (( PROCESSED % 50 == 0 )); then
    echo "{\"processed\": $PROCESSED, \"total\": $TOTAL}" > .batch-checkpoint.json
  fi
done

echo "批处理完成"
echo "成功：$((TOTAL - FAILED))"
echo "失败：$FAILED（已保存至$FAILED_ITEMS供重试）"
```

## FAQ

### Q1：免费版能处理多少个项目？

免费版不限处理数量，可以处理任意规模的批次。免费版聚焦安全批处理基础能力（dry-run、进度追踪、检查点、错误处理），帮助用户避免批量操作中的数据丢失。

### Q2：检查点保存是什么？为什么重要？

检查点保存是每处理10-50项后，将当前进度（已处理数、失败项、位置）保存到文件。如果批处理被中断（如系统重启、网络断开），可以从检查点恢复，无需从头开始。这对大规模批处理尤为重要。

### Q3：Dry-run模式具体做什么？

Dry-run模式用2-3项数据模拟执行处理逻辑，展示"如果实际执行会怎样"的效果，但不实际修改数据。帮助你在执行前验证处理规则是否正确，避免对大量数据执行错误操作。

### Q4：单条失败会中断整个批次吗？

不会。对于可恢复的错误（格式错误、数据缺失），Agent会跳过该项、记录错误、继续处理其余项。只有遇到不可恢复的错误（认证失败、磁盘满）才会中止整批，保护数据安全。

### Q5：免费版与专业版有什么区别？

免费版提供批处理生命周期管理、进度追踪、基础检查点、三级错误处理与dry-run预览。专业版新增并行vs串行决策矩阵、高级检查点恢复、回滚模式、多策略错误处理、性能优化（并行化、批处理调优）、子Agent委派、幂等性设计与大规模处理策略。
