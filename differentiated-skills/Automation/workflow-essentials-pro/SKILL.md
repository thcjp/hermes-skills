## 核心能力
### Step 1：识别自动化机会

**自动化黄金法则**：每周重复两次以上、不需要创造性思维的任务，都值得自动化.
**适合自动化的四特征**：

| 特征 | 说明 | 示例 |
|:-----|:-----|:-----|
| 重复性 | 每次步骤相同 | 每周格式相同的周报 |
| 规则性 | 规则驱动，无需判断 | 付款后创建发票 |
| 高频率 | 每日或每周执行 | 表单数据入库 |
| 耗时长 | 单次10分钟以上 | 数据手动汇总 |

**快速识别清单**：

```text
问自己：
  □ 这个任务每次步骤都一样吗？
  □ 这个任务每周至少做两次吗？
  □ 单次耗时超过10分钟吗？
  □ 不需要创造性判断吗？
# ...
四个都"是" → 立即自动化
三个"是" → 建议自动化
两个以下"是" → 手动即可
```

**时间成本计算**：

```text
时间成本 = (单次耗时分钟 × 月频率) / 60
# ...
示例：15分钟的任务 × 每月20次 = 每月5小时
# ...
按时间成本从高到低排序，优先自动化成本最高的任务.
```

**输入**: 用户提供Step 1：识别自动化机会所需的指令和必要参数.
**处理**: 解析Step 1：识别自动化机会的输入参数,完成核心逻辑,返回结构化响应.
**输出**: 返回Step 1：识别自动化机会的响应数据,包含状态码、结果和日志.
### Step 2：设计自动化方案

#### 六种设计模式 — 专业版独有

每种模式都是一个可复用的设计模板，根据任务特征选择合适的模式.
**模式1：定期汇总**

```text
特征：多个数据源 → 合并 → 统一报告
触发：定时
适用：周报、月报、数据汇总
# ...
模板：
  触发：定时（如每周五）
  步骤1：从多个源收集数据
  步骤2：合并整理
  步骤3：生成报告
  步骤4：发送/保存
  错误处理：失败通知，跳过缺失源继续处理
```

**模式2：数据同步**

```text
特征：系统A → 转换 → 系统B
触发：定时或事件
适用：CRM↔ERP同步、跨工具数据流转
# ...
模板：
  触发：定时或新数据事件
  步骤1：从源系统读取数据
  步骤2：按映射规则转换格式
  步骤3：写入目标系统
  错误处理：失败重试3次，记录同步日志
```

**模式3：事件驱动**

```text
特征：事件发生 → 判断 → 响应
触发：事件（Webhook/文件变化/状态变更）
适用：订单处理、告警通知、客户跟进
# ...
模板：
  触发：事件发生
  条件：判断事件是否满足响应条件
  步骤1：执行响应操作
  步骤2：发送通知
  错误处理：失败告警，确保不遗漏关键事件
```

**模式4：批量转换**

```text
特征：多个文件 → 统一处理 → 输出
触发：手动或文件监控
适用：格式转换、数据清洗、批量生成
# ...
模板：
  触发：手动或目录有新文件
  步骤1：枚举所有待处理文件
  步骤2：逐个执行转换
  步骤3：输出到目标目录
  错误处理：单文件失败跳过，记录错误列表
```

**模式5：流程编排**

```text
特征：多步骤任务 → 按序执行 → 确认完成
触发：事件或手动
适用：审批流程、入驻流程、发布流程
# ...
模板：
  触发：流程启动事件
  步骤1-N：按序执行各步骤
  检查点：每步完成后确认，失败则暂停
  完成：所有步骤完成后发送通知
  错误处理：失败暂停流程，通知人工介入
```

**模式6：增量处理**

```text
特征：只处理新增/变更数据
触发：定时
适用：日志分析、数据同步、状态更新
# ...
模板：
  触发：定时
  步骤1：记录上次处理位置（水位线）
  步骤2：只读取水位线之后的新数据
  步骤3：处理新数据
  步骤4：更新水位线
  错误处理：失败不更新水位线，下次重试
```

**模式选择速查卡**：

| 任务特征 | 推荐模式 | 典型场景 |
|---:|---:|---:|
| 多源数据合并成报告 | 定期汇总 | 周报、月报 |
| 两个系统间数据流转 | 数据同步 | CRM↔ERP |
| 事件发生后需要响应 | 事件驱动 | 订单处理 |
| 多个文件统一处理 | 批量转换 | 格式转换 |
| 多步骤按序执行 | 流程编排 | 入驻流程 |
| 只处理新增数据 | 增量处理 | 日志分析 |

**三条设计原则**：

1. **从简单开始**：先做2-3步，跑通后再加复杂度
2. **逐步测试**：每步单独测试成功后再串联
3. **必加错误通知**：出问题要能知道，否则静默失败

### Step 3：构建与验证

**工具选择速查**：

| 工具 | 适合 | 价格 | 难度 | 能力 |
|:---:|:---:|:---:|:---:|:---:|
| Zapier | 简单2-3步工作流 | $20-50/月 | 简单 | 低-中 |
| Make | 可视化多步骤工作流 | $9-30/月 | 中等 | 中-高 |
| n8n | 复杂工作流，自托管 | 免费或$20/月 | 较难 | 高 |

**构建步骤（通用流程）**：

1. 选择触发器应用
2. 连接账户（OAuth认证）
3. 测试触发器（确认数据能过来）
4. 添加第一个操作
5. 映射字段（匹配数据到目标）
6. 测试操作（确认结果正确）
7. 重复添加更多操作
8. 开启工作流

**验证清单**：

- [ ] 提交测试数据，触发器是否响应
- [ ] 每个操作是否正确执行
- [ ] 数据是否映射到正确字段
- [ ] 边界情况测试（空字段、特殊字符）
- [ ] 错误处理测试（故意制造失败）

**常见问题速修**：

| 问题 | 修复方法 |
|:------|------:|
| 工作流不触发 | 检查触发条件，放宽过滤 |
| 操作失败 | 检查API限流，添加延迟 |
| 数据错误 | 检查字段映射是否正确 |
| 重复执行 | 基于唯一ID去重 |

### Step 4：维护策略 — 专业版独有

自动化不是"设置完就忘了"。系统化的维护是持续有效的关键.
**维护节奏**：

| 频率 | 时间 | 检查内容 |
|---:|:---|---:|
| 每周 | 5分钟 | 扫描日志，处理失败 |
| 每月 | 15分钟 | 审计所有工作流，淘汰低效的 |
| 每季 | 30分钟 | 评估ROI，优化高效工作流 |

**每周检查清单**：
- 扫描工作流日志，检查是否有错误
- 立即处理任何失败
- 记录本周新增的工作流

**月度审计清单**：
- 审查所有活跃的工作流
- 检查：还在用吗？还在节省时间吗？
- 禁用或删除不用的工作流
- 更新依赖已切换工具的工作流

**文档管理**：

为每个工作流创建文档，包含：
```text
工作流名称：__________
创建日期：__________
最后检查：__________
状态：__________
# ...
本月运行次数：____
本月失败次数：____
# ...
连接的应用：
  - __________（触发器）
  - __________（操作1）
  - __________（操作2）
# ...
排障指南：
  1. 如果不触发：__________
  2. 如果操作失败：__________
  3. 如果数据错误：__________
```

**错误通知集中化**：
- 将所有工作流的错误通知路由到一个地方（Slack频道/邮件/任务管理器）
- 设置："如果任何工作流失败，发消息到[错误频道]"
- 每周审查错误并修复根本原因

**输入**: 用户提供Step 4：维护策略 — 专业版独有所需的指令和必要参数.
**处理**: 解析Step 4：维护策略 — 专业版独有的输入参数,完成核心逻辑,返回结构化响应.
**输出**: 返回Step 4：维护策略 — 专业版独有的响应数据,包含状态码、结果和日志.

## 常见错误清单

### 错误1：先自动化再优化流程

**问题**：自动化一个糟糕的流程，只是让糟糕的流程跑得更快.
**正确做法**：先优化流程，再自动化。如果手动做都很痛苦，自动化后只会更痛苦.
### 错误2：过度自动化

**问题**：试图自动化一切，包括罕见或需要判断的任务.
**正确做法**：不是所有事情都需要自动化。如果任务罕见或需要判断，手动做。聚焦高频、规则驱动的任务.
### 错误3：没有错误处理

**问题**：自动化坏了你却不知道，造成静默失败.
**正确做法**：始终设置错误告警。将所有错误通知路由到一个地方，每周审查.
### 错误4：测试不充分

**问题**：一个坏掉的自动化比没有自动化更糟——它会产生错误数据或遗漏任务.
**正确做法**：用真实数据彻底测试，包括边界情况。不要等真实客户遇到了才发现问题.
### 错误5：太快变得复杂

**问题**：一开始就建10步工作流，出问题时无法定位.
**正确做法**：从简单的2-3步开始，简单版本完美运行后再加复杂度.
### 错误6：不写文档

**问题**：未来的你会忘记这个工作流怎么工作的.
**正确做法**：为每个工作流写文档，包括做什么、何时运行、连接哪些应用、如何排障.

## FAQ

### Q1：免费版与专业版有什么区别？

免费版提供三步精要框架（识别→设计→构建）和三个基础场景。专业版解锁六步完整框架（新增维护、进阶、优化）、六种设计模式、四种进阶场景、维护策略、常见错误清单（6项）、多角色场景指南（7种角色）、工具迁移指南和完整故障排查表（8项）。专业版还提供设计模式速查卡和ROI优化方法.
### Q2：六种设计模式有什么用？

设计模式是可复用的工作流设计模板。面对一个自动化需求时，先判断它属于哪种模式（定期汇总/数据同步/事件驱动/批量转换/流程编排/增量处理），然后套用对应模式的设计模板，可以大幅加速设计过程。模式之间可以组合使用，例如"定期汇总+事件驱动"用于监控告警场景.
### Q3：什么时候该从Zapier迁移到Make或n8n？

当满足以下任一条件时考虑迁移：月费超过$50（成本太高）；工作流步骤数受限（需要更复杂的逻辑）；需要分支逻辑或循环（Zapier支持有限）；有技术能力且想降低成本（n8n自托管免费）；需要自定义代码节点（Make和n8n支持更好）。专业版提供详细的迁移指南.
### Q4：如何系统化管理多个工作流？

专业版建议建立维护节奏：每周5分钟扫描日志处理失败，每月15分钟审计所有工作流淘汰低效的，每季30分钟评估ROI优化高效的。为每个工作流创建文档，记录做什么、何时运行、连接哪些应用、如何排障。将错误通知集中到一个渠道，统一管理.
### Q5：增量处理模式是什么意思？

增量处理是只处理新增或变更的数据，而非全量处理。通过记录"水位线"（上次处理到的位置），每次只读取水位线之后的新数据。这适合数据量大且变更频率低的场景（如日志分析、数据同步），可以大幅减少处理时间和资源占用。如果处理失败，不更新水位线，下次自动从同一位置重试.
### Q6：流程编排模式和其他模式有什么区别？

流程编排是唯一一个强调"步骤顺序"和"检查点"的模式。其他模式（如定期汇总、数据同步）的步骤相对独立，某步失败不影响整体。而流程编排的步骤有严格的先后顺序，每步完成后需要确认，失败则暂停整个流程并通知人工介入。适合需要人工确认的多步骤场景（如审批流程、入驻流程）.
### Q7：如何计算自动化的ROI？

ROI计算公式：月节省时间=（单次耗时/60）×月频率；月节省价值=月节省时间×时薪；回收周期=建设成本/月节省价值。回收周期<3个月值得保留并优化，>6个月考虑淘汰。专业版建议每季度重新评估所有工作流的ROI，淘汰低效的，优化高效的.
### Q8：如何在团队中推广自动化？

专业版建议以下推广路径：第一步，自己先做1-2个示范工作流，让团队看到效果；第二步，梳理团队常见重复任务，按角色分类，为每类选择设计模式；第三步，建立团队维护节奏，分配每个工作流的责任人；第四步，逐步推广到更多场景；第五步，季度评估ROI，分享成功案例，形成自动化文化.
### Q9：自动化工作流会失效吗？怎么应对？

会失效，主要原因是第三方工具API变更。专业版建议：每周检查工作流日志及时发现失效；关注工具的变更通知；为关键工作流准备手动备份方案；在文档中记录依赖的API版本；使用支持自动重试的工具。工具迁移时，新旧并行运行确认新工作流正常后再关闭旧的.
### Q10：可以从免费版升级到专业版吗？

可以，无需迁移数据。专业版完全兼容免费版的三步框架。升级后建议对已有的自动化工作流补充维护策略（Step 4），尝试用设计模式重新审视已有工作流看是否可以优化，并计算每个工作流的ROI（Step 6）验证是否值得继续维护.
## 故障排查表

| 问题 | 可能原因 | 解决方案 | 优先级 |
|---:|---:|---:|---:|
| 工作流不触发 | 触发器配置错误或源无新数据 | 检查触发器设置；确认源有新事件 | 高 |
