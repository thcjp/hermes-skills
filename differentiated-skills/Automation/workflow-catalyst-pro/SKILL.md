## 核心能力
### 六步自动化框架

#### Step 1：观察（Observe）

识别重复任务是自动化的前提。Agent应主动观察用户的操作模式.
**观察清单**：
- 用户是否在重复执行相同的操作？
- 操作步骤是否每次都一样？
- 操作频率如何（每日/每周/每月）？
- 单次操作耗时多少？
- 操作是否跨多个工具/系统？

**自动化适合度评分**：

| 指标 | 权重 | 适合自动化 | 不适合自动化 |
|:-----|:-----|:-----|:-----|
| 重复性 | 30% | 每次步骤相同 | 每次都不同 |
| 频率 | 25% | 每周2次以上 | 每月1次以下 |
| 耗时 | 20% | 单次10分钟以上 | 单次2分钟以下 |
| 规则性 | 15% | 规则驱动，无需判断 | 需要创意或人际判断 |
| 跨系统 | 10% | 涉及多个系统切换 | 单系统内操作 |

**评分标准**：总分≥70分（满分100）建议自动化，50-70分可考虑，<50分手动即可.
#### Step 2：抽象（Abstract）

将具体操作抽象为通用模式，与模式库匹配.
```text
具体操作：每周五打开5个部门邮件 → 下载附件 → 复制数据到汇总表 → 发送给老板
# ...
抽象模式：
  输入：多个数据源（邮件附件）
  处理：收集 → 整理 → 合并
  输出：汇总报告（发送邮件）
  触发：定时（每周五）
# ...
模式库匹配：→ "定期收集汇总"模式
```

#### Step 3：设计（Design）

设计自动化方案，明确触发方式、处理步骤和输出.
```text
设计方案：
  触发：每周五下午5点（定时触发）
  步骤1：从5个部门邮箱下载附件
  步骤2：提取数据并合并到汇总表
  步骤3：生成格式化报告
  步骤4：发送邮件给老板
  错误处理：任一步骤失败时发送通知
  超时处理：单步骤超过30分钟自动终止并告警
```

将设计方案转化为可执行的脚本或配置.

### ROI计算工具 — 专业版独有

量化自动化的投入产出比，辅助决策.
```text
ROI计算公式：
# ...
  月节省时间(小时) = (单次耗时分钟 / 60) × 月执行次数
  月节省价值(元) = 月节省时间 × 时薪
  建设成本(元) = 建设耗时(小时) × 时薪 + 工具月费
  回收周期(月) = 建设成本 / 月节省价值
# ...
  判断标准：
    回收周期 < 3个月  → 立即自动化
    回收周期 3-6个月  → 建议自动化
    回收周期 > 6个月  → 暂不自动化
```

**示例计算**：

```text
任务：周报汇总
  单次耗时：120分钟
  月执行次数：4次
  时薪：100元/小时
  建设耗时：3小时
  工具月费：0元（用免费脚本）
# ...
计算：
  月节省时间 = (120/60) × 4 = 8小时
  月节省价值 = 8 × 100 = 800元
  建设成本 = 3 × 100 + 0 = 300元
  回收周期 = 300 / 800 = 0.375个月
# ...
结论：回收周期<1个月 → 立即自动化 ✅
```

## 自动化思维模式

工作流催化器的核心不是"回答问题"，而是"看到重复→立刻构建自动化".
### 主动识别信号

当Agent观察到以下信号时，应主动提出自动化建议：

- 用户第二次描述相同操作时
- 用户说"我又要..."或"每次都要..."时
- 用户描述的操作包含"复制""粘贴""导出""导入"时
- 用户描述的操作耗时超过10分钟时
- 用户描述的操作涉及多个系统切换时

### 自动化决策树

```text
用户描述一个任务
  ├─ 每次步骤相同吗？
  │   ├─ 是 → 继续判断
  │   └─ 否 → 不适合自动化
  ├─ 频率如何？
  │   ├─ 每周2次以上 → 强烈建议自动化
  │   ├─ 每月2-8次 → 建议自动化
  │   └─ 每月1次以下 → 手动即可
  ├─ 需要创造性判断吗？
  │   ├─ 不需要 → 适合自动化
  │   └─ 需要 → 不适合自动化
  └─ ROI计算
      ├─ 回收周期<3个月 → 立即自动化
      ├─ 回收周期3-6个月 → 建议自动化
      └─ 回收周期>6个月 → 暂不自动化
```

## FAQ

### Q1：免费版与专业版有什么区别？

免费版提供四步核心框架（观察→抽象→设计→实现）和3个基础模板。专业版解锁六步完整框架（新增测试和优化两步）、12种模式库、ROI计算工具、6个进阶模板、多角色场景指南（7种角色）和完整故障排查表（11项）。专业版还提供自动化适合度评分和升级迁移指南.
### Q2：六步框架比四步框架多了什么？

专业版的六步框架新增了测试（Step 5）和优化（Step 6）两步。测试步骤提供6项标准化测试（正常流程、空输入、部分失败、超时处理、重复执行、错误恢复），确保自动化方案可靠。优化步骤提供6个优化方向（重试机制、执行日志、性能监控、增量处理、并行化、缓存策略），持续提升方案质量.
### Q3：模式库有什么用？

模式库包含12种常见自动化模式（定期收集汇总、数据同步、文件整理、定时抓取、事件触发等），每种模式都标注了输入、处理、输出和典型场景。当Agent在Step 2（抽象）阶段识别出任务模式后，可以在模式库中找到参考实现，加速设计和实现阶段.
### Q4：ROI计算工具如何使用？

ROI计算工具通过四个参数量化自动化的投入产出：单次耗时、月执行次数、时薪、建设耗时。计算结果给出月节省时间、月节省价值、建设成本和回收周期。回收周期<3个月建议立即自动化，3-6个月建议自动化，>6个月暂不自动化。这让自动化决策有数据支撑，而非凭感觉.
### Q5：自动化方案上线后如何维护？

专业版建议建立维护机制：
- 每周检查执行日志，处理失败记录
- 每月审查自动化效果，优化性能瓶颈
- 每季度评估ROI，淘汰低效自动化
- 工具/API变更时及时更新脚本
- 保持错误告警渠道畅通

### Q6：测试阶段需要测什么？

专业版提供6项标准化测试：
1. 正常流程测试：用真实数据执行，验证全步骤成功
2. 空输入测试：数据源为空时，验证优雅处理
3. 部分失败测试：模拟某个数据源不可用，验证跳过失败项继续处理
4. 超时处理测试：模拟单步骤超时，验证自动终止和告警
5. 重复执行测试：同一时间执行两次，验证幂等性
6. 错误恢复测试：中断后重新执行，验证从断点继续

### Q7：如何从免费版升级到专业版？

无需迁移数据。专业版完全兼容免费版的四步框架和3个基础模板。升级后原有脚本和配置无需修改，新增的测试和优化步骤可以立即应用到已有自动化方案上。建议对已运行的自动化方案补充测试和优化两个步骤.
### Q8：并行化优化如何实现？

将自动化方案中的独立步骤改为并行执行。例如周报自动化中，5个部门的数据收集是独立的，可以并行执行（5个线程同时收集），而非串行（一个接一个）。并行化可将总耗时从5×10分钟=50分钟缩短到max(10分钟)=10分钟。需要注意有依赖关系的步骤不能并行.
### Q9：增量处理是什么意思？

增量处理是只处理新增或变更的数据，而非全量处理。例如数据同步场景，每次只同步上次以来的变更数据，而非重新同步全部数据。增量处理可以大幅减少处理时间和资源占用，适合数据量大且变更频率低的场景.
### Q10：自动化方案失败了怎么办？

专业版提供完整的故障排查表（11项），覆盖常见故障场景。每个自动化方案都应包含错误处理机制：失败步骤自动重试3次、重试失败发送告警通知、记录详细错误日志、支持从断点恢复。建议建立错误处理SOP，明确每种错误的处理流程.
### Q11：可以同时管理多个自动化方案吗？

可以。建议为每个自动化方案建立独立的目录结构，包含脚本、配置、日志和数据。用统一的调度器（cron/任务计划）管理所有定时任务。每月进行一次审查，评估各方案的运行状态和ROI，淘汰低效方案，优化高效方案.
## 故障排查表

| 问题 | 可能原因 | 解决方案 | 优先级 |
|----|:--:|---:|----|
| 脚本执行失败 | 依赖缺失或路径错误 | 检查Python环境和依赖；使用绝对路径 | 高 |
| 定时任务未触发 | cron配置错误或服务未运行 | 检查cron表达式；确认调度服务运行 | 高 |
| 数据源连接失败 | API变更或认证过期 | 更新API配置；重新认证；添加重试机制 | 高 |
| 输出数据不正确 | 字段映射错误或格式变更 | 检查字段映射；验证数据源格式 | 高 |
| 执行超时 | 数据量过大或网络延迟 | 添加超时处理；拆分批次；优化查询 | 中 |
| 重复执行产生重复数据 | 缺少幂等检查 | 添加唯一标识检查；使用upsert操作 | 中 |
| 邮件/通知未发送 | SMTP配置错误或频率限制 | 检查邮件配置；添加发送重试 | 中 |
| 内存不足 | 一次性加载过多数据 | 改为分块处理；使用流式读写 | 中 |
| 脚本权限不足 | 文件或目录权限问题 | 检查执行权限；chmod +x | 低 |
| 日志文件过大 | 未设置日志轮转 | 配置logrotate；按日期分割日志 | 低 |
| 跨平台兼容问题 | 路径或编码差异 | 使用pathlib；统一UTF-8编码 | 低 |
