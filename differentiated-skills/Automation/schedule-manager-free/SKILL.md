# 任务调度管家（免费版）

> **让Agent准确知道"何时"执行"何事"。自然语言驱动，时区感知，持久化存储。**

任务调度管家免费版解决一个核心痛点：用户说"每天早上提醒我站会"，Agent需要知道"每天早上"到底是几点、在哪个时区、执行什么动作。本技能把模糊的人类时间表达转化为精确的机器调度，并把任务定义持久化，确保重启后依然有效。

## 核心理念

**职责分离原则**：
- **WHAT（做什么）**：由用户显式定义，可能需要其他技能配合
- **WHEN（何时做）**：由本技能负责解析与触发
- **HOW（怎么做）**：由用户显式授予所需权限

本技能绝不假设拥有任何外部服务的访问权限，绝不修改系统crontab或launchd，绝不在用户未授权时执行任务。

## 核心能力
### 功能一：自然语言时间解析

支持中英文混合的时间表达，自动转化为标准cron表达式。

| 用户表达 | 解析结果 | cron表达式 |
|----------|----------|-----------|
| 每天早上 | 每日 09:00（默认） | `0 9 * * *` |
| 每周一上午 | 每周一 10:00（默认） | `0 10 * * 1` |
| 工作日9点 | 周一至周五 09:00 | `0 9 * * 1-5` |
| 每月1号 | 每月1日 00:00 | `0 0 1 * *` |
| 下周五下午3点 | 一次性 2026-07-25 15:00 | 一次性任务 |
| 每隔2小时 | 每2小时执行 | `0 */2 * * *` |
| 每天8点和20点 | 每日 08:00、20:00 | `0 8,20 * * *` |

**解析规则**：
- "早上"默认09:00，"上午"默认10:00，"中午"默认12:00，"下午"默认14:00，"晚上"默认20:00，"深夜"默认23:00
- 这些默认值会被preferences.json中的用户偏好覆盖
- 一次性任务不使用cron，而是存储具体的ISO时间戳

**输入**: 用户提供功能一：自然语言时间解析所需的指令和必要参数。
**处理**: 解析功能一：自然语言时间解析的输入参数,完成核心逻辑,返回结构化响应。
**输出**: 返回功能一：自然语言时间解析的响应数据,包含状态码、结果和日志。

### 功能二：任务定义持久化

所有任务存储在`~/workspace/schedule/jobs.json`：

```json
{
  "daily_email_summary": {
    "cron": "0 8 * * 1-5",
    "task": "汇总今日邮件，列出前5封重要邮件",
    "requires": ["mail"],
    "timezone": "Asia/Shanghai",
    "enabled": true,
    "created": "2026-07-18",
    "last_run": null,
    "next_run": "2026-07-21T08:00:00+08:00",
    "id": "job_d4e5f6"
  },
  "weekly_report_reminder": {
    "type": "once",
    "datetime": "2026-07-25T15:00:00+08:00",
    "task": "提醒：提交本周周报",
    "requires": [],
    "enabled": true,
    "created": "2026-07-18",
    "id": "job_a1b2c3"
  }
}
```

**字段说明**：
- `cron`/`datetime`：周期任务用cron，一次性任务用datetime
- `requires`：显式声明该任务需要的技能与权限
- `enabled`：是否启用（false表示暂停）
- `next_run`：下次执行时间（ISO 8601带时区）

**输入**: 用户提供功能二：任务定义持久化所需的指令和必要参数。
**处理**: 解析功能二：任务定义持久化的输入参数,完成核心逻辑,返回结构化响应。
**输出**: 返回功能二：任务定义持久化的响应数据,包含状态码、结果和日志。

### 功能三：时区感知与偏好学习

首次创建任务后，自动在`preferences.json`记录用户偏好：

```json
{
  "timezone": "Asia/Shanghai",
  "time_preferences": {
    "morning": "08:00",
    "noon": "12:00",
    "afternoon": "14:00",
    "evening": "20:00",
    "night": "23:00"
  },
  "notification_style": "concise",
  "created_at": "2026-07-18"
}
```

**偏好学习规则**：
- 用户说"早上8点"时，更新`morning`偏好为08:00
- 用户明确指定时区时（如"用UTC时间"），更新`timezone`
- 偏好仅在被用户明确确认后更新，不擅自猜测

**输入**: 用户提供功能三：时区感知与偏好学习所需的指令和必要参数。
**处理**: 解析功能三：时区感知与偏好学习的输入参数,完成核心逻辑,返回结构化响应。
**输出**: 返回功能三：时区感知与偏好学习的响应数据,包含状态码、结果和日志。

### 功能四：执行历史日志

每次任务执行后，记录到`~/workspace/schedule/history/YYYY-MM.jsonl`：

```json
{"id":"job_d4e5f6","ts":"2026-07-21T08:00:00+08:00","status":"success","duration_ms":1200,"summary":"汇总了12封邮件，5封重要"}
{"id":"job_d4e5f6","ts":"2026-07-22T08:00:00+08:00","status":"failed","duration_ms":3000,"error":"mail技能未授权","retry":0}
```

**日志字段**：
- `status`：success / failed / skipped
- `duration_ms`：执行耗时
- `error`：失败原因（若失败）
- `summary`：执行摘要（若成功）

**输入**: 用户提供功能四：执行历史日志所需的指令和必要参数。

**输出**: 返回功能四：执行历史日志的响应数据,包含状态码、结果和日志。

## 使用场景

### 场景一：个人开发者每日提醒（独立开发者角色）

**痛点**：独立开发者容易沉浸编码忘记休息、忘记站会、忘记提交代码。

**配置**：
```text
"每天下午5点提醒我提交代码并push"
"每周五下午4点提醒我写周报"
"每工作日中午12点提醒我午休"
```

**效果**：Agent在指定时间主动提醒，无需手动设置手机闹钟。任务定义持久化，即使关闭终端重新打开依然有效。

### 场景二：跨时区远程协作（远程工作者角色）

**痛点**：跨时区团队协作时，容易搞错对方当地时间，导致在不合适的时间发送消息。

**配置**：
```text
"每周二晚上9点（北京时间）提醒我与纽约团队同步，对方时间是早上9点"
```

**效果**：Agent自动换算时区，在回执中同时显示两个时区的时间，避免打扰对方休息。

### 场景三：学习计划打卡（学生角色）

**痛点**：制定学习计划但忘记执行，缺乏外部提醒。

**配置**：
```text
"每天晚上8点提醒我复习英语单词30分钟"
"每周六上午10点提醒我做一套模拟题"
"每天早上7点提醒我背5个新单词"
```

**效果**：多任务并行管理，通过`jobs.json`一目了然查看所有学习计划，执行历史可追溯坚持情况。

## FAQ

### Q1：免费版能创建多少个任务？

免费版不限制任务数量，但仅支持基础调度功能（一次性任务、周期任务、时区感知、执行日志）。高级功能如任务依赖链、失败重试、多通道告警、日历集成需要专业版。

### Q2：任务定义存储在哪里？安全吗？

所有任务定义存储在本地`~/workspace/schedule/`目录下，纯JSON格式，不涉及云端上传。可随时用文本编辑器查看与修改。建议将该目录纳入版本控制（如git），实现额外的版本备份。

### Q3：关闭终端后任务还会执行吗？

任务定义持久化在`jobs.json`中，但实际触发执行需要Agent处于运行状态。如果Agent关闭，任务不会触发，但会在Agent下次启动时检查`next_run`字段，补执行错过的任务（仅限一次性任务，周期任务跳过）。

### Q4：如何修改或删除已有任务？

两种方式：
1. **自然语言**：对Agent说"取消提交周报的任务"或"把邮件汇总改到9点"
2. **手动编辑**：直接编辑`~/workspace/schedule/jobs.json`，修改或删除对应条目

### Q5：支持哪些时区？

支持所有IANA时区标识符（如`Asia/Shanghai`、`America/New_York`、`Europe/London`、`UTC`）。首次使用时会根据系统时区自动推断，用户也可显式指定。
